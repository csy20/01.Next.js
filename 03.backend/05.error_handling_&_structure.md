## Why Does Structure Matter?

Imagine you have a small notebook where you write down everything — your grocery list, homework, diary entries, phone numbers, and meeting notes — all on the same page, mixed together. Finding anything would be a nightmare.

That's exactly what happens when you write all your server code in a single file. It works when the file is 50 lines long. It becomes chaos when it grows to 500 or 5000 lines.

**Structure is about organizing your code so that:**

- Each file has a **single, clear responsibility**
- You can find things quickly
- Multiple developers can work on different parts without stepping on each other's toes
- You can change one part of the system without accidentally breaking another
- Testing individual pieces becomes easy

---

## The Problem: Everything in One File

Here's what a typical beginner server looks like — everything crammed into one `index.js`:

```js
// index.js — THE MONOLITH OF DOOM 💀
const express = require("express");
const mongoose = require("mongoose");
const app = express();

app.use(express.json());

// Database connection
mongoose.connect("mongodb://localhost:27017/myapp");

// Schema definition
const userSchema = new mongoose.Schema({
  name: { type: String, required: true },
  email: { type: String, required: true, unique: true },
  age: { type: Number, min: 0 },
});

const User = mongoose.model("User", userSchema);

// Route: Get all users
app.get("/api/users", async (req, res) => {
  try {
    const users = await User.find();
    res.json(users);
  } catch (err) {
    res.status(500).json({ error: err.message });
  }
});

// Route: Create a user
app.post("/api/users", async (req, res) => {
  try {
    const user = new User(req.body);
    await user.save();
    res.status(201).json(user);
  } catch (err) {
    if (err.code === 11000) {
      res.status(409).json({ error: "Email already exists" });
    } else if (err.name === "ValidationError") {
      res.status(400).json({ error: err.message });
    } else {
      res.status(500).json({ error: "Something went wrong" });
    }
  }
});

// Route: Get a user by ID
app.get("/api/users/:id", async (req, res) => {
  try {
    const user = await User.findById(req.params.id);
    if (!user) {
      return res.status(404).json({ error: "User not found" });
    }
    res.json(user);
  } catch (err) {
    res.status(500).json({ error: err.message });
  }
});

// ... imagine 20 more routes like this for products, orders, reviews...

app.listen(3000, () => console.log("Server running on port 3000"));
```

### What's Wrong With This?

```
Problems with a single-file approach:
─────────────────────────────────────────────────────────
1. READABILITY    → Hard to find anything in a 2000-line file
2. MAINTAINABILITY → Changing one thing risks breaking another
3. REUSABILITY    → Can't reuse the User schema in another file
4. TESTABILITY    → Can't test the database logic independently
5. COLLABORATION  → Two developers editing the same file = merge conflicts
6. SCALABILITY    → Adding features means the file grows forever
```

The solution? **Split your code into organized layers.** That's where the MVC pattern comes in.

## MVC Pattern — The Big Idea

MVC stands for **Model-View-Controller**. It's a design pattern — a proven way to organize code that has been used for decades across almost every programming language and framework.

### The Restaurant Analogy 🍽️

Think of a restaurant:

```
┌─────────────────────────────────────────────────────────────────────┐
│                        THE RESTAURANT ANALOGY                       │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│   Customer                                                          │
│   (Client / Browser)                                                │
│       │                                                             │
│       │  "I'd like a margherita pizza, please"                      │
│       ▼                                                             │
│   ┌───────────┐                                                     │
│   │  WAITER   │  ──── This is the CONTROLLER                       │
│   │           │  Takes the order, doesn't cook, doesn't know        │
│   │           │  the recipe. Just coordinates between customer      │
│   │           │  and kitchen.                                       │
│   └─────┬─────┘                                                     │
│         │                                                           │
│         │  "Kitchen, I need one margherita pizza"                    │
│         ▼                                                           │
│   ┌───────────┐                                                     │
│   │  KITCHEN  │  ──── This is the MODEL                            │
│   │  (Chef)   │  Has the recipe (schema), has the ingredients       │
│   │           │  (data), knows HOW to make the pizza (business      │
│   │           │  logic). Does the actual work.                      │
│   └─────┬─────┘                                                     │
│         │                                                           │
│         │  "Here's the cooked pizza"                                │
│         ▼                                                           │
│   ┌───────────┐                                                     │
│   │   PLATE   │  ──── This is the VIEW                             │
│   │ (Presenta-│  How the food is presented — on a nice plate,       │
│   │   tion)   │  garnished, looking appetizing. The response        │
│   │           │  format (JSON, HTML, etc.)                          │
│   └───────────┘                                                     │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

### The Core Principle: Separation of Concerns

Each layer has **ONE job** and doesn't concern itself with the others:

| Layer          | Responsibility                      | Knows About          | Does NOT Know About     |
| -------------- | ----------------------------------- | -------------------- | ----------------------- |
| **Model**      | Data structure, database operations | Database, schemas    | HTTP, requests, routes  |
| **View**       | Presentation, response format       | What data to display | Database, business logic|
| **Controller** | Request handling, coordination      | Routes, req/res      | Database queries        |

> **Key Insight:** In a REST API (which is what we build with Express), the **View** layer is essentially the JSON response we send back. We don't have traditional HTML templates as views — our "view" is the JSON format of our response.

## Understanding Each Layer of MVC

### Model — The Data Layer

The Model is the **foundation** of your application. It defines:

1. **What your data looks like** (the schema/structure)
2. **Rules for the data** (validation — name is required, email must be unique)
3. **How to interact with the database** (CRUD operations)

Think of the Model as the **blueprint + the warehouse** of your application:

```
┌─────────────────────────────────────────────────┐
│                    MODEL LAYER                   │
├─────────────────────────────────────────────────┤
│                                                  │
│  Blueprint (Schema):                             │
│  ┌─────────────────────────────┐                │
│  │  User {                     │                │
│  │    name:  String (required) │                │
│  │    email: String (unique)   │                │
│  │    age:   Number (min: 0)   │                │
│  │  }                          │                │
│  └─────────────────────────────┘                │
│                                                  │
│  Operations:                                     │
│  • User.find()        → Get all users           │
│  • User.findById(id)  → Get one user            │
│  • User.create(data)  → Create a user           │
│  • user.save()        → Save changes            │
│  • User.findByIdAndUpdate() → Update            │
│  • User.findByIdAndDelete() → Delete            │
│                                                  │
│  The Model talks ONLY to the database.           │
│  It has NO idea about HTTP, routes, or Express.  │
│                                                  │
└─────────────────────────────────────────────────┘
```

```js
// models/User.js — A typical Model file

const mongoose = require("mongoose");

// The schema defines the SHAPE of the data
const userSchema = new mongoose.Schema(
  {
    name: {
      type: String,
      required: [true, "Name is required"],
      trim: true,
      minlength: [2, "Name must be at least 2 characters"],
      maxlength: [50, "Name cannot exceed 50 characters"],
    },
    email: {
      type: String,
      required: [true, "Email is required"],
      unique: true,
      lowercase: true,
      match: [/^\S+@\S+\.\S+$/, "Please provide a valid email"],
    },
    age: {
      type: Number,
      min: [0, "Age cannot be negative"],
      max: [150, "Age cannot exceed 150"],
    },
    role: {
      type: String,
      enum: ["user", "admin", "moderator"],
      default: "user",
    },
  },
  {
    timestamps: true, // Automatically adds createdAt and updatedAt
  }
);

// Export the model — this is what other files will import
module.exports = mongoose.model("User", userSchema);
```

**Notice:** This file has ZERO knowledge of Express, routes, or HTTP. It's pure data definition. You could take this same Model and use it in a command-line tool, a cron job, or a completely different framework.

### View — The Presentation Layer

In traditional web apps (like PHP or Rails), the View would be HTML templates (EJS, Pug, Handlebars). But in a **REST API** built with Express, we're sending **JSON responses** — so the "View" is how we **format our responses**.

```
┌─────────────────────────────────────────────────┐
│                    VIEW LAYER                    │
│             (In a REST API context)              │
├─────────────────────────────────────────────────┤
│                                                  │
│  Traditional Web App View:                       │
│  ┌─────────────────────────────┐                │
│  │  <html>                     │                │
│  │    <h1>Welcome, Alice!</h1> │                │
│  │    <p>Email: alice@...</p>  │                │
│  │  </html>                    │                │
│  └─────────────────────────────┘                │
│                                                  │
│  REST API "View" (What we use):                  │
│  ┌─────────────────────────────┐                │
│  │  {                          │                │
│  │    "status": "success",     │                │
│  │    "data": {                │                │
│  │      "name": "Alice",      │                │
│  │      "email": "alice@..."  │                │
│  │    }                        │                │
│  │  }                          │                │
│  └─────────────────────────────┘                │
│                                                  │
│  In REST APIs, the Controller directly sends     │
│  JSON. The "View" is the JSON structure itself.  │
│                                                  │
└─────────────────────────────────────────────────┘
```

> **In our Express API projects**, the View layer is absorbed into the Controller. The controller formats and sends the JSON response. That's perfectly fine for APIs. MVC is a guideline, not a law. We adapt it to fit our needs.

---

### Controller — The Logic Layer

The Controller is the **traffic cop** of your application. When a request comes in, the controller:

1. **Receives** the request from the route
2. **Extracts** data from the request (params, query, body)
3. **Validates** the input (optional, sometimes done in middleware)
4. **Calls the Model** to perform database operations
5. **Formats** the response and sends it back

```
┌─────────────────────────────────────────────────┐
│                 CONTROLLER LAYER                 │
├─────────────────────────────────────────────────┤
│                                                  │
│  Responsibilities:                               │
│  • Receive the request (req)                     │
│  • Extract data from req.body, req.params, etc. │
│  • Call the Model to interact with the database  │
│  • Handle errors from the Model                  │
│  • Format and send the response (res)            │
│                                                  │
│  Does NOT:                                       │
│  • Define database schemas                       │
│  • Write raw database queries                    │
│  • Define routes (that's the router's job)       │
│                                                  │
└─────────────────────────────────────────────────┘
```

```js
// controllers/userController.js — A typical Controller file

const User = require("../models/User"); // Import the Model

// GET all users
const getAllUsers = async (req, res) => {
  try {
    const users = await User.find(); // Ask the MODEL for data
    res.status(200).json({
      // Send the response (the VIEW)
      status: "success",
      results: users.length,
      data: { users },
    });
  } catch (err) {
    res.status(500).json({
      status: "error",
      message: err.message,
    });
  }
};

// GET a single user by ID
const getUserById = async (req, res) => {
  try {
    const user = await User.findById(req.params.id);

    if (!user) {
      return res.status(404).json({
        status: "fail",
        message: "No user found with that ID",
      });
    }

    res.status(200).json({
      status: "success",
      data: { user },
    });
  } catch (err) {
    res.status(500).json({
      status: "error",
      message: err.message,
    });
  }
};

// POST — Create a new user
const createUser = async (req, res) => {
  try {
    const newUser = await User.create(req.body);

    res.status(201).json({
      status: "success",
      data: { user: newUser },
    });
  } catch (err) {
    res.status(400).json({
      status: "fail",
      message: err.message,
    });
  }
};

// Export all controller functions
module.exports = {
  getAllUsers,
  getUserById,
  createUser,
};
```

**Notice:** The controller imports the Model and uses it, but it doesn't know any MongoDB query syntax directly. It just calls `User.find()`, `User.create()`, etc. The Model handles the database details.

## Building an MVC Project Step by Step

### Folder Structure

Here's the standard MVC folder structure for an Express application:

```
my-express-app/
│
├── models/              ← Data layer (schemas + database operations)
│   ├── User.js
│   ├── Product.js
│   └── Order.js
│
├── controllers/         ← Logic layer (request handling)
│   ├── userController.js
│   ├── productController.js
│   └── orderController.js
│
├── routes/              ← Route definitions (URL → Controller mapping)
│   ├── userRoutes.js
│   ├── productRoutes.js
│   └── orderRoutes.js
│
├── middleware/           ← Custom middleware (auth, error handling, etc.)
│   ├── errorHandler.js
│   ├── auth.js
│   └── validate.js
│
├── utils/               ← Helper functions and utilities
│   ├── AppError.js
│   └── asyncHandler.js
│
├── config/              ← Configuration files
│   └── db.js
│
├── app.js               ← Express app setup (middleware, routes)
├── server.js            ← Server startup (listen + DB connection)
├── package.json
└── .env                 ← Environment variables (SECRET! Never commit this)
```

### Why Separate `app.js` and `server.js`?

```
┌─────────────────────────────┐    ┌──────────────────────────────┐
│          app.js              │    │          server.js            │
├─────────────────────────────┤    ├──────────────────────────────┤
│                              │    │                              │
│  • Creates Express app       │    │  • Connects to database      │
│  • Sets up middleware        │    │  • Starts the server         │
│  • Mounts routes             │    │  • Handles unhandled errors  │
│  • Sets up error handling    │    │  • Imports app from app.js   │
│                              │    │                              │
│  Exports: app                │    │  Does NOT export anything    │
│                              │    │  This is the entry point     │
│  Can be imported for         │    │                              │
│  testing WITHOUT starting    │    │  node server.js ← Run this   │
│  the server!                 │    │                              │
└─────────────────────────────┘    └──────────────────────────────┘
```

This separation lets you **test your app** without actually starting the server.

```js
const mongoose = require("mongoose");
```

### Step 2: The Controller

```js
// controllers/userController.js

const User = require("../models/User");

// @desc    Get all users
// @route   GET /api/users
exports.getAllUsers = async (req, res) => {
  try {
    const users = await User.find();
    res.status(200).json({
      status: "success",
      results: users.length,
      data: { users },
    });
  } catch (err) {
    res.status(500).json({ status: "error", message: err.message });
  }
};

// @desc    Get a single user
// @route   GET /api/users/:id
exports.getUserById = async (req, res) => {
  try {
    const user = await User.findById(req.params.id);
    if (!user) {
      return res
        .status(404)
        .json({ status: "fail", message: "User not found" });
    }
    res.status(200).json({ status: "success", data: { user } });
  } catch (err) {
    res.status(500).json({ status: "error", message: err.message });
  }
};

// @desc    Create a new user
// @route   POST /api/users
exports.createUser = async (req, res) => {
  try {
    const user = await User.create(req.body);
    res.status(201).json({ status: "success", data: { user } });
  } catch (err) {
    res.status(400).json({ status: "fail", message: err.message });
  }
};

// @desc    Update a user
// @route   PUT /api/users/:id
exports.updateUser = async (req, res) => {
  try {
    const user = await User.findByIdAndUpdate(req.params.id, req.body, {
      new: true, // Return the updated document
      runValidators: true, // Run schema validators on update
    });
    if (!user) {
      return res
        .status(404)
        .json({ status: "fail", message: "User not found" });
    }
    res.status(200).json({ status: "success", data: { user } });
  } catch (err) {
    res.status(400).json({ status: "fail", message: err.message });
  }
};

// @desc    Delete a user
// @route   DELETE /api/users/:id
exports.deleteUser = async (req, res) => {
  try {
    const user = await User.findByIdAndDelete(req.params.id);
    if (!user) {
      return res
        .status(404)
        .json({ status: "fail", message: "User not found" });
    }
    res.status(204).json({ status: "success", data: null });
  } catch (err) {
    res.status(500).json({ status: "error", message: err.message });
  }
};
```

### Step 3: The Routes

```js
// routes/userRoutes.js

const express = require("express");
const router = express.Router();
const userController = require("../controllers/userController");

// Routes map URLs to controller functions
// The router is a mini-app that handles routing for a specific resource

router
  .route("/")
  .get(userController.getAllUsers) // GET    /api/users
  .post(userController.createUser); // POST   /api/users

router
  .route("/:id")
  .get(userController.getUserById) // GET    /api/users/:id
  .put(userController.updateUser) // PUT    /api/users/:id
  .delete(userController.deleteUser); // DELETE /api/users/:id

module.exports = router;
```

### Step 4: The Entry Point

```js
// app.js — Express app configuration

const express = require("express");
const userRoutes = require("./routes/userRoutes");

const app = express();

// ─── Global Middleware ────────────────────────────
app.use(express.json()); // Parse JSON request bodies

// ─── Routes ──────────────────────────────────────
// All user routes will be prefixed with /api/users
app.use("/api/users", userRoutes);

// ─── 404 Handler ──────────────────────────────────
// If no route matched, this runs
app.all("*", (req, res) => {
  res.status(404).json({
    status: "fail",
    message: `Cannot find ${req.originalUrl} on this server`,
  });
});

module.exports = app;
```

```js
// server.js — Server startup

const mongoose = require("mongoose");
const app = require("./app");

const PORT = process.env.PORT || 3000;

// Connect to MongoDB, then start the server
mongoose
  .connect("mongodb://localhost:27017/myapp")
  .then(() => {
    console.log("✅ Connected to MongoDB");
    app.listen(PORT, () => {
      console.log(`🚀 Server running on port ${PORT}`);
    });
  })
  .catch((err) => {
    console.error("❌ MongoDB connection error:", err.message);
  });
```

## The Flow of a Request Through MVC

Let's trace **exactly** what happens when a client sends `GET /api/users/123`:

```
Step 1: Client sends request
─────────────────────────────────────────────────────────────────

  Client (Browser/Postman)
    │
    │  GET /api/users/123
    ▼

Step 2: Express receives request, runs through global middleware
─────────────────────────────────────────────────────────────────

  server.js (entry point)
    │
    ▼
  app.js
    │
    ├── express.json() middleware runs (no body to parse, skip)
    │
    ▼

Step 3: Express matches the URL to a route
─────────────────────────────────────────────────────────────────

  app.use("/api/users", userRoutes)
    │
    │  URL starts with /api/users ✓
    │  Remaining path: /123
    │
    ▼
  routes/userRoutes.js
    │
    │  router.route("/:id").get(...)
    │  /:id matches /123 → req.params.id = "123"
    │
    ▼

Step 4: Controller function executes
─────────────────────────────────────────────────────────────────

  controllers/userController.js → getUserById()
    │
    │  const user = await User.findById("123")
    │
    ▼

Step 5: Model queries the database
─────────────────────────────────────────────────────────────────

  models/User.js → User.findById("123")
    │
    │  MongoDB query: db.users.findOne({ _id: ObjectId("123") })
    │
    ▼

Step 6: Database returns result
─────────────────────────────────────────────────────────────────

  MongoDB → { _id: "123", name: "Alice", email: "alice@..." }
    │
    ▼

Step 7: Controller sends response
─────────────────────────────────────────────────────────────────

  Controller sends back:
  res.status(200).json({
    status: "success",
    data: { user: { _id: "123", name: "Alice", ... } }
  })
    │
    ▼

Step 8: Client receives the response
─────────────────────────────────────────────────────────────────

  Client receives: 200 OK with JSON data
```

Here's the same flow as a compact diagram:

```
┌────────┐    ┌─────────┐    ┌────────────┐    ┌───────┐    ┌──────────┐
```

## Why Errors Happen — The Nature of the Problem

Errors are **inevitable** in any application. They're not bugs — they're expected situations that your code must handle gracefully. Here are the main categories:

```
Types of Errors in a Web Application:
──────────────────────────────────────────────────────────────────

1. CLIENT ERRORS (The user did something wrong)
   ├── Sent invalid data         → "email field is missing"
   ├── Requested something       → "user with ID 999 doesn't exist"
   │   that doesn't exist
   ├── Not authenticated         → "you must log in first"
   └── Not authorized            → "you don't have permission"

2. SERVER ERRORS (Something broke on our end)
   ├── Database is down          → "cannot connect to MongoDB"
   ├── Code bug                  → "Cannot read property 'x' of undefined"
   ├── External service failed   → "Payment gateway timeout"
   └── Server ran out of memory  → "JavaScript heap out of memory"

3. VALIDATION ERRORS (Data doesn't meet the rules)
   ├── Missing required fields   → "Name is required"
   ├── Invalid format            → "Email must be valid"
   ├── Value out of range        → "Age must be between 0 and 150"
   └── Duplicate entry           → "Email already exists"
```

The question is: **How do you handle all of these consistently?**

## Error Handling Without Middleware (The Messy Way)

Without a centralized error handler, you end up repeating error-handling code in **every single route**:

```js
// ❌ THE MESSY WAY — Error handling repeated everywhere

// Route 1
app.get("/api/users", async (req, res) => {
  try {
    const users = await User.find();
    res.json(users);
  } catch (err) {
    // Error handling duplicated here
    console.error(err);
    res.status(500).json({ error: err.message });
  }
});

// Route 2
app.get("/api/users/:id", async (req, res) => {
  try {
    const user = await User.findById(req.params.id);
    if (!user) return res.status(404).json({ error: "Not found" });
    res.json(user);
  } catch (err) {
    // Same error handling duplicated again
    console.error(err);
    res.status(500).json({ error: err.message });
  }
});

// Route 3
app.post("/api/users", async (req, res) => {
  try {
    const user = await User.create(req.body);
    res.status(201).json(user);
  } catch (err) {
    // And again...
    if (err.code === 11000) {
      res.status(409).json({ error: "Duplicate" });
    } else if (err.name === "ValidationError") {
      res.status(400).json({ error: err.message });
    } else {
      console.error(err);
      res.status(500).json({ error: err.message });
    }
  }
});

// Imagine doing this for 50 routes... 😰
```

### Problems With This Approach

```
1. DRY VIOLATION     → Same error code copy-pasted in every route
2. INCONSISTENCY     → Different routes format errors differently
3. HARD TO MAINTAIN  → Want to change error format? Edit 50 routes.
4. EASY TO FORGET    → Miss a try-catch? Unhandled error crashes the server!
5. NO LOGGING        → Some routes log, some don't. No consistency.
```

## What is Error-Handling Middleware?

Express has a special type of middleware specifically for handling errors. It's a centralized place where ALL errors end up, no matter which route they came from.

### The Key Difference: 4 Parameters Instead of 3

Normal middleware has 3 parameters: `(req, res, next)`.
Error-handling middleware has **4 parameters**: `(err, req, res, next)`.

Express recognizes this signature and treats it specially:

```js
// Normal middleware — 3 parameters
app.use((req, res, next) => {
  console.log("I run on every request");
  next();
});

// Error-handling middleware — 4 parameters (the first one is 'err')
app.use((err, req, res, next) => {
  console.log("I only run when an error occurs");
  res.status(500).json({ error: err.message });
});
```

> **Critical Rule:** Express determines if middleware is an error handler **solely by counting the parameters**. If your function doesn't have exactly 4 parameters, Express will NOT treat it as an error handler. Even if you don't use `next`, you must include it.

---

## How Express Error-Handling Middleware Works

### How Errors Reach the Error Handler

There are **two ways** errors get sent to the error-handling middleware:

```
Method 1: Calling next(error)
─────────────────────────────────────────────

  app.get("/api/users/:id", (req, res, next) => {
    const user = findUser(req.params.id);
    if (!user) {
      const error = new Error("User not found");
      error.statusCode = 404;
      return next(error);  // ← Pass the error to next()
    }
    res.json(user);
  });

  // When you call next() with an argument, Express SKIPS all
  // remaining normal middleware and routes, and jumps straight
  // to the error-handling middleware.


Method 2: Throwing an error (in synchronous code)
─────────────────────────────────────────────

  app.get("/sync-route", (req, res) => {
    throw new Error("Something went wrong!");
    // Express catches this automatically in sync code
    // and forwards it to the error handler
  });

  // ⚠️ WARNING: This does NOT work in async code!
  // In async functions, you MUST use next(error) or try-catch.
```

### The Flow of Error Handling

```
  Request arrives
       │
       ▼
  ┌──────────────┐
  │  Middleware 1  │  ──── express.json()
  └──────┬───────┘
         │
         ▼
  ┌──────────────┐
  │  Middleware 2  │  ──── auth check
  └──────┬───────┘
         │
         ▼
  ┌──────────────┐
  │    Route      │  ──── controller function
  │   Handler     │
  └──────┬───────┘
         │
    Error occurs!
    next(error)
         │
         │  ╔═══════════════════════════════════════╗
         │  ║  Express SKIPS all remaining normal   ║
         │  ║  middleware and routes. Jumps to the   ║
         │  ║  first error-handling middleware.      ║
         │  ╚═══════════════════════════════════════╝
         │
         ▼
  ┌──────────────────┐
  │  Error-Handling   │  ──── (err, req, res, next)
  │   Middleware      │  ──── Handles ALL errors here!
  └──────────────────┘
         │
         ▼
  Response sent to client
```

## Building a Global Error Handler Step by Step

### Step 1: Create a Basic Error Handler

```js
// middleware/errorHandler.js

const errorHandler = (err, req, res, next) => {
  // 1. Log the error (for developers)
  console.error("💥 ERROR:", err);

  // 2. Set default values
  const statusCode = err.statusCode || 500;
  const status = err.status || "error";

  // 3. Send the response
  res.status(statusCode).json({
    status: status,
    message: err.message || "Something went wrong",
  });
};

module.exports = errorHandler;
```

### Step 2: Mount It in Your App (MUST be last!)

```js
// app.js

const express = require("express");
const userRoutes = require("./routes/userRoutes");
const errorHandler = require("./middleware/errorHandler");

const app = express();

app.use(express.json());
app.use("/api/users", userRoutes);

// 404 handler — for routes that don't exist
app.all("*", (req, res, next) => {
  const err = new Error(`Cannot find ${req.originalUrl} on this server`);
  err.statusCode = 404;
  err.status = "fail";
  next(err); // Pass to error handler
});

// ⬇️ ERROR HANDLER MUST BE THE LAST app.use() ⬇️
// It must be after ALL routes and other middleware
app.use(errorHandler);

module.exports = app;
```

> **Why must the error handler be last?** Because Express processes middleware **in order**. If the error handler is before your routes, errors from routes would have nowhere to go. The error handler needs to be at the bottom to catch errors from everything above it.

### Step 3: Use It in Controllers

Now your controllers can pass errors to `next()` instead of handling them inline:

```js
// controllers/userController.js

const User = require("../models/User");

exports.getUserById = async (req, res, next) => {
  try {
    const user = await User.findById(req.params.id);

    if (!user) {
      const error = new Error("No user found with that ID");
      error.statusCode = 404;
      error.status = "fail";
      return next(error); // ← Pass error to the global handler
    }

    res.status(200).json({
      status: "success",
      data: { user },
    });
  } catch (err) {
    next(err); // ← Pass any unexpected errors to the global handler
  }
};
```

## Custom Error Classes

Creating errors with `new Error()` and manually setting `statusCode` and `status` every time is tedious. Let's create a custom error class:

```js
// utils/AppError.js

class AppError extends Error {
  constructor(message, statusCode) {
    // Call the parent Error constructor with the message
    super(message);

    this.statusCode = statusCode;

    // Determine status based on the status code
    // 4xx = "fail" (client error), 5xx = "error" (server error)
    this.status = `${statusCode}`.startsWith("4") ? "fail" : "error";

    // Mark this as an "operational" error (expected, handled)
    // vs a "programming" error (unexpected bug)
    this.isOperational = true;

    // Capture the stack trace (for debugging)
    // This removes the constructor call from the stack trace
    Error.captureStackTrace(this, this.constructor);
  }
}

module.exports = AppError;
```

### Understanding Operational vs Programming Errors

```
┌──────────────────────────────────────────────────────────────┐
│                     TYPES OF ERRORS                          │
├───────────────────────────┬──────────────────────────────────┤
│   OPERATIONAL ERRORS      │   PROGRAMMING ERRORS             │
│   (Expected, Handled)     │   (Bugs, Unexpected)             │
├───────────────────────────┼──────────────────────────────────┤
│ • Invalid user input      │ • Reading property of undefined  │
│ • Resource not found      │ • Using wrong variable name      │
│ • Authentication failure  │ • Forgetting to await a promise  │
│ • Database validation     │ • Type errors                    │
│ • Network timeout         │ • Syntax errors                  │
│                           │ • Logical bugs in code           │
├───────────────────────────┼──────────────────────────────────┤
│ We ANTICIPATE these.      │ We DON'T anticipate these.       │
│ Send clean error to user. │ Log for devs, generic msg to     │
│                           │ user. These are real bugs.        │
└───────────────────────────┴──────────────────────────────────┘
```

### Enhanced Error Handler with Mongoose Error Detection

```js
// middleware/errorHandler.js

const AppError = require("../utils/AppError");

// ─── Handle specific Mongoose errors ─────────────────────

// When an invalid MongoDB ObjectId is provided (e.g., "abc" instead of a valid ID)
const handleCastErrorDB = (err) => {
  const message = `Invalid ${err.path}: ${err.value}`;
  return new AppError(message, 400);
};

// When a duplicate field value is provided (unique constraint violation)
const handleDuplicateFieldsDB = (err) => {
  const value = err.errmsg.match(/(["'])(\\?.)*?\1/)[0];
  const message = `Duplicate field value: ${value}. Please use another value.`;
  return new AppError(message, 400);
};

// When Mongoose validation fails (required fields, min/max, enum, etc.)
const handleValidationErrorDB = (err) => {
  const errors = Object.values(err.errors).map((el) => el.message);
  const message = `Invalid input data: ${errors.join(". ")}`;
  return new AppError(message, 400);
};

// ─── Main Error Handler ──────────────────────────────────

const errorHandler = (err, req, res, next) => {
  err.statusCode = err.statusCode || 500;
  err.status = err.status || "error";

  // In development: send full error details
  if (process.env.NODE_ENV === "development") {
    res.status(err.statusCode).json({
      status: err.status,
      error: err,
      message: err.message,
      stack: err.stack, // Full stack trace for debugging
    });
  }
  // In production: send clean, user-friendly messages
  else {
    let error = { ...err };
    error.message = err.message;

    // Transform known Mongoose errors into AppErrors
    if (err.name === "CastError") error = handleCastErrorDB(error);
    if (err.code === 11000) error = handleDuplicateFieldsDB(error);
    if (err.name === "ValidationError")
      error = handleValidationErrorDB(error);

    // Operational error: send the message to the client
    if (error.isOperational) {
      res.status(error.statusCode).json({
        status: error.status,
        message: error.message,
      });
    }
    // Programming error: don't leak details to the client
    else {
      console.error("💥 PROGRAMMING ERROR:", err);
      res.status(500).json({
        status: "error",
        message: "Something went very wrong!",
      });
    }
  }
};

module.exports = errorHandler;
```

### How the Error Handler Decides What to Send

```
Error comes in
     │
     ▼
Is it development mode?
     │
     ├── YES → Send EVERYTHING (status, error object, message, stack trace)
     │         Developers need all the details to debug.
     │
     └── NO (production) →
              │
              ▼
         Is it a known Mongoose error?
              │
              ├── CastError?        → "Invalid ID: abc"      (400)
              ├── Code 11000?       → "Duplicate email"       (400)
              └── ValidationError?  → "Name is required"      (400)
              │
              ▼
         Is it operational (AppError)?
              │
              ├── YES → Send the clean error message to the user
              │         (e.g., "User not found" with status 404)
              │
              └── NO  → It's a programming bug!
                        Log the full error for developers.
                        Send generic "Something went wrong" to user.
                        NEVER leak stack traces or internal details!
```

## Async Error Handling — The Catch Problem

### The Problem With Async/Await

Every async controller function needs a `try-catch` block. This creates a LOT of repetitive code:

```js
// 😩 Every single async controller needs try-catch

exports.getAllUsers = async (req, res, next) => {
  try {
    // actual logic
    const users = await User.find();
    res.json(users);
  } catch (err) {
    next(err); // Always the same catch block
  }
};

exports.getUser = async (req, res, next) => {
  try {
    // actual logic
    const user = await User.findById(req.params.id);
    res.json(user);
  } catch (err) {
    next(err); // Same thing again
  }
};

exports.createUser = async (req, res, next) => {
  try {
    // actual logic
    const user = await User.create(req.body);
    res.json(user);
  } catch (err) {
    next(err); // And again...
  }
};
```

Every controller has the same `try { ... } catch (err) { next(err) }` wrapper. Can we eliminate this repetition?

---

## The asyncHandler Wrapper

The solution is a small utility function that **wraps any async function** and automatically catches errors:

```js
// utils/asyncHandler.js

// This function takes an async function and returns a new function
// that automatically catches any errors and passes them to next()

const asyncHandler = (fn) => {
  return (req, res, next) => {
    // fn(req, res, next) returns a Promise (because fn is async)
    // .catch(next) catches any rejected promise and passes the error to next()
    fn(req, res, next).catch(next);
  };
};

module.exports = asyncHandler;
```

### How asyncHandler Works — Step by Step

```
Without asyncHandler:
─────────────────────────────────────────────
  exports.getUser = async (req, res, next) => {
    try {
      const user = await User.findById(req.params.id);
      if (!user) return next(new AppError("Not found", 404));
      res.json(user);
    } catch (err) {
      next(err);  ← You must remember this EVERY time
    }
  };


With asyncHandler:
─────────────────────────────────────────────
  exports.getUser = asyncHandler(async (req, res, next) => {
    const user = await User.findById(req.params.id);
    if (!user) return next(new AppError("Not found", 404));
    res.json(user);
  });
  // No try-catch needed! asyncHandler catches errors automatically.
```

### Let's Trace Through What Happens

```
1. You define:
   exports.getUser = asyncHandler(async (req, res, next) => { ... });

2. asyncHandler receives your async function as 'fn'

3. asyncHandler returns a NEW function: (req, res, next) => { ... }
   This new function is what Express actually calls when the route is hit.

4. When a request comes in, Express calls the returned function with (req, res, next)

5. The returned function calls fn(req, res, next) — your original async function

6. fn() returns a Promise (because it's async)

7. If the Promise RESOLVES (no error):
   → Everything works normally, response is sent

8. If the Promise REJECTS (error thrown or await fails):
   → .catch(next) catches the error
   → next(error) is called automatically
   → Express forwards to the error-handling middleware

It's like an INVISIBLE try-catch around every controller!
```

### Full Controller with asyncHandler

```js

const AppError = require("../utils/AppError");
const asyncHandler = require("../utils/asyncHandler");

// ✅ No try-catch needed! Clean and focused on business logic.

  res.status(200).json({
    status: "success",
    results: users.length,
    data: { users },
  });

  res.status(201).json({
    status: "success",
    data: { user },
  });
  // If User.create() fails (validation, duplicate), the error
  // is automatically caught by asyncHandler and sent to
  // the error-handling middleware
});

exports.updateUser = asyncHandler(async (req, res, next) => {
```

## HTTP Status Codes — The Complete Guide

### What Are Status Codes?

Every HTTP response includes a **3-digit number** called a status code. This number tells the client (browser, Postman, frontend app) **what happened** with their request — before they even look at the response body.

```
HTTP/1.1 200 OK                    ← Status code (200) + Reason phrase (OK)
Content-Type: application/json

{ "name": "Alice", "email": "alice@example.com" }
```

Think of status codes as **traffic signals for the web**:

```
┌────────────────────────────────────────────────────────────────┐
│                    STATUS CODES = TRAFFIC SIGNALS               │
├────────────────────────────────────────────────────────────────┤
│                                                                 │
│   🟢  2xx  =  GREEN LIGHT    →  "Everything went well!"       │
│   🔵  3xx  =  DETOUR SIGN    →  "Go somewhere else."          │
│   🟡  4xx  =  YELLOW LIGHT   →  "YOU did something wrong."    │
│   🔴  5xx  =  RED LIGHT      →  "WE messed up on our end."   │
│                                                                 │
│   The FIRST digit tells you the CATEGORY.                      │
│   The remaining digits give the SPECIFIC situation.            │
│                                                                 │
└────────────────────────────────────────────────────────────────┘
```

### The Five Families of Status Codes

```
┌───────┬──────────────────┬───────────────────────────────────────┐
│ Range │     Category     │              Meaning                  │
├───────┼──────────────────┼───────────────────────────────────────┤
│ 1xx   │  Informational   │ "Hold on, I'm still processing..."   │
│ 2xx   │  Success         │ "Here you go, everything worked!"    │
│ 3xx   │  Redirection     │ "What you want is somewhere else."   │
│ 4xx   │  Client Error    │ "You made a mistake in your request."│
│ 5xx   │  Server Error    │ "We screwed up. Sorry!"              │
└───────┴──────────────────┴───────────────────────────────────────┘
```

---

### 1xx Informational

These are rare in REST APIs. They indicate that the server has received the request and is continuing to process it.

| Code | Name                | When Used                              |
| ---- | ------------------- | -------------------------------------- |
| 100  | Continue            | Server received headers, client should send body |
| 101  | Switching Protocols | Server is switching to WebSocket, etc. |
| 102  | Processing          | Server is still working (WebDAV)       |

> **In practice:** You almost never set these manually. The server handles them automatically. Don't worry about these for now.

---

### 2xx Success

These are the **happy path** codes — the request worked!

| Code | Name       | Meaning                               | When to Use                          |
| ---- | ---------- | ------------------------------------- | ------------------------------------ |
| 200  | OK         | Request succeeded                     | GET, PUT, PATCH — default success    |
| 201  | Created    | New resource was created              | POST — when you create something new |
| 204  | No Content | Success, but no body to return        | DELETE — deleted successfully        |

#### Using 2xx in Express

```js
// 200 — OK (default, most common)
// Used for: successful GET, PUT, PATCH requests
app.get("/api/users", async (req, res) => {
  const users = await User.find();
  res.status(200).json({ status: "success", data: { users } });
  // Note: 200 is the default status, so res.json(...) alone also works
});

// 201 — Created
// Used for: successful POST requests that create new resources
app.post("/api/users", async (req, res) => {
  const user = await User.create(req.body);
  res.status(201).json({ status: "success", data: { user } });
});

// 204 — No Content
// Used for: successful DELETE requests
// Note: The body won't actually be sent with a 204 response
app.delete("/api/users/:id", async (req, res) => {
  await User.findByIdAndDelete(req.params.id);
  res.status(204).json({ status: "success", data: null });
});
```

### How to Choose Between 200, 201, and 204

```
Did the request CREATE something new?
│
├── YES → Use 201 Created
│
└── NO → Did the request DELETE something?
         │
         ├── YES → Use 204 No Content
         │
         └── NO → Use 200 OK  (default for GET, PUT, PATCH)
```

---

### 3xx Redirection

These tell the client to go somewhere else to find what they're looking for.

| Code | Name               | Meaning                                     | When to Use                          |
| ---- | ------------------ | ------------------------------------------- | ------------------------------------ |
| 301  | Moved Permanently  | Resource has permanently moved to a new URL | Old URL is retired forever           |
| 302  | Found              | Resource temporarily at a different URL     | Temporary redirect                   |
| 304  | Not Modified       | Resource hasn't changed, use cached version | Browser caching optimization         |
| 307  | Temporary Redirect | Temporary redirect, keep method             | Like 302 but preserves POST method   |
| 308  | Permanent Redirect | Permanent redirect, keep method             | Like 301 but preserves POST method   |

```js
// 301 — Moved Permanently
// "This URL is retired. The new URL is..."
app.get("/old-api/users", (req, res) => {
  res.redirect(301, "/api/v2/users");
});

// 302 — Found (Temporary Redirect)
// "Temporarily go here instead"
app.get("/maintenance", (req, res) => {
  res.redirect(302, "/alternative-service");
});

// 304 — Not Modified
// Usually handled automatically by Express/browser caching
// You rarely set this manually
```

> **In REST APIs**, 3xx codes are used less frequently than in traditional web apps. You'll mostly encounter them when versioning your API or handling URL changes.

### 4xx Client Errors

These mean **the client did something wrong**. The request was malformed, unauthorized, or asking for something that doesn't exist. This is the most important family for API development.

| Code | Name                  | Meaning                                    | When to Use                              |
| ---- | --------------------- | ------------------------------------------ | ---------------------------------------- |
| 400  | Bad Request           | The request data is invalid or malformed   | Missing fields, wrong format, bad JSON   |
| 401  | Unauthorized          | Authentication is required but missing     | No token, expired token, invalid token   |
| 403  | Forbidden             | Authenticated but lacks permission         | User can't access admin-only resource    |
| 404  | Not Found             | The requested resource doesn't exist       | Invalid ID, deleted resource, wrong URL  |
| 405  | Method Not Allowed    | HTTP method not supported for this route   | POST to a GET-only endpoint              |
| 409  | Conflict              | Request conflicts with current state       | Duplicate email, conflicting update      |
| 422  | Unprocessable Entity  | Server understands but can't process       | Semantic validation failure              |
| 429  | Too Many Requests     | Client has sent too many requests          | Rate limiting                            |

#### Understanding Each 4xx Code in Detail

**400 — Bad Request: "Your data is wrong"**

```js
// The client sent data that doesn't make sense
exports.createUser = asyncHandler(async (req, res, next) => {
  const { name, email } = req.body;

  // Missing required fields
  if (!name || !email) {
    return next(new AppError("Please provide both name and email", 400));
  }

  // Invalid email format
  if (!email.includes("@")) {
    return next(new AppError("Please provide a valid email address", 400));
  }

  // Invalid data type
  if (typeof name !== "string") {
    return next(new AppError("Name must be a string", 400));
  }

  const user = await User.create(req.body);
  res.status(201).json({ status: "success", data: { user } });
});
```

**401 — Unauthorized: "Who are you? Log in first!"**

```js
// The client didn't provide authentication credentials
// or the credentials are invalid

const protect = asyncHandler(async (req, res, next) => {
  // Check if token exists in the Authorization header
  let token;
  if (
    req.headers.authorization &&
    req.headers.authorization.startsWith("Bearer")
  ) {
    token = req.headers.authorization.split(" ")[1];
  }

  if (!token) {
    return next(
      new AppError("You are not logged in. Please log in to get access.", 401)
    );
  }

  // Verify the token (more on this in the Authentication chapter)
  // If token is invalid or expired → 401
  next();
});
```

**403 — Forbidden: "I know who you are, but you can't do this"**

```js
// The difference between 401 and 403:
//
// 401 = "I don't know who you are" (no authentication)
// 403 = "I know who you are, but you don't have permission" (no authorization)

const restrictTo = (...roles) => {
  return (req, res, next) => {
    // req.user was set by the protect middleware
    if (!roles.includes(req.user.role)) {
      return next(
        new AppError("You do not have permission to perform this action", 403)
      );
    }
    next();
  };
};

// Usage:
// Only admins can delete users
router.delete(
  "/:id",
  protect, // First: check if logged in (401 if not)
  restrictTo("admin"), // Then: check if admin (403 if not)
  userController.deleteUser
);
```

```
401 vs 403 — The Nightclub Analogy:
──────────────────────────────────────────────────────

401 (Unauthorized):
  You walk up to the nightclub without an ID.
  Bouncer: "Sorry, I need to see your ID first."
  → You haven't PROVEN who you are.

403 (Forbidden):
  You show your ID. The bouncer sees you're 16.
  Bouncer: "I know who you are, but you're not old enough."
  → You've proven who you are, but you LACK PERMISSION.
```

**404 — Not Found: "That doesn't exist"**

```js
// The most commonly used error in REST APIs

exports.getUserById = asyncHandler(async (req, res, next) => {
  const user = await User.findById(req.params.id);

  if (!user) {
    return next(new AppError("No user found with that ID", 404));
  }

  res.status(200).json({ status: "success", data: { user } });
});

// Also used for routes that don't exist at all
app.all("*", (req, res, next) => {
  next(new AppError(`Cannot find ${req.originalUrl} on this server`, 404));
});
```

**409 — Conflict: "This clashes with something that already exists"**

```js
// Duplicate entry — something with that unique value already exists

exports.createUser = asyncHandler(async (req, res, next) => {
  // Check if email already exists
  const existingUser = await User.findOne({ email: req.body.email });

  if (existingUser) {
    return next(
      new AppError("A user with this email already exists", 409)
    );
  }

  const user = await User.create(req.body);
  res.status(201).json({ status: "success", data: { user } });
});
```

**429 — Too Many Requests: "Slow down!"**

```js
// Rate limiting — the client is sending too many requests

// Using the express-rate-limit package:
const rateLimit = require("express-rate-limit");

const limiter = rateLimit({
  max: 100, // Maximum 100 requests
  windowMs: 60 * 60 * 1000, // Per 1 hour
  message: "Too many requests from this IP, please try again in an hour",
  // This automatically sends a 429 status code
});

// Apply to all API routes
app.use("/api", limiter);
```

### 5xx Server Errors

These mean **the server messed up**. The client's request may have been perfectly valid, but something went wrong on the server side.

| Code | Name                  | Meaning                                   | When It Happens                        |
| ---- | --------------------- | ----------------------------------------- | -------------------------------------- |
| 500  | Internal Server Error | Generic server error                      | Unhandled exception, bug in code       |
| 502  | Bad Gateway           | Server got invalid response from upstream | Reverse proxy can't reach backend      |
| 503  | Service Unavailable   | Server is temporarily down                | Maintenance mode, server overloaded    |
| 504  | Gateway Timeout       | Upstream server didn't respond in time    | Backend is too slow                    |

```js
// 500 — Internal Server Error
// This is the DEFAULT when something unexpected goes wrong
// Your error handler should use this as the fallback:

const errorHandler = (err, req, res, next) => {
  const statusCode = err.statusCode || 500; // Default to 500
  res.status(statusCode).json({
    status: "error",
    message:
      statusCode === 500
        ? "Something went wrong on our end" // Generic message for 500
        : err.message, // Specific message for known errors
  });
};
```

```
⚠️  CRITICAL RULE FOR 5xx ERRORS:

NEVER send detailed error messages to the client for 5xx errors!

❌ BAD:  res.status(500).json({ error: "Cannot read property 'name' of undefined at line 42 of userController.js" })
✅ GOOD: res.status(500).json({ error: "Something went wrong. Please try again later." })

Why? Detailed server errors can reveal:
- File paths (reveals your directory structure)
- Database queries (reveals your schema)
- Stack traces (reveals your code)
- Dependencies and versions (reveals potential vulnerabilities)

This is a SECURITY RISK. Always send generic messages for 5xx errors.
Log the details server-side for your developers.
```

### Quick Reference: Which Status Code Should I Use?

```
┌──────────────────────────────────────────────────────────────┐
│              STATUS CODE DECISION TREE                        │
├──────────────────────────────────────────────────────────────┤
│                                                               │
│  Did the request SUCCEED?                                     │
│  │                                                            │
│  ├── YES                                                      │
│  │   ├── Created something new?  → 201 Created               │
│  │   ├── Deleted something?      → 204 No Content            │
│  │   └── Everything else?        → 200 OK                    │
│  │                                                            │
│  └── NO — What went wrong?                                    │
│      │                                                        │
│      ├── CLIENT's fault?                                      │
│      │   ├── Bad/missing data?    → 400 Bad Request           │
│      │   ├── Not logged in?       → 401 Unauthorized          │
│      │   ├── No permission?       → 403 Forbidden             │
│      │   ├── Doesn't exist?       → 404 Not Found             │
│      │   ├── Already exists?      → 409 Conflict              │
│      │   └── Too many requests?   → 429 Too Many Requests     │
│      │                                                        │
│      └── SERVER's fault?                                      │
│          └── Something crashed?   → 500 Internal Server Error │
│                                                               │
└──────────────────────────────────────────────────────────────┘
```

```js
class AppError extends Error {
  constructor(message, statusCode) {
    super(message);
    this.statusCode = statusCode;
    this.status = `${statusCode}`.startsWith("4") ? "fail" : "error";

}
```

### middleware/errorHandler.js

```js
const AppError = require("../utils/AppError");

const handleCastErrorDB = (err) => {
  return new AppError(`Invalid ${err.path}: ${err.value}`, 400);
};

const handleDuplicateFieldsDB = (err) => {
  const value = err.errmsg.match(/(["'])(\\?.)*?\1/)[0];
  return new AppError(`Duplicate field value: ${value}. Use another value.`, 400);
};

const handleValidationErrorDB = (err) => {
  const errors = Object.values(err.errors).map((el) => el.message);
  return new AppError(`Invalid input: ${errors.join(". ")}`, 400);
};

module.exports = (err, req, res, next) => {
  err.statusCode = err.statusCode || 500;
  err.status = err.status || "error";

  if (process.env.NODE_ENV === "development") {
    return res.status(err.statusCode).json({
      status: err.status,
      error: err,
      message: err.message,
      stack: err.stack,
    });
  }

  // Production
  let error = { ...err };
  error.message = err.message;

  if (err.name === "CastError") error = handleCastErrorDB(error);
  if (err.code === 11000) error = handleDuplicateFieldsDB(error);
  if (err.name === "ValidationError") error = handleValidationErrorDB(error);

  if (error.isOperational) {
    return res.status(error.statusCode).json({
      status: error.status,
      message: error.message,
    });
  }

  console.error("💥 ERROR:", err);
  res.status(500).json({
    status: "error",
    message: "Something went very wrong!",
  });
};
```

### server.js

```js
const mongoose = require("mongoose");
const dotenv = require("dotenv");

// Load environment variables BEFORE importing app
dotenv.config({ path: "./.env" });

const app = require("./app");

const PORT = process.env.PORT || 3000;
const DB = process.env.DATABASE_URL || "mongodb://localhost:27017/myapp";

// ─── Handle uncaught exceptions (synchronous bugs) ───
process.on("uncaughtException", (err) => {
  console.error("💥 UNCAUGHT EXCEPTION! Shutting down...");
  console.error(err.name, err.message);
  process.exit(1); // Exit immediately — app is in unreliable state
});

// ─── Connect to database and start server ─────────────
mongoose
  .connect(DB)
  .then(() => console.log("✅ Database connection successful"))
  .catch((err) => console.error("❌ Database connection failed:", err));

const server = app.listen(PORT, () => {
  console.log(`🚀 Server running on port ${PORT}`);
});

// ─── Handle unhandled promise rejections (async bugs) ──
process.on("unhandledRejection", (err) => {
  console.error("💥 UNHANDLED REJECTION! Shutting down...");
  console.error(err.name, err.message);
  // Gracefully close server, then exit
  server.close(() => {
    process.exit(1);
  });
});
```

### Understanding the Unhandled Error Catchers in server.js

```
┌──────────────────────────────────────────────────────────────┐
│             TWO SAFETY NETS IN server.js                      │
├──────────────────────────────────────────────────────────────┤
│                                                               │
│  1. uncaughtException                                         │
│     ─────────────────                                         │
│     Catches: Synchronous errors that were never caught        │
│     Example:  undefined.foo  (TypeError outside try-catch)    │
│     Action:   Log the error → Exit immediately (process.exit) │
│     Why exit? The process is in an undefined state.            │
│               Continuing could cause unpredictable behavior.  │
│                                                               │
│  2. unhandledRejection                                        │
│     ────────────────────                                      │
│     Catches: Promise rejections that were never .catch()-ed   │
│     Example:  await failingFunction() without try-catch       │
│     Action:   Log the error → Close server → Exit             │
│     Why close first? Let pending requests finish before dying. │
│                                                               │
│  These are LAST RESORTS. Your code should handle errors       │
│  properly. These catch the ones that slip through.            │
│                                                               │
└──────────────────────────────────────────────────────────────┘
```

## Common Patterns and Best Practices

### 1. Consistent Response Format

Always send responses in a consistent format so the frontend knows what to expect:

```js
// ✅ GOOD: Consistent structure
// Success response
{
  "status": "success",
  "data": {
    "user": { "name": "Alice", "email": "alice@example.com" }
  }
}

// Error response
{
  "status": "fail",       // or "error"
  "message": "User not found"
}
```

### 2. Use Environment Variables for Sensitive Data

```bash
# .env file — NEVER commit this to Git!
NODE_ENV=development
PORT=3000
DATABASE_URL=mongodb://localhost:27017/myapp
JWT_SECRET=my-super-secret-key-12345
JWT_EXPIRES_IN=90d
```

```js
// Access with process.env
const dbUrl = process.env.DATABASE_URL;
```

### 3. Separate Concerns Strictly

```
✅ DO:
  • Model  → Database logic ONLY
  • Controller → Request handling ONLY
  • Routes → URL mapping ONLY
  • Middleware → Cross-cutting concerns (auth, logging, errors)

❌ DON'T:
  • Database queries in route files
  • HTTP response logic in models
  • Business logic scattered across routes and models
```

### 4. Use Descriptive Error Messages

```js
// ❌ BAD
next(new AppError("Error", 400));
next(new AppError("Not found", 404));
next(new AppError("Invalid", 422));

// ✅ GOOD
next(new AppError("Please provide both name and email", 400));
next(new AppError("No tour found with ID: 507f1f77bcf86cd799439011", 404));
next(new AppError("Email format is invalid. Expected: user@example.com", 422));
```

### 5. Always Handle 404 Routes

```js
// This should be AFTER all your route definitions
// but BEFORE the error handler
app.all("*", (req, res, next) => {
  next(new AppError(`Cannot find ${req.originalUrl} on this server`, 404));
});
```
