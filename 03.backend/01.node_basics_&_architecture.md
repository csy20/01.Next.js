# Node.js Basics & Architecture


## 🎯 What is Node.js?

### 🧠 The Problem Before Node.js

Before Node.js, JavaScript could **only** run inside a web browser. Every browser has a built-in JavaScript engine:

| Browser        | JS Engine          |
| -------------- | ------------------ |
| Google Chrome  | V8                 |
| Mozilla Firefox| SpiderMonkey       |
| Safari         | JavaScriptCore     |
| Microsoft Edge | V8 (Chromium-based)|

If you wrote JavaScript, you **had** to open a browser to run it. You couldn't use JavaScript to build a server, read files from your computer, or interact with a database. JavaScript was purely a **client-side** (browser) language.

### 🌟 The Birth of Node.js

In **2009**, a developer named **Ryan Dahl** had an idea: *"What if we took the V8 engine out of Chrome and wrapped it with C++ code that gives JavaScript access to the operating system?"*

That's exactly what Node.js is:

> **Node.js** is a **runtime environment** that lets you run JavaScript **outside** the browser — on servers, your laptop, IoT devices, or anywhere.

### 📝 What "Runtime Environment" Means

Think about it like this:

```
JavaScript (the language)  →  is like an actor
Browser (Chrome, Firefox)  →  is like a movie theater (one stage to perform)
Node.js                    →  is like a TV studio (a completely different stage)
```

The "actor" (JavaScript) is the same, but the **stage** (runtime) determines what props and tools are available:

| Feature                  | Browser Runtime     | Node.js Runtime      |
| ------------------------ | ------------------- | -------------------- |
| DOM manipulation         | ✅ Yes              | ❌ No                |
| `window` object          | ✅ Yes              | ❌ No                |
| `document` object        | ✅ Yes              | ❌ No                |
| File system access       | ❌ No               | ✅ Yes (`fs` module) |
| HTTP server creation     | ❌ No               | ✅ Yes (`http` module)|
| `process` object         | ❌ No               | ✅ Yes               |
| `global` object          | ❌ (uses `window`)  | ✅ Yes               |
| Database access          | ❌ No               | ✅ Yes               |

### 🏗️ Node.js Architecture — The Big Picture

Here's how Node.js is built, layer by layer:

```
┌─────────────────────────────────────────────────┐
│              Your JavaScript Code                │
├─────────────────────────────────────────────────┤
│              Node.js APIs                        │
│   (fs, http, path, crypto, os, events, etc.)     │
├──────────────────────┬──────────────────────────┤
│       V8 Engine      │        libuv             │
│  (Compiles JS → C++) │  (Async I/O, Event Loop) │
├──────────────────────┴──────────────────────────┤
│          Operating System (Linux/Mac/Win)         │
└─────────────────────────────────────────────────┘
```

Node.js sits on top of **two** foundational pillars:

1. **V8 Engine** — Converts JavaScript into machine code the CPU can execute
2. **libuv** — A C library that handles asynchronous I/O and the Event Loop

## ⚙️ The V8 Engine — The Brain Behind Node.js

### 🎯 What is V8?

V8 is an **open-source JavaScript engine** built by Google, written in **C++**. It was originally created for Google Chrome, but Node.js uses it to run JavaScript outside the browser.

### 🧠 What Does "Engine" Mean?

Your computer's CPU doesn't understand JavaScript. It only understands **machine code** (binary: 0s and 1s). The V8 engine's job is to **translate** JavaScript into machine code.

```
Your JS Code  →  V8 Engine  →  Machine Code  →  CPU executes it
```

### 🔄 How V8 Compiles JavaScript — Step by Step

V8 doesn't just interpret JavaScript line-by-line like older engines. It uses a technique called **Just-In-Time (JIT) Compilation**, which is a hybrid of interpretation and compilation.

#### Step 1: Parsing

When V8 receives your JavaScript code, it first **parses** it:

```
Source Code → Tokenizer → Tokens → Parser → Abstract Syntax Tree (AST)
```

**Example:**

```js
const greeting = "Hello, Node!";
```

The **Tokenizer** (also called Lexer) breaks this into tokens:

```
[const] [greeting] [=] ["Hello, Node!"] [;]
```

The **Parser** takes these tokens and builds an **AST** (Abstract Syntax Tree), which is a tree representation of your code:

```
VariableDeclaration
├── kind: "const"
└── declarations
    └── VariableDeclarator
        ├── id: Identifier (name: "greeting")
        └── init: Literal (value: "Hello, Node!")
```

#### Step 2: Ignition (The Interpreter)

V8 has a component called **Ignition**. It takes the AST and converts it into **bytecode** — an intermediate, lower-level representation that's faster to execute than raw JavaScript but not yet full machine code.

```
AST  →  Ignition (Interpreter)  →  Bytecode
```

Bytecode is executed immediately. This is why JavaScript "starts fast" — it doesn't wait for full compilation before running.

#### Step 3: TurboFan (The Optimizing Compiler)

While Ignition is running bytecode, V8 **monitors** which functions are called frequently (called **"hot functions"**). When it identifies hot code, it sends it to **TurboFan**, the optimizing compiler.

TurboFan compiles the hot bytecode into highly optimized **machine code**:

```
Hot Bytecode  →  TurboFan (Compiler)  →  Optimized Machine Code
```

This machine code runs **much** faster than bytecode.

#### Step 4: Deoptimization

Sometimes TurboFan makes assumptions (e.g., "this variable is always a number"). If those assumptions are proven wrong at runtime, V8 **deoptimizes** — it throws away the optimized machine code and falls back to bytecode.

```
Optimized Code  →  Assumption Broken  →  Deoptimize  →  Back to Bytecode
```

### 🏗️ The Full V8 Pipeline

```
┌────────────────────────────────────────────────────────┐
│                    JavaScript Source Code               │
│                          │                             │
│                     ┌────▼─────┐                       │
│                     │  Parser  │                       │
│                     └────┬─────┘                       │
│                          │ AST                         │
│                     ┌────▼─────┐                       │
│                     │ Ignition │ (Interpreter)          │
│                     └────┬─────┘                       │
│                          │ Bytecode                    │
│                          │                             │
│              ┌───────────┤ (profiles hot functions)     │
│              │           │                             │
│         ┌────▼─────┐    │                              │
│         │ TurboFan │    │ (Optimizing Compiler)        │
│         └────┬─────┘    │                              │
│              │           │                             │
│    Optimized Machine     │                             │
│         Code             │                             │
│              │           │                             │
│              └───────────┘                             │
│         (deoptimize if assumptions break)               │
└────────────────────────────────────────────────────────┘
```

### ⚡ Why V8 Matters for Node.js Performance

Because of JIT compilation, Node.js is **extremely fast** for a dynamically-typed language. V8 constantly optimizes your code while it's running — it's not just a dumb interpreter.

**Performance tip:** Write consistent code. If a function always receives numbers, V8 can optimize it. If sometimes it gets a number and sometimes a string, V8 has to deoptimize.

```js
// ✅ Good — V8 can optimize (consistent types)
function add(a, b) {
  return a + b;
}
add(1, 2);
add(3, 4);
add(5, 6);

// ❌ Bad — V8 will deoptimize (mixed types)
add(1, 2);
add("hello", "world");  // type changes → deoptimization
```

### 🗑️ V8's Memory Management — Garbage Collection

V8 automatically manages memory using a **Garbage Collector (GC)**. When objects are no longer referenced, V8 reclaims that memory.

V8 divides memory (the **heap**) into two areas:

| Area         | Purpose                                 | GC Strategy    |
| ------------ | --------------------------------------- | -------------- |
| Young Space  | Short-lived objects (most objects)       | Scavenge (fast)|
| Old Space    | Long-lived objects that survived GC      | Mark-Sweep-Compact (thorough) |

```
Heap Memory
├── Young Generation (Small, collected frequently)
│   ├── Semi-space "From"
│   └── Semi-space "To"
└── Old Generation (Large, collected less often)
    ├── Old Pointer Space
    ├── Old Data Space
    ├── Large Object Space
    └── Code Space
```

**Scavenge (Young Generation):** New objects are allocated in the "From" space. When it fills up, live objects are copied to the "To" space, and the "From" space is wiped clean. Then the two spaces swap roles.

**Mark-Sweep-Compact (Old Generation):** V8 marks all reachable objects, sweeps (frees) unreachable ones, and optionally compacts memory to reduce fragmentation.

## 🧵 Single-Threaded Nature of Node.js

### 📝 What is a "Thread"?

A **thread** is the smallest unit of execution in a CPU. Think of it as a **worker** who can execute one task at a time.

```
Thread = A single worker who follows instructions sequentially
```

### 🔀 Multi-Threaded Languages (Traditional Approach)

Languages like Java, C#, and Python (with threading) create **multiple threads** to handle multiple tasks simultaneously:

```
Server receives 4 requests:

Thread 1: → Handle Request A → Read file → Respond
Thread 2: → Handle Request B → Query DB → Respond
Thread 3: → Handle Request C → Read file → Respond
Thread 4: → Handle Request D → Query DB → Respond
```

Each request gets its own thread. With 10,000 requests, you need ~10,000 threads. Each thread consumes **memory** (typically 1-2 MB per thread), so this approach has limits:

```
10,000 threads × 2 MB = ~20 GB of RAM just for threads!
```

This is the **C10K problem** — handling 10,000 concurrent connections was a challenge.

### 🚀 Node.js Approach: Single-Threaded with Non-Blocking I/O

Node.js uses a **single main thread** for executing JavaScript. But it doesn't block and wait on slow operations. Instead, it **delegates** them and moves on.

#### The Restaurant Analogy

Imagine two restaurants:

**Multi-Threaded Restaurant (Traditional):**
- 100 customers? Hire 100 waiters.
- Each waiter takes one order, goes to the kitchen, **stands there waiting**, brings the food back.
- Expensive! You need many waiters just standing around.

**Single-Threaded Restaurant (Node.js):**
- 100 customers? **One very fast waiter**.
- The waiter takes Order 1, passes it to the kitchen, **immediately** goes to take Order 2.
- When the kitchen signals "Order 1 is ready!", the waiter delivers it.
- The waiter is **never idle** — always taking or delivering orders.

```
Traditional (Multi-Threaded):
Waiter 1: Take order → Wait in kitchen → Serve food → (idle until next customer)
Waiter 2: Take order → Wait in kitchen → Serve food → (idle until next customer)
Waiter 3: Take order → Wait in kitchen → Serve food → (idle until next customer)

Node.js (Single-Threaded):
Waiter: Take order 1 → Pass to kitchen →
        Take order 2 → Pass to kitchen →
        Take order 3 → Pass to kitchen →
        Kitchen says "Order 1 ready!" → Serve order 1 →
        Kitchen says "Order 3 ready!" → Serve order 3 →
        Kitchen says "Order 2 ready!" → Serve order 2
```

### ⚡ Blocking vs. Non-Blocking Code

**Blocking (Synchronous):** The thread stops and waits until the operation completes.

```js
const fs = require('fs');

// ❌ BLOCKING — the entire thread stops here until the file is fully read
console.log("Before reading");
const data = fs.readFileSync('/path/to/large-file.txt', 'utf-8');
console.log("After reading");  // This line WAITS
console.log(data);
```

Output:
```
Before reading
(... waits 3 seconds while reading ...)
After reading
<file contents>
```

**Non-Blocking (Asynchronous):** The thread starts the operation, hands it off, and moves to the next line. When the operation finishes, a callback runs.

```js
const fs = require('fs');

// ✅ NON-BLOCKING — hands off the file read and continues immediately
console.log("Before reading");
fs.readFile('/path/to/large-file.txt', 'utf-8', (err, data) => {
  console.log("File read complete!");
  console.log(data);
});
console.log("After reading");  // This runs IMMEDIATELY — doesn't wait!
```

Output:
```
Before reading
After reading          ← This prints FIRST (non-blocking!)
File read complete!    ← This prints LATER when the file is done
<file contents>
```

### 🔍 "But Wait — If It's Single-Threaded, How Does It Handle File Reads / Network Calls?"

Great question! Node.js is single-threaded **for your JavaScript code**, but behind the scenes, it uses **libuv**, which maintains a **thread pool** (default: 4 threads) for heavy operations:

```
┌─────────────────────────────────────────────┐
│         Node.js (Your JavaScript)            │
│              SINGLE THREAD                   │
│                    │                         │
│              Delegates to libuv              │
│                    │                         │
│    ┌───────────────┼───────────────┐         │
│    │               │               │         │
│    ▼               ▼               ▼         │
│ Thread Pool    OS Async APIs    Network I/O  │
│ (4 threads)    (epoll/kqueue)   (OS handles) │
│ - fs ops       - TCP/UDP                     │
│ - DNS lookup   - Pipes                       │
│ - Crypto       - Timers                      │
│ - Zlib                                       │
└─────────────────────────────────────────────┘
```

So Node.js is more accurately described as:

> **Single-threaded for JavaScript execution, but multi-threaded under the hood for I/O operations.**

## 🔄 The Event Loop — The Heart of Node.js

### 🎯 What Is the Event Loop?

The Event Loop is the mechanism that allows Node.js to perform non-blocking I/O operations despite being single-threaded. It continuously checks: *"Is there any work to do?"* and processes ready tasks.

Think of it like a **traffic controller** at an intersection:

```
The Event Loop is a traffic controller that:
1. Checks if there are any cars (callbacks) waiting
2. Lets them through in a specific order
3. Goes around and around, checking each lane (phase)
4. Stops only when there are no more cars anywhere
```

### 📝 The Event Loop Phases — In Detail

The Event Loop has **6 phases**, and each phase has a **FIFO queue** of callbacks to execute:

```
   ┌───────────────────────────┐
┌─►│         Timers             │  ← setTimeout, setInterval callbacks
│  └─────────────┬─────────────┘
│  ┌─────────────▼─────────────┐
│  │    Pending Callbacks       │  ← I/O callbacks deferred from previous loop
│  └─────────────┬─────────────┘
│  ┌─────────────▼─────────────┐
│  │      Idle / Prepare        │  ← Internal use only (Node internals)
│  └─────────────┬─────────────┘
│  ┌─────────────▼─────────────┐
│  │          Poll              │  ← Retrieve new I/O events; execute I/O callbacks
│  └─────────────┬─────────────┘
│  ┌─────────────▼─────────────┐
│  │          Check             │  ← setImmediate() callbacks
│  └─────────────┬─────────────┘
│  ┌─────────────▼─────────────┐
│  │     Close Callbacks        │  ← socket.on('close'), etc.
│  └─────────────┬─────────────┘
│                │
└────────────────┘  (loop back to Timers)
```

### 🔍 Phase-by-Phase Breakdown

#### Phase 1: Timers

Executes callbacks scheduled by `setTimeout()` and `setInterval()`.

**Important:** Timers specify a *minimum* delay, not an exact delay. If the Event Loop is busy in another phase, the timer callback will be delayed.

```js
console.log("Start");

setTimeout(() => {
  console.log("Timer 1 - 0ms");
}, 0);

setTimeout(() => {
  console.log("Timer 2 - 100ms");
}, 100);

console.log("End");
```

Output:
```
Start
End
Timer 1 - 0ms        ← Even 0ms timers run AFTER synchronous code
Timer 2 - 100ms      ← Runs after ~100ms (at minimum)
```

#### Phase 2: Pending Callbacks

Executes I/O callbacks that were **deferred** to the next loop iteration. For example, certain TCP error callbacks.

This phase handles system-level callbacks like `ECONNREFUSED` errors when trying to connect to a TCP socket.

#### Phase 3: Idle / Prepare

Used **internally** by Node.js only. You don't interact with this phase directly.

#### Phase 4: Poll

This is the **most important** phase. It does two things:

1. **Calculates** how long it should block and wait for I/O
2. **Processes** events in the poll queue (I/O callbacks)

If there are **no timers scheduled**, the poll phase will:
- If the poll queue is **not empty**: iterate through callbacks and execute them synchronously
- If the poll queue **is empty**:
  - If `setImmediate()` callbacks exist → move to Check phase
  - If not → wait here for callbacks to be added, then execute them immediately

```js
const fs = require('fs');

// The callback from fs.readFile ends up in the Poll phase
fs.readFile('/etc/passwd', (err, data) => {
  console.log("File read callback - Poll phase");
});
```

#### Phase 5: Check

Executes `setImmediate()` callbacks. `setImmediate()` is designed to execute **after** the Poll phase completes.

```js
const fs = require('fs');

fs.readFile('/etc/passwd', () => {
  // Inside an I/O callback...
  
  setTimeout(() => {
    console.log("setTimeout");  // Timers phase (next iteration)
  }, 0);

  setImmediate(() => {
    console.log("setImmediate"); // Check phase (current iteration)
  });
});
```

Output (always in this order inside I/O callbacks):
```
setImmediate     ← Check phase comes right after Poll
setTimeout       ← Timers phase is in the NEXT iteration
```

#### Phase 6: Close Callbacks

Handles close events like `socket.on('close', ...)`. If a socket or handle is closed abruptly, the `'close'` event will be emitted in this phase.

```js
const net = require('net');

const server = net.createServer((socket) => {
  socket.on('close', () => {
    console.log("Socket closed - Close Callbacks phase");
  });
});
```

### ⚡ Microtasks: process.nextTick() & Promises

**Outside** of the 6 phases, there are **microtask queues** that run **between** every phase (and even between every callback within a phase):

```
┌──────────────────────────────────┐
│   Between EVERY phase AND        │
│   between EVERY callback:        │
│                                  │
│   1. process.nextTick() queue    │  ← Runs FIRST (highest priority)
│   2. Promise microtask queue     │  ← Runs SECOND
└──────────────────────────────────┘
```

#### process.nextTick()

`process.nextTick()` fires **immediately** after the current operation completes, **before** the Event Loop continues to the next phase.

```js
console.log("Start");

process.nextTick(() => {
  console.log("nextTick");
});

console.log("End");
```

Output:
```
Start
End
nextTick    ← Runs after synchronous code but BEFORE any phase
```

#### Promise .then() / async-await

Promises are microtasks too, but they run **after** `process.nextTick()`:

```js
console.log("1. Start");

setTimeout(() => console.log("5. setTimeout"), 0);

Promise.resolve().then(() => console.log("3. Promise"));

process.nextTick(() => console.log("2. nextTick"));

console.log("4. End");
```

Wait — let me renumber the actual output:

```
1. Start
4. End
2. nextTick       ← process.nextTick runs first (microtask)
3. Promise        ← Promise runs second (microtask)
5. setTimeout     ← Timers phase runs last
```

**Execution order priority:**
```
Synchronous code
    ↓
process.nextTick() queue
    ↓
Promise microtask queue
    ↓
Event Loop phases (Timers → Pending → Idle → Poll → Check → Close)
```

### 🎨 Complete Event Loop Example

```js
const fs = require('fs');

console.log("1. Synchronous - Start");

setTimeout(() => {
  console.log("6. setTimeout (0ms) - Timers phase");
}, 0);

setImmediate(() => {
  console.log("7. setImmediate - Check phase");
});

fs.readFile(__filename, () => {
  console.log("5. File read - Poll phase");
  
  setTimeout(() => {
    console.log("9. setTimeout inside I/O - Timers phase (next tick)");
  }, 0);
  
  setImmediate(() => {
    console.log("8. setImmediate inside I/O - Check phase");
  });
  
  process.nextTick(() => {
    console.log("  → nextTick inside I/O callback");
  });
});

process.nextTick(() => {
  console.log("3. nextTick - Microtask");
});

Promise.resolve().then(() => {
  console.log("4. Promise - Microtask");
});

console.log("2. Synchronous - End");
```

Output:
```
1. Synchronous - Start
2. Synchronous - End
3. nextTick - Microtask
4. Promise - Microtask
6. setTimeout (0ms) - Timers phase       (order of 6,7 is non-deterministic*)
7. setImmediate - Check phase
5. File read - Poll phase
  → nextTick inside I/O callback
8. setImmediate inside I/O - Check phase
9. setTimeout inside I/O - Timers phase (next tick)
```

> \* Note: When `setTimeout(fn, 0)` and `setImmediate()` are called from the **main module** (not inside an I/O callback), the order is **non-deterministic** — it depends on process performance. Inside an I/O callback, `setImmediate()` always fires first.

### 🗺️ Visual Summary of the Event Loop

```
Your Code Runs (Synchronous)
         │
         ▼
┌─ Microtask Queue ──────────────────┐
│  process.nextTick() callbacks      │  ← Drain completely
│  Promise .then() callbacks         │  ← Drain completely
└────────────────────────────────────┘
         │
         ▼
┌─ Event Loop ───────────────────────────────────────┐
│                                                     │
│  ┌─ Timers ──────────┐   Between each phase AND    │
│  │ setTimeout         │   between each callback:    │
│  │ setInterval        │   → Drain nextTick queue    │
│  └────────┬───────────┘   → Drain Promise queue     │
│           ▼                                         │
│  ┌─ Pending Callbacks ┐                             │
│  │ Deferred I/O       │                             │
│  └────────┬───────────┘                             │
│           ▼                                         │
│  ┌─ Poll ─────────────┐                             │
│  │ I/O callbacks      │  (may block here waiting)   │
│  │ fs.readFile, etc.  │                             │
│  └────────┬───────────┘                             │
│           ▼                                         │
│  ┌─ Check ────────────┐                             │
│  │ setImmediate       │                             │
│  └────────┬───────────┘                             │
│           ▼                                         │
│  ┌─ Close Callbacks ──┐                             │
│  │ socket.on('close') │                             │
│  └────────┬───────────┘                             │
│           │                                         │
│           └──── Loop back to Timers ────────────────│
│                                                     │
└─────────────────────────────────────────────────────┘
         │
         ▼
   Process exits when no more work
```

## 📦 Setting Up package.json & Understanding npm

### 🎯 What is npm?

**npm** (Node Package Manager) is two things:

1. **A command-line tool** — comes installed with Node.js, used to install/manage packages
2. **A registry** — a massive online database of open-source JavaScript packages (https://www.npmjs.com)

Think of npm like an **app store** for Node.js:

```
App Store (Phone)  ↔  npm Registry (Node.js)
App                ↔  Package (library/module)
Install App        ↔  npm install <package>
```

### ✅ Verifying Your Installation

```bash
# Check Node.js version
node --version      # e.g., v20.11.0

# Check npm version
npm --version       # e.g., 10.2.4
```

### 📝 What is package.json?

`package.json` is a **JSON file** at the root of every Node.js project. It's the **identity card** of your project — it describes:

- What's the project name and version?
- What packages (dependencies) does it need?
- What scripts can you run?
- What's the entry point?
- Who wrote it?
- What license is it under?

> Every Node.js project — from a tiny script to a massive application — should have a `package.json`.

### 🚀 Creating package.json

#### Method 1: Interactive Setup

```bash
# Creates package.json with interactive prompts
npm init
```

It asks you questions:

```
package name: (my-project)
version: (1.0.0)
description: My awesome Node.js project
entry point: (index.js)
test command: jest
git repository:
keywords: nodejs, api, backend
author: Your Name
license: (ISC) MIT
```

#### Method 2: Quick Setup (Accept All Defaults)

```bash
# Creates package.json with all defaults — no questions asked
npm init -y
```

This generates:

```json
{
  "name": "my-project",
  "version": "1.0.0",
  "description": "",
  "main": "index.js",
  "scripts": {
    "test": "echo \"Error: no test specified\" && exit 1"
  },
  "keywords": [],
  "author": "",
  "license": "ISC"
}
```

### 🔍 Understanding Every Field in package.json

Here's a comprehensive `package.json` with explanations:

```jsonc
{
  // ───── BASIC INFO ─────
  "name": "my-awesome-api",          // Project name (lowercase, no spaces)
  "version": "1.0.0",                // Semantic versioning (major.minor.patch)
  "description": "A REST API",       // Short description
  "author": "John Doe <john@example.com>",
  "license": "MIT",                  // License type
  "keywords": ["api", "express"],    // Search keywords on npm

  // ───── ENTRY POINT ─────
  "main": "index.js",                // Entry point for CommonJS (require)
  "module": "index.mjs",             // Entry point for ES Modules (import)

  // ───── SCRIPTS ─────
  "scripts": {
    "start": "node index.js",                    // npm start
    "dev": "nodemon index.js",                    // npm run dev
    "build": "tsc",                               // npm run build
    "test": "jest",                               // npm test
    "lint": "eslint .",                           // npm run lint
    "prestart": "echo 'About to start!'",         // Runs BEFORE npm start
    "poststart": "echo 'Started!'"                // Runs AFTER npm start
  },

  // ───── DEPENDENCIES ─────
  "dependencies": {
    "express": "^4.18.2",            // Production dependencies
    "mongoose": "~7.6.0",
    "dotenv": "16.3.1"
  },
  "devDependencies": {
```

### 🏷️ Semantic Versioning (SemVer) — Explained

Every package follows the **MAJOR.MINOR.PATCH** pattern:

```
  1  .  2  .  3
  │     │     │
  │     │     └── PATCH: Bug fixes (backwards-compatible)
  │     └──────── MINOR: New features (backwards-compatible)
  └────────────── MAJOR: Breaking changes (NOT backwards-compatible)
```

**Version symbols in package.json:**

| Symbol  | Meaning                        | Example             | Allows            |
| ------- | ------------------------------ | ------------------- | ----------------- |
| `^`     | Compatible with minor version  | `^4.18.2`           | `4.18.2` to `4.x.x` (not `5.0.0`) |
| `~`     | Compatible with patch version  | `~4.18.2`           | `4.18.2` to `4.18.x` (not `4.19.0`) |
| (none)  | Exact version only             | `4.18.2`            | Only `4.18.2`     |
| `*`     | Any version                    | `*`                 | Any available      |
| `>=`    | Greater than or equal          | `>=4.18.0`          | `4.18.0` and above |

```
Caret ^  →  "I trust minor updates" (most common, default)
Tilde ~  →  "I only trust patch fixes"
None     →  "I don't trust anyone, lock it"
```

## 📦 CommonJS vs ES Modules

### 🧠 Why Do We Need Modules?

Without modules, all your code would be in **one giant file**. Modules let you split code into **separate files**, each with its own scope, and import/export functionality between them.

```
Without Modules:         With Modules:
┌──────────────┐         ┌──────────┐  ┌──────────┐  ┌──────────┐
│ app.js       │         │ server.js│  │ routes.js│  │ utils.js │
│              │         │          │  │          │  │          │
│ (5000 lines) │   →     │ imports  │  │ imports  │  │ exports  │
│              │         │ from     │  │ from     │  │ helper   │
│              │         │ routes   │  │ utils    │  │ functions│
└──────────────┘         └──────────┘  └──────────┘  └──────────┘
```

Node.js supports **two** module systems:

| System      | Syntax                       | File Extension | Introduced |
| ----------- | ---------------------------- | -------------- | ---------- |
| CommonJS    | `require()` / `module.exports` | `.js` or `.cjs`   | 2009 (original) |
| ES Modules  | `import` / `export`           | `.mjs` or `.js` (with config) | 2015 (standard) |

### 📋 CommonJS (CJS) — The Original Node.js Way

CommonJS was the **first** module system in Node.js. It was created because JavaScript had no built-in module system in 2009.

#### Exporting in CommonJS

**Single export:**

```js
// math.js — exporting a single thing

function add(a, b) {
  return a + b;
}

// The entire module.exports is replaced with this one function
module.exports = add;
```

**Multiple exports:**

```js
// math.js — exporting multiple things

function add(a, b) {
  return a + b;
}

function subtract(a, b) {
  return a - b;
}

function multiply(a, b) {
  return a * b;
}

// Option 1: Assign an object to module.exports
module.exports = {
  add,
  subtract,
  multiply
};

// Option 2: Add properties one by one (same result)
// module.exports.add = add;
// module.exports.subtract = subtract;
// module.exports.multiply = multiply;

// Option 3: Use the shorthand "exports" (alias for module.exports)
// exports.add = add;
// exports.subtract = subtract;
// exports.multiply = multiply;
```

#### Importing in CommonJS

```js
// app.js

// Importing a single default export
const add = require('./math');
console.log(add(2, 3)); // 5

// Importing multiple exports
const { add, subtract, multiply } = require('./math');
console.log(add(2, 3));       // 5
console.log(subtract(10, 4)); // 6
console.log(multiply(3, 4));  // 12

// Importing the whole module as an object
const math = require('./math');
console.log(math.add(2, 3));  // 5
```

#### How CommonJS Works Under the Hood

When you `require()` a module, Node.js does this behind the scenes:

```js
// What you write:
const math = require('./math');

// What Node.js actually does:

// 1. RESOLVES the file path
//    './math' → '/home/user/project/math.js'

// 2. LOADS the file content from disk

// 3. WRAPS the module in a function (called the Module Wrapper):
(function(exports, require, module, __filename, __dirname) {
  // ← your module code runs inside this wrapper
  
  function add(a, b) {
    return a + b;
  }
  
  module.exports = { add };
  
});

// 4. EXECUTES the wrapper function

// 5. CACHES the module (won't re-execute on subsequent requires)

// 6. RETURNS module.exports
```

The **Module Wrapper** is why:
- Each file has its own scope (variables don't leak)
- `__filename` and `__dirname` are available
- `exports`, `require`, and `module` exist in every file

#### CommonJS Key Characteristics

```
✅ Synchronous loading (loads file → executes → returns)
✅ Loaded at RUNTIME (can use require() inside if-statements)
✅ Cached after first load (singleton pattern)
✅ Works everywhere in Node.js
```

```js
// ✅ CommonJS allows dynamic/conditional imports
if (process.env.NODE_ENV === 'production') {
  const logger = require('./production-logger');
} else {
  const logger = require('./dev-logger');
}
```

### 📦 ES Modules (ESM) — The Modern JavaScript Standard

ES Modules were introduced in **ES2015 (ES6)** as the official JavaScript module system. They work in browsers AND Node.js (since Node.js v12+).

#### Enabling ES Modules in Node.js

There are **two** ways to use ESM in Node.js:

**Method 1:** Set `"type": "module"` in `package.json`

```json
{
  "name": "my-project",
  "type": "module"
}
```

Now all `.js` files in the project use ES Modules.

**Method 2:** Use the `.mjs` file extension

```
my-project/
├── math.mjs       ← ES Module
├── utils.cjs      ← CommonJS
└── package.json
```

#### Exporting in ES Modules

**Named Exports:**

```js
// math.js (with "type": "module" in package.json)

// Named exports — you can have MANY per file
export function add(a, b) {
  return a + b;
}

export function subtract(a, b) {
  return a - b;
}

export const PI = 3.14159;

// Or export them all at once at the bottom:
// export { add, subtract, PI };
```

**Default Export:**

```js
// logger.js

// Default export — only ONE per file
class Logger {
  log(message) {
    console.log(`[LOG]: ${message}`);
  }
  
  error(message) {
    console.error(`[ERROR]: ${message}`);
  }
}

export default Logger;
```

**Mixing Named and Default Exports:**

```js
// utils.js

export default function mainFunction() {
  return "I'm the default";
}

export function helper1() {
  return "I'm a named export";
}

export function helper2() {
  return "I'm also a named export";
}
```

#### Importing in ES Modules

```js
// app.js

// Import named exports (use curly braces)
import { add, subtract, PI } from './math.js';  // ← MUST include .js extension!
console.log(add(2, 3));    // 5
console.log(PI);           // 3.14159

// Import default export (no curly braces, any name you want)
import Logger from './logger.js';
const log = new Logger();
log.log("Hello!");         // [LOG]: Hello!

// Import default AND named from the same file
import mainFunction, { helper1, helper2 } from './utils.js';

// Import everything as a namespace
import * as math from './math.js';
console.log(math.add(2, 3));  // 5
console.log(math.PI);         // 3.14159

// Rename imports (aliasing)
import { add as addition, subtract as sub } from './math.js';
console.log(addition(2, 3));  // 5
```

#### ES Modules Key Characteristics

```
✅ Asynchronous loading (non-blocking)
✅ Static analysis at COMPILE TIME (imports are hoisted)
✅ Tree-shaking support (dead code elimination)
✅ Works in browsers AND Node.js
✅ The official JavaScript standard
❌ Cannot use dynamic paths in import statements (at top level)
```

```js
// ❌ This is NOT allowed in ESM (static imports must be top-level)
if (condition) {
  import something from './module.js';  // SyntaxError!
}

// ✅ Use dynamic import() for conditional loading
if (condition) {
  const module = await import('./module.js');  // Returns a Promise
}
```

### 📦 CommonJS vs ES Modules — Complete Comparison

| Feature                  | CommonJS (`require`)              | ES Modules (`import`)            |
| ------------------------ | --------------------------------- | -------------------------------- |
| **Syntax (export)**      | `module.exports = ...`            | `export` / `export default`      |
| **Syntax (import)**      | `const x = require('...')`       | `import x from '...'`           |
| **Loading**              | **Synchronous** (blocking)        | **Asynchronous** (non-blocking) |
| **When parsed**          | **Runtime** (dynamic)             | **Compile time** (static)       |
| **File extension**       | `.js` or `.cjs`                   | `.mjs` or `.js` (with config)   |
| **Conditional import**   | ✅ `require()` anywhere           | ❌ Top-level only (use `import()` for dynamic) |
| **Tree-shaking**         | ❌ Not supported                  | ✅ Supported (bundlers can remove unused code) |
| **Top-level `await`**    | ❌ Not supported                  | ✅ Supported                    |
| **`this` at top level**  | `module.exports` (an object)      | `undefined`                      |
| **Browser support**      | ❌ No (needs bundler)             | ✅ Yes (native `<script type="module">`) |
| **Caching**              | ✅ Cached after first `require()` | ✅ Cached (singleton per URL)   |
| **Circular deps**        | Returns partial export             | Live bindings (always up-to-date)|

### 🏗️ Interoperability — Using CJS and ESM Together

#### Importing CommonJS from ES Modules

```js
// old-library.cjs (CommonJS)
module.exports = {
  greet: (name) => `Hello, ${name}!`
};

// app.mjs (ES Module)
import oldLib from './old-library.cjs';  // ✅ Default import works
console.log(oldLib.greet("World"));
```

#### Importing ES Modules from CommonJS

```js
// modern-lib.mjs (ES Module)
export function greet(name) {
  return `Hello, ${name}!`;
}

// app.cjs (CommonJS)
// ❌ This does NOT work:
// const { greet } = require('./modern-lib.mjs');

// ✅ Use dynamic import() instead:
async function main() {
  const { greet } = await import('./modern-lib.mjs');
  console.log(greet("World"));
}
main();
```

### 🎯 Which Should You Use?

```
┌─────────────────────────────────────────────────┐
│                                                  │
│   🟢 Use ES Modules (import/export) if:         │
│   • Starting a brand new project                 │
│   • Using a modern framework (Next.js, etc.)     │
│   • Writing TypeScript                           │
│   • Want tree-shaking and modern tooling         │
│   • Building for both Node.js and browsers       │
│                                                  │
│   🟡 Use CommonJS (require) if:                  │
│   • Working on an existing CommonJS project      │
│   • Required by a specific library or tool       │
│   • Need synchronous conditional loading         │
│                                                  │
│   📌 Industry Trend: ESM is the future.          │
│      New projects should default to ESM.         │
│                                                  │
└─────────────────────────────────────────────────┘
```
