## Why Authentication Matters

Imagine a library:

```
WITHOUT Authentication:                WITH Authentication:
──────────────────────                ──────────────────────

Anyone can walk in and:              You need a library card:
• Take any book                      • Prove who you are (login)
• Access private records             • Get a card (JWT token)
• Delete the entire catalog          • Show your card each visit
• Pretend to be the librarian        • Different cards = different access

Total chaos!                          Orderly and secure!
```

Authentication answers the question: **"WHO are you?"**

Without it, anyone can access any data, modify any record, or delete anything — your API would be completely open to abuse.

## Authentication vs Authorization

These are two **different** concepts that work together:

```
┌──────────────────────────────────────────────────────────────┐
│          AUTHENTICATION vs AUTHORIZATION                      │
├──────────────────────────────────────────────────────────────┤
│                                                               │
│  AUTHENTICATION (AuthN)         AUTHORIZATION (AuthZ)        │
│  ─────────────────────         ────────────────────           │
│  "WHO are you?"                "WHAT can you do?"            │
│                                                               │
│  • Verifying identity           • Checking permissions        │
│  • Login with email/password    • Role-based access control  │
│  • "Prove you are Alice"        • "Can Alice delete users?"  │
│                                                               │
│  Happens FIRST                  Happens AFTER authentication  │
│                                                               │
│  Example:                       Example:                      │
│  Showing your ID at the door    VIP section is only for VIPs │
│                                                               │
│  Middleware: protect            Middleware: restrictTo        │
│                                                               │
└──────────────────────────────────────────────────────────────┘
```

```
Real-World Analogy — Airport Security:
──────────────────────────────────────────────

1. AUTHENTICATION: Show your passport at check-in
   → "You are Alice Johnson" ✓

2. AUTHORIZATION: Show your boarding pass at the gate
   → "You have access to Gate B12, Economy Class"
   → "You do NOT have access to the First Class Lounge"
```

## Password Security — Why We Hash

### The Fundamental Rule

> **NEVER store passwords in plain text.** Not in your database, not in your logs, not anywhere. This is the #1 security rule.

```
❌ DISASTER: Storing plain text passwords

  users collection:
  ┌────────────────────────────────────────┐
  │ { email: "alice@...", password: "MyP@ss123" }  │
  │ { email: "bob@...",   password: "secret456" }  │
  │ { email: "eve@...",   password: "qwerty789" }  │
  └────────────────────────────────────────┘

  If a hacker gets access to your database:
  → They instantly have EVERYONE's passwords
  → Many users reuse passwords across sites
  → Hacker now has access to their email, bank, etc.

✅ CORRECT: Storing hashed passwords

  users collection:
  ┌─────────────────────────────────────────────────────────┐
  │ { email: "alice@...", password: "$2b$12$LJ3m4..." }    │
  │ { email: "bob@...",   password: "$2b$12$K9pQ5..." }    │
  │ { email: "eve@...",   password: "$2b$12$R8wN2..." }    │
  └─────────────────────────────────────────────────────────┘

  If a hacker gets access to your database:
  → They see meaningless strings
  → Hashes CANNOT be reversed back to passwords
  → Users' other accounts remain safe
```

---

### What is Hashing?

Hashing is a **one-way** mathematical function that converts any input into a fixed-length string of characters.

```
┌──────────────────────────────────────────────────────────────┐
│                    HASHING = ONE-WAY STREET                   │
├──────────────────────────────────────────────────────────────┤
│                                                               │
│  Input (any length)      →    Hash (fixed length)            │
│  ─────────────────            ──────────────────              │
│                                                               │
│  "password"              →    "5f4dcc3b5aa765d61d8327..."    │
│  "Password"              →    "dc647eb65e6711e155375..."     │
│  "password1"             →    "7c6a180b36896a65c4ae..."      │
│  "The quick brown fox"   →    "9e107d9d372bb6826bd81..."     │
│                                                               │
│  Key Properties:                                              │
│  ✓ Same input ALWAYS produces the same hash                  │
│  ✓ Tiny input change → COMPLETELY different hash             │
│  ✓ Fixed output length regardless of input length            │
│  ✗ CANNOT reverse: hash → input (one-way only!)             │
│                                                               │
│  "password"  →  "5f4dcc..."  ✓ (can compute)                │
│  "5f4dcc..." →  "password"   ✗ (IMPOSSIBLE)                  │
│                                                               │
└──────────────────────────────────────────────────────────────┘
```

---

### Hashing vs Encryption

These are NOT the same thing:

```
┌──────────────────────────┬───────────────────────────────┐
│       HASHING             │       ENCRYPTION              │
├──────────────────────────┼───────────────────────────────┤
│ ONE-WAY                  │ TWO-WAY                       │
│ Cannot be reversed       │ CAN be reversed (with key)    │
│                          │                               │
│ "hello" → "5d41402..."  │ "hello" + key → "x7Kp9..."   │
│ "5d41402..." → ???      │ "x7Kp9..." + key → "hello"   │
│                          │                               │
│ No key needed            │ Requires a secret key         │
│                          │                               │
│ Used for:                │ Used for:                     │
│ • Passwords              │ • Credit card numbers         │
│ • Data integrity checks  │ • Private messages            │
│ • Digital signatures     │ • Sensitive data at rest      │
│                          │                               │
│ Algorithm: bcrypt, SHA   │ Algorithm: AES, RSA           │
└──────────────────────────┴───────────────────────────────┘

For PASSWORDS, always use HASHING (not encryption).
We never need to get the original password back.
We only need to CHECK if a given password MATCHES the hash.
```

---

### What is bcrypt?

**bcrypt** is a password hashing algorithm specifically designed to be:

1. **Slow on purpose** — makes brute force attacks impractical
2. **Salted** — same password produces different hashes each time
3. **Configurable** — you control how slow it is (cost factor)

```
Why "slow" is a FEATURE, not a bug:
──────────────────────────────────────────────

  Fast hash (like MD5 or SHA-256):
  A hacker can try 10 BILLION passwords per second
  → 8-character password cracked in ~2 hours

  bcrypt (with 12 rounds):
  A hacker can try ~13 passwords per second
  → 8-character password cracked in ~770 MILLION years

  bcrypt is intentionally slow to make attacks impractical.
  Each login takes ~250ms (unnoticeable to users),
  but makes batch guessing impossibly slow for hackers.
```

---

### How bcrypt Works — Salt and Rounds

#### What is a Salt?

A **salt** is a random string added to the password BEFORE hashing. This ensures that even identical passwords produce different hashes.

```
Without Salt:
──────────────────────────────────────────────

  "password123" → hash("password123") → "abc123..."
  "password123" → hash("password123") → "abc123..."  ← SAME hash!

  If a hacker sees two users with the same hash,
  they know both users have the same password.
  They can also use pre-computed "rainbow tables"
  to look up common password hashes.

With Salt (bcrypt):
──────────────────────────────────────────────

  User 1: "password123" + salt="Xy7k" → hash("password123Xy7k") → "def456..."
  User 2: "password123" + salt="Mn3p" → hash("password123Mn3p") → "ghi789..."

  DIFFERENT hashes! Even though the passwords are the same.
  Rainbow tables are useless because every hash is unique.
  The salt is stored alongside the hash (it's not secret).
```

#### What are Rounds (Cost Factor)?

Rounds control how many times the hashing algorithm runs internally. More rounds = slower hashing = more secure.

```
Rounds   Iterations          Time per Hash     Security
──────   ──────────          ──────────────    ────────
  10     2^10 = 1,024        ~100ms              Good
  11     2^11 = 2,048        ~200ms              Better
  12     2^12 = 4,096        ~250ms              Recommended ✓
  13     2^13 = 8,192        ~500ms              Strong
  14     2^14 = 16,384       ~1 second           Very Strong

  Recommended: 12 rounds
  • Fast enough for users (250ms per login)
  • Slow enough to stop attackers
  • Can increase as hardware gets faster
```

### Using bcrypt in Node.js

```bash
npm install bcryptjs
# We use bcryptjs (pure JavaScript) instead of bcrypt (requires C++ compilation)
# bcryptjs is easier to install and works everywhere
```

```js
const bcrypt = require("bcryptjs");

// ──── Hashing a Password ────────────────────────────

const password = "MySecurePassword123";

// Method 1: Two-step (generate salt, then hash)
const salt = await bcrypt.genSalt(12); // Generate salt with 12 rounds
const hash = await bcrypt.hash(password, salt); // Hash the password
console.log(hash);
// "$2b$12$LJ3m4xR6N7Kp5O8Q9R0S1eT2U3V4W5X6Y7Z8A9B0C1D2E3F4G5H6"

// Method 2: One-step (bcrypt auto-generates the salt)
const hash = await bcrypt.hash(password, 12); // 12 = number of rounds
console.log(hash);
// "$2b$12$K9pQ5rS7tU9vW1xY3zA5bCd7eF9gH1iJ3kL5mN7oP9qR1sT3uV5w"

// ──── Verifying a Password ──────────────────────────

const isCorrect = await bcrypt.compare("MySecurePassword123", hash);
console.log(isCorrect); // true ✓

const isWrong = await bcrypt.compare("WrongPassword", hash);
console.log(isWrong); // false ✗
```

### Anatomy of a bcrypt Hash

```
$2b$12$LJ3m4xR6N7Kp5O8Q9R0S1eT2U3V4W5X6Y7Z8A9B0C1D2E3F4G5H6
 │   │  │                      │
 │   │  │                      └── The actual hash (31 characters)
 │   │  └───────────────────────── The salt (22 characters)
 │   └──────────────────────────── Cost factor (12 rounds = 2^12 iterations)
 └──────────────────────────────── Algorithm version ($2b = bcrypt)

The salt is stored RIGHT INSIDE the hash string.
bcrypt.compare() knows how to extract the salt and re-hash
the candidate password to check if it matches.
```

### Hashing Passwords in Mongoose with Pre-Save Hooks

Instead of manually hashing in every controller, use a **pre-save middleware** on the User model:

```js
// models/User.js

const mongoose = require("mongoose");
const bcrypt = require("bcryptjs");

const userSchema = new mongoose.Schema({
  name: {
    type: String,
    required: [true, "Please provide your name"],
  },
  email: {
    type: String,
    required: [true, "Please provide your email"],
    unique: true,
    lowercase: true,
  },
  password: {
    type: String,
    required: [true, "Please provide a password"],
    minlength: [8, "Password must be at least 8 characters"],
    select: false, // NEVER include in queries by default
  },
  passwordConfirm: {
    type: String,
    required: [true, "Please confirm your password"],
    validate: {
      // Only works on .create() and .save()!
      validator: function (value) {
        return value === this.password;
      },
      message: "Passwords do not match",
    },
  },
  role: {
    type: String,
    enum: ["user", "admin"],
    default: "user",
  },
  passwordChangedAt: Date,
});

// ──── Pre-save: Hash password before saving ─────────
userSchema.pre("save", async function (next) {
  // Only hash if the password was actually modified (or is new)
  if (!this.isModified("password")) return next();

  // Hash the password with cost factor 12
  this.password = await bcrypt.hash(this.password, 12);

  // Delete passwordConfirm — we only needed it for validation
  // We don't want to persist it to the database
  this.passwordConfirm = undefined;

  next();
});

// ──── Instance method: Compare passwords ────────────
userSchema.methods.correctPassword = async function (candidatePassword) {
  // 'this.password' is not available because select: false
  // We need to explicitly select it when querying
  // candidatePassword = the password the user is trying to login with
  // this.password = the hashed password stored in the database
  return await bcrypt.compare(candidatePassword, this.password);
};

module.exports = mongoose.model("User", userSchema);
```

```
What Happens When a User Signs Up:
──────────────────────────────────────────────

  User sends: { name: "Alice", email: "alice@...",
                password: "MyPass123", passwordConfirm: "MyPass123" }
    │
    ▼
  1. Mongoose validates all fields
     • name is present? ✓
     • email is present and unique? ✓
     • password is at least 8 chars? ✓
     • passwordConfirm matches password? ✓
    │
    ▼
  2. Pre-save hook runs:
     • this.isModified("password") → true (it's new)
     • bcrypt.hash("MyPass123", 12) → "$2b$12$abc..."
     • this.password = "$2b$12$abc..."
     • this.passwordConfirm = undefined (deleted)
    │
    ▼
  3. Document saved to MongoDB:
     {
       name: "Alice",
       email: "alice@example.com",
       password: "$2b$12$abc...",  ← Hashed!
       role: "user"
       // passwordConfirm is NOT saved
     }
```

## JSON Web Tokens (JWT) — The Complete Guide

### What is a JWT?

A **JSON Web Token (JWT)** is a compact, URL-safe string that represents claims (pieces of information) about a user. Think of it as a **digital ID card** that your server issues to authenticated users.

```
┌──────────────────────────────────────────────────────────────┐
│                    JWT = DIGITAL ID CARD                       │
├──────────────────────────────────────────────────────────────┤
│                                                               │
│   Physical ID card:                                           │
│   ┌───────────────────────────────┐                          │
│   │ Name: Alice Johnson          │                           │
│   │ ID: 12345                    │                           │
│   │ Role: Employee               │                           │
│   │ Expires: 2025-12-31          │                           │
│   │ ────────────────────────     │                           │
│   │ [Official Stamp/Signature]   │ ← Proves it's genuine    │
│   └───────────────────────────────┘                          │
│                                                               │
│   JWT token:                                                  │
│   ┌───────────────────────────────┐                          │
│   │ { id: "507f1f...",           │                           │
│   │   iat: 1705312200,          │ ← Issued at               │
│   │   exp: 1713088200 }         │ ← Expires at              │
│   │ ────────────────────────     │                           │
│   │ [Cryptographic Signature]    │ ← Proves it's genuine    │
│   └───────────────────────────────┘                          │
│                                                               │
│   Both prove "I am who I say I am" to anyone who checks.     │
│                                                               │
└──────────────────────────────────────────────────────────────┘
```

### Why JWT? The Problem with Sessions

Before JWT, the main approach was **session-based authentication**:

```
Session-Based (traditional):              Token-Based (JWT):
──────────────────────────               ──────────────────

  Client sends login                      Client sends login
       ▼                                       ▼
  Server creates a SESSION                Server creates a TOKEN
  and stores it in memory                 (NOT stored on server)
       ▼                                       ▼
  Server sends session ID                 Server sends the token
  (stored in a cookie)                    (client stores it)
       ▼                                       ▼
  Every request sends the cookie          Every request sends the token
       ▼                                       ▼
  Server looks up the session             Server VERIFIES the token
  in its memory/database                  using its secret key
                                          (no lookup needed!)

Problems with Sessions:                   Advantages of JWT:
• Server must STORE every session        • Server stores NOTHING
• Doesn't scale well                     • Scales perfectly ✓
  (what if you have 5 servers?)          • Works across multiple servers
• Requires sticky sessions               • No database lookup needed
  or shared session store                • Works with mobile apps
• Tied to cookies (browser only)         • Works with any client
```

```
Session Scaling Problem:
──────────────────────────────────────────────

  With sessions (stored in server memory):

  Server 1: { session_abc: { userId: 1 } }
  Server 2: { }  ← doesn't have the session!
  Server 3: { }  ← doesn't have the session!

  If request goes to Server 2, user appears logged out!
  You need a shared session store (Redis) to fix this.

  With JWT (stored on client, verified with secret):

  Server 1: Verifies token with SECRET_KEY ✓
  Server 2: Verifies token with SECRET_KEY ✓
  Server 3: Verifies token with SECRET_KEY ✓

  ALL servers can verify the token independently!
  No shared storage needed. Just share the secret key.
```

### Anatomy of a JWT — The Three Parts

A JWT looks like this:

```
eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9.eyJpZCI6IjUwN2YxZjc3YmNmODZjZDc5OTQzOTAxMSIsImlhdCI6MTcwNTMxMjIwMCwiZXhwIjoxNzEzMDg4MjAwfQ.SflKxwRJSMeKKF2QT4fwpMeJf36POk6yJV_adQssw5c
```

It has **three parts** separated by dots (`.`):

```
Header.Payload.Signature
──────.───────.─────────

┌──────────────────────────────────────────────────────────────┐
│   PART 1: HEADER (metadata)                                   │
│   eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9                      │
│                                                               │
│   Decoded (Base64):                                           │
│   {                                                           │
│     "alg": "HS256",    ← Algorithm used for signing          │
│     "typ": "JWT"       ← Type of token                       │
│   }                                                           │
├──────────────────────────────────────────────────────────────┤
│   PART 2: PAYLOAD (the actual data/claims)                    │
│   eyJpZCI6IjUwN2YxZjc3...                                    │
│                                                               │
│   Decoded (Base64):                                           │
│   {                                                           │
│     "id": "507f1f77bcf86cd799439011",  ← User ID             │
│     "iat": 1705312200,                 ← Issued At           │
│     "exp": 1713088200                  ← Expires At          │
│   }                                                           │
│                                                               │
│   ⚠️ The payload is NOT encrypted!                            │
│   Anyone can decode it (it's just Base64).                    │
│   NEVER put secrets (passwords, credit cards) in the payload │
├──────────────────────────────────────────────────────────────┤
│   PART 3: SIGNATURE (proof of authenticity)                   │
│   SflKxwRJSMeKKF2QT4fwpMeJf36POk6yJV_adQssw5c              │
│                                                               │
│   Created by:                                                 │
│   HMACSHA256(                                                 │
│     base64(header) + "." + base64(payload),                  │
│     SECRET_KEY                                                │
│   )                                                           │
│                                                               │
│   The signature ensures:                                      │
│   ✓ The token was created by YOUR server (not forged)        │
│   ✓ The payload hasn't been tampered with                    │
│   ✓ Only someone with the SECRET_KEY can create valid tokens │
│                                                               │
└──────────────────────────────────────────────────────────────┘
```

### What Happens If Someone Tampers with a JWT?

```
Original token payload:
{ "id": "user123", "role": "user" }     Signature: valid ✓

Attacker changes the payload:
{ "id": "user123", "role": "admin" }    Signature: INVALID ✗

The server recalculates the signature using the SECRET_KEY.
The new signature doesn't match the one in the token.
→ Token is REJECTED! The attacker can't fake a valid signature
  because they don't know the SECRET_KEY.

This is why the SECRET_KEY must NEVER be shared or leaked.
```

### How JWT Authentication Works — Step by Step

```
The Complete Flow:
──────────────────────────────────────────────

1. USER SIGNS UP / LOGS IN
   Client: POST /api/login { email: "alice@...", password: "pass123" }
       │
       ▼
   Server:
   • Find user by email
   • Compare password with bcrypt
   • If valid, create a JWT:
     jwt.sign({ id: user._id }, SECRET, { expiresIn: '90d' })
   • Send token back to client
       │
       ▼
   Client receives: { token: "eyJhbGci..." }
   Client stores the token (localStorage, cookie, etc.)

2. USER MAKES A PROTECTED REQUEST
   Client: GET /api/profile
           Authorization: Bearer eyJhbGci...
       │
       ▼
   Server (protect middleware):
   • Extract token from Authorization header
   • Verify token: jwt.verify(token, SECRET)
   • If valid, decode the payload: { id: "507f...", iat: ..., exp: ... }
   • Find user by id from the payload
   • Attach user to req: req.user = user
   • Call next() → request reaches the controller
       │
       ▼
   Controller:
   • Access the authenticated user via req.user
   • Send back their data

3. TOKEN EXPIRES
   After 90 days (or whatever you set):
   • jwt.verify() throws "TokenExpiredError"
   • Server responds with 401 Unauthorized
   • Client must login again to get a new token
```

### Creating (Signing) a Token

```js
const jwt = require("jsonwebtoken");

// ──── Basic token creation ──────────────────────────

const token = jwt.sign(
  { id: user._id },         // Payload — data to embed in the token
  process.env.JWT_SECRET,    // Secret key — used to sign the token
  { expiresIn: "90d" }      // Options — token expires in 90 days
);

console.log(token);
// "eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9.eyJpZCI6..."
```

### What Happens When You Sign

```
jwt.sign({ id: "507f..." }, SECRET, { expiresIn: "90d" })
    │
    ├── 1. Create the Header:
    │      { "alg": "HS256", "typ": "JWT" }
    │      → Base64 encode → "eyJhbGci..."
    │
    ├── 2. Create the Payload:
    │      { "id": "507f...", "iat": 1705312200, "exp": 1713088200 }
    │      → Base64 encode → "eyJpZCI6..."
    │      (iat and exp are added automatically)
    │
    ├── 3. Create the Signature:
    │      HMACSHA256("eyJhbGci..." + "." + "eyJpZCI6...", SECRET)
    │      → "SflKxwRJ..."
    │
    └── 4. Combine all three:
           "eyJhbGci...eyJpZCI6...SflKxwRJ..."
```

### ExpiresIn Options

```js
jwt.sign(payload, secret, { expiresIn: "24h" });   // 24 hours
jwt.sign(payload, secret, { expiresIn: "60m" });   // 60 minutes
jwt.sign(payload, secret, { expiresIn: "30s" });   // 30 seconds (testing)
jwt.sign(payload, secret, { expiresIn: 7776000 }); // Seconds (90 days)
```

### Verifying a Token

```js
const jwt = require("jsonwebtoken");

// ──── Verify a token ────────────────────────────────

try {
  const decoded = jwt.verify(token, process.env.JWT_SECRET);
  console.log(decoded);
  // {
  //   id: "507f1f77bcf86cd799439011",
  //   iat: 1705312200,    ← Issued at (Unix timestamp)
  //   exp: 1713088200     ← Expires at (Unix timestamp)
  // }
} catch (error) {
  if (error.name === "JsonWebTokenError") {
    console.log("Invalid token! Someone tampered with it.");
  }
  if (error.name === "TokenExpiredError") {
    console.log("Token has expired! User must login again.");
  }
}
```

```
What jwt.verify() Checks:
──────────────────────────────────────────────

  1. Is the token a valid JWT format? (three parts, properly encoded)
  2. Was it signed with OUR secret key? (signature verification)
  3. Has it expired? (exp claim vs current time)
  4. Was it issued in the future? (iat check)

  If ANY check fails → throws an error
  If ALL checks pass → returns the decoded payload
```

## Building the Authentication System

### The User Model

```js
// models/User.js — Complete authentication-ready User model

const mongoose = require("mongoose");
const bcrypt = require("bcryptjs");
const crypto = require("crypto"); // Built-in Node.js module

const userSchema = new mongoose.Schema({
  name: {
    type: String,
    required: [true, "Please provide your name"],
    trim: true,
  },
  email: {
    type: String,
    required: [true, "Please provide your email"],
    unique: true,
    lowercase: true,
    match: [/^\S+@\S+\.\S+$/, "Please provide a valid email"],
  },
  role: {
    type: String,
    enum: ["user", "admin"],
    default: "user",
  },
  password: {
    type: String,
    required: [true, "Please provide a password"],
    minlength: [8, "Password must be at least 8 characters"],
    select: false, // Never include in query results
  },
  passwordConfirm: {
    type: String,
    required: [true, "Please confirm your password"],
    validate: {
      validator: function (val) {
        return val === this.password;
      },
      message: "Passwords do not match",
    },
  },
  passwordChangedAt: Date,
  passwordResetToken: String,
  passwordResetExpires: Date,
  isActive: {
    type: Boolean,
    default: true,
    select: false,
  },
});

// ──── Hash password before saving ───────────────────
userSchema.pre("save", async function (next) {
  if (!this.isModified("password")) return next();
  this.password = await bcrypt.hash(this.password, 12);
  this.passwordConfirm = undefined;
  next();
});

// ──── Set passwordChangedAt when password changes ───
userSchema.pre("save", function (next) {
  if (!this.isModified("password") || this.isNew) return next();
  this.passwordChangedAt = Date.now() - 1000;
  // Subtract 1 second to ensure the token is created AFTER
  // the password was changed (accounts for DB save delay)
  next();
});

// ──── Filter out inactive users from queries ────────
userSchema.pre(/^find/, function (next) {
  this.find({ isActive: { $ne: false } });
  next();
});

// ──── Instance Methods ──────────────────────────────

// Compare candidate password with stored hash
userSchema.methods.correctPassword = async function (candidatePassword) {
  return await bcrypt.compare(candidatePassword, this.password);
};

// Check if user changed password after the token was issued
userSchema.methods.changedPasswordAfter = function (JWTTimestamp) {
  if (this.passwordChangedAt) {
    const changedTimestamp = parseInt(
      this.passwordChangedAt.getTime() / 1000,
      10
    );
    return JWTTimestamp < changedTimestamp;
    // Returns true if password was changed AFTER token was issued
    // Meaning: the token is outdated!
  }
  return false; // Password was never changed
};

// Generate a random password reset token
userSchema.methods.createPasswordResetToken = function () {
  // Create a random token
  const resetToken = crypto.randomBytes(32).toString("hex");

  // Hash it and store in the database
  this.passwordResetToken = crypto
    .createHash("sha256")
    .update(resetToken)
    .digest("hex");

  // Set expiration (10 minutes)
  this.passwordResetExpires = Date.now() + 10 * 60 * 1000;

  // Return the UN-hashed token (to be sent via email)
  return resetToken;
};

module.exports = mongoose.model("User", userSchema);
```

### Login — Authenticating Existing Users

```js
exports.login = asyncHandler(async (req, res, next) => {
  const { email, password } = req.body;

  // 1. Check if email and password were provided
  if (!email || !password) {
    return next(new AppError("Please provide email and password", 400));
  }

  // 2. Find user by email AND include the password field
  //    (password has select: false, so we need +password)
  const user = await User.findOne({ email }).select("+password");

  // 3. Check if user exists AND if password is correct
  if (!user || !(await user.correctPassword(password, user.password))) {
    return next(new AppError("Incorrect email or password", 401));
    // Intentionally vague! Don't say "user not found" or "wrong password"
    // That would tell attackers which emails exist in your system
  }

  // 4. Everything is good — send the token
  createSendToken(user, 200, res);
});
```

```
Login Flow:

    │
    ▼
  2. Find user by email (with password included)
     User.findOne({ email }).select("+password")

    │

  3. Compare passwords:
```

### Sending JWT in Responses

There are two main ways to send the JWT to the client:

```js
// Method 1: In the response body (for API/mobile clients)
const createSendToken = (user, statusCode, res) => {
  const token = signToken(user._id);
  user.password = undefined;

  res.status(statusCode).json({
    status: "success",
    token,            // Client stores this (localStorage, state, etc.)
    data: { user },
  });
};

// Method 2: In an HTTP-only cookie (for browser clients — MORE SECURE)
const createSendToken = (user, statusCode, res) => {
  const token = signToken(user._id);

  const cookieOptions = {
    expires: new Date(
      Date.now() + process.env.JWT_COOKIE_EXPIRES_IN * 24 * 60 * 60 * 1000
    ),
    httpOnly: true,   // Cookie cannot be accessed by JavaScript (XSS protection)
    secure: process.env.NODE_ENV === "production", // Only HTTPS in production
    sameSite: "strict", // Prevents CSRF attacks
  };

  res.cookie("jwt", token, cookieOptions);

  user.password = undefined;

  res.status(statusCode).json({
    status: "success",
    token,
    data: { user },
  });
};
```

```
Cookie vs localStorage for JWT:
──────────────────────────────────────────────

  httpOnly Cookie:                localStorage:
  ─────────────────              ──────────────

  ✅ Cannot be read by JS       ❌ CAN be read by JS
     (XSS protection)              (vulnerable to XSS)
  ✅ Sent automatically          ❌ Must manually attach
     with every request              to every request
  ✅ Server controls expiry      ❌ Client controls expiry
  ⚠️  Needs CSRF protection      ✅ No CSRF concern

  Recommendation: Use httpOnly cookies for browsers
                   Use response body for mobile/API clients
```

## Protecting Routes — The protect Middleware

### How Route Protection Works

The `protect` middleware sits BEFORE your route handlers. It checks if the user has a valid JWT before allowing access.

```
Unprotected route:

  Client → Route Handler → Response
  (anyone can access)

Protected route:

  Client → [protect middleware] → Route Handler → Response
           │
           ├── No token?          → 401 Unauthorized
           ├── Invalid token?      → 401 Unauthorized
           ├── Token expired?      → 401 Unauthorized
           ├── User no longer exists? → 401 Unauthorized
           ├── Password changed after token? → 401 Unauthorized
           │
           └── All checks pass? → next() → Route Handler
               req.user = authenticated user
```

---

### Building the protect Middleware

```js
// controllers/authController.js (continued)

const { promisify } = require("util");

exports.protect = asyncHandler(async (req, res, next) => {
  // 1. Get the token and check if it exists
  let token;

  if (
    req.headers.authorization &&
    req.headers.authorization.startsWith("Bearer")
  ) {
    // Token from Authorization header: "Bearer eyJhbGci..."
    token = req.headers.authorization.split(" ")[1];
  } else if (req.cookies && req.cookies.jwt) {
    // Token from cookie (for browser-based auth)
    token = req.cookies.jwt;
  }

  if (!token) {
    return next(
      new AppError("You are not logged in. Please log in to get access.", 401)
    );
  }

  // 2. Verify the token (is it valid and not expired?)
  const decoded = await promisify(jwt.verify)(token, process.env.JWT_SECRET);
  // decoded = { id: "507f...", iat: 1705312200, exp: 1713088200 }

  // 3. Check if the user still exists
  //    (maybe the user was deleted after the token was issued)
  const currentUser = await User.findById(decoded.id);

  if (!currentUser) {
    return next(
      new AppError("The user belonging to this token no longer exists.", 401)
    );
  }

  // 4. Check if user changed password after the token was issued
  //    (if they changed password, old tokens should be invalid)
  if (currentUser.changedPasswordAfter(decoded.iat)) {
    return next(
      new AppError("Password was recently changed. Please log in again.", 401)
    );
  }

  // 5. ALL CHECKS PASSED — Grant access!
  req.user = currentUser; // Attach user to the request object
  next(); // Continue to the next middleware / route handler
});
```

```
The protect Middleware — Step by Step:
──────────────────────────────────────────────

  Request arrives with header:
  Authorization: Bearer eyJhbGci...
       │
       ▼
  Step 1: Extract token from header
       │
       ├── No token? → 401 "You are not logged in"
       │
       ▼
  Step 2: jwt.verify(token, SECRET)
       │
       ├── Invalid signature? → 401 "Invalid token"
       ├── Token expired?     → 401 "Token expired"
       │
       ▼
  Step 3: User.findById(decoded.id)
       │
       ├── User not found? → 401 "User no longer exists"
       │   (maybe the user deleted their account)
       │
       ▼
  Step 4: user.changedPasswordAfter(decoded.iat)
       │
       ├── Password changed? → 401 "Please log in again"
       │   (old tokens must be invalidated)
       │
       ▼
  Step 5: req.user = user
          next() → Controller has access to req.user ✓
```

### Using the protect Middleware on Routes


```
How router.use(protect) works:
```

## Authorization — Role-Based Access Control

Authentication tells us WHO the user is. Authorization tells us WHAT they're allowed to do.

### The restrictTo Middleware

```js
// controllers/authController.js (continued)

exports.restrictTo = (...roles) => {
  // roles is an array, e.g. ["admin", "moderator"]

  return (req, res, next) => {
    // req.user is set by the protect middleware (runs first)
    if (!roles.includes(req.user.role)) {
      return next(
        new AppError("You do not have permission to perform this action", 403)
      );
      // 403 = Forbidden (authenticated but NOT authorized)
    }

    next(); // User's role is in the allowed list — proceed
  };
};
```

```
How restrictTo Works:
──────────────────────────────────────────────

  restrictTo("admin", "moderator")
    │
    │ Returns a middleware function that checks:
    │ Is req.user.role in ["admin", "moderator"]?
    │
    ├── req.user.role = "admin"     → ✓ next()
    ├── req.user.role = "moderator" → ✓ next()
    ├── req.user.role = "user"      → ✗ 403 Forbidden
    │
    │ Note: protect middleware MUST run first to set req.user
```

> **Why is this a function that RETURNS a function?**
> Because middleware functions have a fixed signature `(req, res, next)`. We can't pass extra arguments (like roles) directly. So we use a **closure** — a wrapper function that accepts roles and returns the actual middleware.

## Password Management Features

### Update Password (Logged-In User)

When a user wants to change their password while they are already logged in:

```js
exports.updatePassword = asyncHandler(async (req, res, next) => {
  // 1. Get the user from the database (with password)
  //    req.user is set by the protect middleware
  const user = await User.findById(req.user.id).select("+password");

  // 2. Check if the current password they provided is correct
  if (!(await user.correctPassword(req.body.passwordCurrent, user.password))) {
    return next(new AppError("Your current password is incorrect", 401));
  }

  // 3. Update the password
  user.password = req.body.password;
  user.passwordConfirm = req.body.passwordConfirm;
  await user.save();
  // DON'T use findByIdAndUpdate — pre-save middleware won't run!
  // The password wouldn't get hashed!

  // 4. Log the user in again (send new JWT)
  //    The old token is now outdated because passwordChangedAt was updated
  createSendToken(user, 200, res);
});
```

```
Why user.save() and NOT findByIdAndUpdate()?
──────────────────────────────────────────────

  User.findByIdAndUpdate(id, { password: newPassword })
    │
    ├── ❌ Pre-save middleware DOES NOT run
    │      → Password is saved in PLAIN TEXT!
    │
    ├── ❌ passwordConfirm validator DOES NOT run
    │      → No password matching check
    │
    └── ❌ passwordChangedAt NOT updated
           → Old tokens still work

  user.password = newPassword;
  await user.save();
    │
    ├── ✅ Pre-save hook runs → password is hashed
    ├── ✅ Validators run → passwordConfirm is checked
    └── ✅ passwordChangedAt is updated → old tokens invalidated
```

### Forgot Password Flow

When a user forgets their password, they can request a reset link via email:

```js
const sendEmail = require("../utils/email");

exports.forgotPassword = asyncHandler(async (req, res, next) => {
  // 1. Find the user by email
  const user = await User.findOne({ email: req.body.email });

  if (!user) {
    return next(new AppError("There is no user with that email address", 404));
  }

  // 2. Generate a random reset token
  const resetToken = user.createPasswordResetToken();
  await user.save({ validateBeforeSave: false });
  // validateBeforeSave: false — skip validation because we're not
  // providing password/passwordConfirm, we're just saving the token

  // 3. Send the token via email
  const resetURL = `${req.protocol}://${req.get("host")}/api/auth/resetPassword/${resetToken}`;

  const message = `Forgot your password? Submit a PATCH request with your new password and passwordConfirm to: ${resetURL}\nIf you didn't forget your password, please ignore this email.`;

  try {
    await sendEmail({
      email: user.email,
      subject: "Your password reset token (valid for 10 min)",
      message,
    });

    res.status(200).json({
      status: "success",
      message: "Token sent to email!",
    });
  } catch (err) {
    // If email sending fails, clean up the reset token
    user.passwordResetToken = undefined;
    user.passwordResetExpires = undefined;
    await user.save({ validateBeforeSave: false });

    return next(
      new AppError("There was an error sending the email. Try again later.", 500)
    );
  }
});
```

```
Forgot Password Flow:
──────────────────────────────────────────────

  1. User: POST /api/auth/forgotPassword
     Body: { email: "alice@example.com" }
       │
       ▼
  2. Server finds user by email
       │
       ▼
  3. Server generates random reset token:
     • Plain token: "a3b2c1d4e5f6..." (sent in email)
     • Hashed token: "8f14e45f..." (stored in database)
     • Expiration: now + 10 minutes
       │
       ▼
  4. Server sends email with reset URL:
     "https://example.com/api/auth/resetPassword/a3b2c1d4e5f6..."
       │
       ▼
  5. User clicks the link and sends new password
     (see Reset Password below)

  Why hash the token in the database?
  • If a hacker gets access to the database, they'd see
    all active reset tokens and could reset anyone's password
  • By hashing it, even a database leak is safe
  • We compare the hash, just like passwords
```

### Reset Password Flow

When the user clicks the reset link from their email:

```js
const crypto = require("crypto");

exports.resetPassword = asyncHandler(async (req, res, next) => {
  // 1. Hash the token from the URL (to compare with what's in the DB)
  const hashedToken = crypto
    .createHash("sha256")
    .update(req.params.token)
    .digest("hex");

  // 2. Find user by the hashed token AND check it hasn't expired
  const user = await User.findOne({
    passwordResetToken: hashedToken,
    passwordResetExpires: { $gt: Date.now() }, // Token must not be expired
  });

  if (!user) {
    return next(new AppError("Token is invalid or has expired", 400));
  }

  // 3. Set the new password
  user.password = req.body.password;
  user.passwordConfirm = req.body.passwordConfirm;

  // 4. Clear the reset token fields
  user.passwordResetToken = undefined;
  user.passwordResetExpires = undefined;

  // 5. Save — this triggers pre-save hooks:
  //    • Password gets hashed
  //    • passwordChangedAt gets updated
  await user.save();

  // 6. Log the user in with a new JWT
  createSendToken(user, 200, res);
});
```

```
Reset Password Flow:
──────────────────────────────────────────────

  User: PATCH /api/auth/resetPassword/a3b2c1d4e5f6...
  Body: { password: "NewPass456", passwordConfirm: "NewPass456" }
       │
       ▼
  1. Hash the URL token:
     sha256("a3b2c1d4e5f6...") → "8f14e45f..."
       │
       ▼
  2. Find user where:
     passwordResetToken === "8f14e45f..."
     AND passwordResetExpires > now
       │
       ├── No user found? → 400 "Token is invalid or has expired"
       │
       ▼
  3. Set new password, clear reset token
       │
       ▼
  4. user.save() → password hashed, passwordChangedAt updated
       │
       ▼
  5. Send new JWT → user is logged in with new password
```

## Security Best Practices

### Rate Limiting

Prevent brute force attacks by limiting how many requests a client can make:

```bash
npm install express-rate-limit
```

```js
// app.js

const rateLimit = require("express-rate-limit");

// Allow 100 requests per IP per 15 minutes
const limiter = rateLimit({
  max: 100,
  windowMs: 15 * 60 * 1000, // 15 minutes
  message: "Too many requests from this IP, please try again in 15 minutes",
  standardHeaders: true, // Return rate limit info in headers
  legacyHeaders: false,
});

// Apply to all API routes
app.use("/api", limiter);

// Stricter limiter for login (prevent password guessing)
const loginLimiter = rateLimit({
  max: 5,
  windowMs: 15 * 60 * 1000,
  message: "Too many login attempts, please try again in 15 minutes",
});

app.use("/api/auth/login", loginLimiter);
```

```
Without Rate Limiting:                With Rate Limiting:
──────────────────────               ──────────────────────

  Attacker sends 10,000              Attacker sends 5 requests
  login attempts per minute           │
  trying different passwords          6th request → 429 Too Many Requests
  → Eventually guesses right!         "Please try again in 15 minutes"
                                      → Brute force is impractical!
```

### Data Sanitization

Protect against **NoSQL injection** and **Cross-Site Scripting (XSS)**:

```bash
npm install express-mongo-sanitize xss-clean
```

```js
// app.js

const mongoSanitize = require("express-mongo-sanitize");
const xss = require("xss-clean");

// NoSQL injection protection
app.use(mongoSanitize());
// Removes $ and . from req.body, req.query, req.params
// Without this, an attacker could send:
// { email: { "$gt": "" }, password: "anything" }
// This would match ANY user and bypass authentication!

// XSS protection
app.use(xss());
// Sanitizes user input by converting HTML characters
// <script>alert('hack')</script> → &lt;script&gt;alert('hack')&lt;/script&gt;
```

```
NoSQL Injection Attack:
──────────────────────────────────────────────

  Normal login:
  { "email": "alice@...", "password": "pass123" }
  → Finds user with that email, checks password ✓

  Attack (without sanitization):
  { "email": { "$gt": "" }, "password": { "$gt": "" } }
  → Translates to: find where email > "" AND password > ""
  → This matches EVERY user in the database!
  → Attacker logs in as the first user (usually admin)

  With mongoSanitize():
  → The $ signs are stripped from the input
  → Attack is neutralized ✓
```

### Parameter Pollution Prevention

Prevent parameter pollution attacks:

```bash
npm install hpp
```

```js
// app.js

const hpp = require("hpp");

app.use(
  hpp({
    whitelist: ["price", "rating", "category", "sort"],
    // Allow these fields to have duplicate values in query string
    // e.g., ?price=10&price=20 (valid for $in queries)
  })
);

// Without hpp, a request like:
// GET /api/products?sort=price&sort=name
// Would set sort to an ARRAY: ["price", "name"]
// This could crash your code if you expect a string

// With hpp:
// Uses the LAST value: sort = "name"
// Unless the field is in the whitelist
```

## Storing JWT on the Client Side

```
┌──────────────────────────────────────────────────────────────┐
│             WHERE TO STORE THE JWT ON THE CLIENT              │
├──────────────────────────────────────────────────────────────┤
│                                                               │
│  Option 1: HTTP-Only Cookie (RECOMMENDED for browsers)       │
│  ──────────────────────────────────────────────               │
│  ✅ Cannot be read by JavaScript (XSS-safe)                  │
│  ✅ Sent automatically with every request                    │
│  ✅ Server controls the expiry                               │
│  ⚠️  Vulnerable to CSRF (use SameSite + CSRF tokens)         │
│                                                               │
│  Option 2: localStorage                                      │
│  ──────────────────────────────────────────────               │
│  ✅ Easy to implement                                        │
│  ✅ Persists across page refreshes                           │
│  ❌ Readable by ANY JavaScript on the page (XSS risk)        │
│  ❌ Stays forever until manually removed                     │
│                                                               │
│  Option 3: sessionStorage                                    │
│  ──────────────────────────────────────────────               │
│  ✅ Automatically cleared when tab closes                    │
│  ❌ Readable by JavaScript (XSS risk)                        │
│  ❌ Lost when user closes the browser tab                    │
│                                                               │
│  Option 4: In-Memory (React State/Context)                   │
│  ──────────────────────────────────────────────               │
│  ✅ Most secure (not persisted anywhere)                     │
│  ❌ Lost on page refresh (user must re-login)                │
│  ⚠️  Often combined with refresh tokens                       │
│                                                               │
│  RECOMMENDATION:                                              │
│  • Browser apps: HTTP-Only Cookie + SameSite                 │
│  • Mobile apps: Secure encrypted storage                     │
│  • SPAs: In-memory + refresh token rotation                  │
│                                                               │
└──────────────────────────────────────────────────────────────┘
```

### Using JWT in Client-Side Requests

```js
// ──── If using localStorage ─────────────────────────

// Storing the token after login
const response = await fetch("/api/auth/login", {
  method: "POST",
  headers: { "Content-Type": "application/json" },
  body: JSON.stringify({ email, password }),
});
const data = await response.json();
localStorage.setItem("token", data.token);

// Sending the token with requests
const token = localStorage.getItem("token");
const response = await fetch("/api/products", {
  headers: {
    Authorization: `Bearer ${token}`,
  },
});

// ──── If using HTTP-Only cookies ────────────────────

// No need to manually store or send the token!
// The browser handles it automatically.
const response = await fetch("/api/auth/login", {
  method: "POST",
  credentials: "include", // Important! Sends cookies
  headers: { "Content-Type": "application/json" },
  body: JSON.stringify({ email, password }),
});

// Cookie is automatically sent with every subsequent request
const response = await fetch("/api/products", {
  credentials: "include", // Sends cookies automatically
});
```

### The Complete app.js with Security Middleware

```js
// app.js

const express = require("express");
const helmet = require("helmet");
const rateLimit = require("express-rate-limit");
const mongoSanitize = require("express-mongo-sanitize");
const xss = require("xss-clean");
const hpp = require("hpp");
const cookieParser = require("cookie-parser");

const AppError = require("./utils/AppError");
const globalErrorHandler = require("./controllers/errorController");
const authRoutes = require("./routes/authRoutes");
const userRoutes = require("./routes/userRoutes");
const productRoutes = require("./routes/productRoutes");

const app = express();

// ──── GLOBAL MIDDLEWARE (Order matters!) ─────────────

// 1. Security HTTP headers
app.use(helmet());

// 2. Rate limiting
app.use(
  "/api",
  rateLimit({
    max: 100,
    windowMs: 15 * 60 * 1000,
    message: "Too many requests, please try again later",
  })
);

// 3. Body parsing (with size limit to prevent abuse)
app.use(express.json({ limit: "10kb" }));
app.use(cookieParser());

// 4. Data sanitization against NoSQL injection
app.use(mongoSanitize());

// 5. Data sanitization against XSS
app.use(xss());

// 6. Prevent parameter pollution
app.use(hpp({ whitelist: ["price", "rating", "category"] }));

// ──── ROUTES ────────────────────────────────────────

app.use("/api/auth", authRoutes);
app.use("/api/users", userRoutes);
app.use("/api/products", productRoutes);

// ──── HANDLE UNDEFINED ROUTES ───────────────────────

app.all("*", (req, res, next) => {
  next(new AppError(`Cannot find ${req.originalUrl} on this server`, 404));
});

// ──── GLOBAL ERROR HANDLER ──────────────────────────

app.use(globalErrorHandler);

module.exports = app;
```

```
──────────────────────────────────────────────

  Request arrives

       │

  2. rateLimit()        → Check request count
       │

  3. express.json()     → Parse request body
```
