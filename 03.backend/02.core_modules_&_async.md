## 🎯 Understanding Node.js Core Modules

### 📝 What Are Core Modules?

When you install Node.js, it comes with a set of **built-in modules** — these are called **core modules**. You don't need to install them with `npm`. They're already there, baked into Node.js itself.

Think of core modules like the **built-in apps** on a new phone. You didn't download them — they came pre-installed and ready to use.

### 🔧 How Do You Use a Core Module?

You import them using `require()` (CommonJS) or `import` (ES Modules):

```js
// CommonJS (traditional Node.js way)
const fs = require('fs');
const path = require('path');
const http = require('http');

// ES Modules (modern way — needs "type": "module" in package.json)
import fs from 'fs';
import path from 'path';
import http from 'http';
```

### 🌟 The Most Important Core Modules

| Module   | What It Does                                      |
|----------|---------------------------------------------------|
| `fs`     | Read, write, delete, rename files and folders     |
| `path`   | Build and manipulate file/folder paths safely     |
| `http`   | Create web servers and make HTTP requests          |
| `os`     | Get info about the operating system                |
| `events` | Create and handle custom events                    |
| `url`    | Parse and format URLs                              |
| `crypto` | Hashing, encryption, and security utilities        |
| `util`   | Utility functions (promisify, format, etc.)        |
| `stream` | Handle streaming data (reading/writing in chunks)  |

We'll focus on the **Big Three**: `path`, `fs`, and `http`.

## 📂 The `path` Module — Working with File Paths

### 🧠 Why Do We Need `path`?

File paths are **different on different operating systems**:

- **Windows:** `C:\Users\john\project\index.js` (uses backslash `\`)
- **macOS/Linux:** `/home/john/project/index.js` (uses forward slash `/`)

If you hard-code paths with string concatenation like `folder + '/' + file`, it **might break on Windows**. The `path` module handles this for you — it always builds the correct path for whatever OS your code runs on.

### 📥 Importing the `path` Module

```js
const path = require('path');
```

That's it. No installation needed. It's built into Node.js.

### 🔑 Key `path` Methods — Explained One by One

---

#### `path.join()` — Combine Path Segments Safely

This is **the most commonly used** `path` method. It takes multiple pieces of a path and joins them together with the correct separator for your OS.

```js
const fullPath = path.join('users', 'john', 'documents', 'file.txt');

console.log(fullPath);
// On Linux/Mac: 'users/john/documents/file.txt'
// On Windows:   'users\\john\\documents\\file.txt'
```

**Why not just use string concatenation?**

```js
// ❌ BAD — This breaks on some systems
const bad = 'users' + '/' + 'john' + '/' + 'file.txt';

// ✅ GOOD — Works on ALL systems
const good = path.join('users', 'john', 'file.txt');
```

**It even cleans up messy paths:**

```js
const messy = path.join('users', '//john', '../jane', 'file.txt');
console.log(messy); // 'users/jane/file.txt'
// It resolved the `..` (go up one directory) automatically!
```

---

#### `path.resolve()` — Get an Absolute Path

While `path.join()` just sticks pieces together, `path.resolve()` gives you a **full absolute path** — starting from the root of your file system.

```js
const absolutePath = path.resolve('src', 'index.js');
console.log(absolutePath);
// Something like: '/home/john/my-project/src/index.js'
// It starts from your current working directory
```

**How `path.resolve()` works step by step:**

1. It starts from the **current working directory** (where you ran `node`)
2. It processes each argument from **left to right**
3. If it encounters an **absolute path** (starts with `/`), it restarts from there
4. It gives you the final absolute path

```js
// Starting from /home/john/project

path.resolve('src', 'index.js');
// → /home/john/project/src/index.js

path.resolve('/tmp', 'test.js');
// → /tmp/test.js (restarted because /tmp is absolute)

path.resolve('src', '/tmp', 'test.js');
// → /tmp/test.js (first absolute path wins from right)
```

---

#### `__dirname` and `__filename` — Your Current Location

These are **special global variables** in Node.js (CommonJS):

```js
console.log(__filename);
// Full path to THIS file: /home/john/project/server.js

console.log(__dirname);
// Directory of THIS file: /home/john/project
```

**Common pattern — building paths relative to the current file:**

```js
// You're in /home/john/project/src/server.js
// You want to access /home/john/project/data/users.json

const dataPath = path.join(__dirname, '..', 'data', 'users.json');
// __dirname = /home/john/project/src
// '..'      = go up to /home/john/project
// 'data'    = go into /home/john/project/data
// Result:   /home/john/project/data/users.json
```

> **Note for ES Modules:** `__dirname` and `__filename` don't exist in ES modules. You'd use:
> ```js
> import { fileURLToPath } from 'url';
> import { dirname } from 'path';
>
> const __filename = fileURLToPath(import.meta.url);
> const __dirname = dirname(__filename);
> ```

---

#### `path.basename()` — Get Just the File Name

Extracts the **file name** from a full path:

```js
const filePath = '/home/john/project/src/index.js';

path.basename(filePath);         // 'index.js'
path.basename(filePath, '.js');  // 'index' (removed the extension)
```

---

#### `path.extname()` — Get the File Extension

```js
path.extname('report.pdf');          // '.pdf'
path.extname('server.config.js');    // '.js'
path.extname('Makefile');            // '' (no extension)
path.extname('.env');                // '' (dot files have no extension)
```

---

#### `path.dirname()` — Get the Directory Part

```js
path.dirname('/home/john/project/src/index.js');
// '/home/john/project/src'
```

---

#### `path.parse()` — Break a Path into Its Pieces

Returns an **object** with every part of the path:

```js
const parsed = path.parse('/home/john/project/src/index.js');
console.log(parsed);
/*
{
  root: '/',
  dir: '/home/john/project/src',
  base: 'index.js',
  ext: '.js',
  name: 'index'
}
*/
```

#### `path.format()` — Rebuild a Path from Pieces

The **reverse** of `path.parse()`:

```js
const pathObj = {
  dir: '/home/john/project/src',
  base: 'index.ts'   // Changed extension!
};

path.format(pathObj);
// '/home/john/project/src/index.ts'
```

---

#### `path.isAbsolute()` — Check If a Path Is Absolute

```js
path.isAbsolute('/home/john');    // true  (starts with /)
path.isAbsolute('./src');         // false (relative)
path.isAbsolute('src/index.js'); // false (relative)
```

---

#### `path.relative()` — Get the Path from A to B

```js
const from = '/home/john/project/src';
const to = '/home/john/project/public/images';

path.relative(from, to);
// '../public/images'
// "From src, go up one, then into public/images"
```

---

### 🧠 `path` Module — Mental Model

Think of `path` like a **GPS for your file system**:

- `path.join()` → "Give me directions by combining these road names"
- `path.resolve()` → "Give me the full address from here"
- `path.basename()` → "What's the name of this building?"
- `path.dirname()` → "What street is this building on?"
- `path.extname()` → "What type of building is this?"
- `path.parse()` → "Break down the entire address for me"

## 📁 The `fs` Module — Working with the File System

### 🎯 What Is the `fs` Module?

The `fs` (file system) module lets your Node.js code **interact with files and folders** on the computer. You can:

- **Read** files (get their content)
- **Write** files (create or overwrite)
- **Append** to files (add content without overwriting)
- **Delete** files
- **Rename** files
- **Create/remove** directories
- **Check** if files/folders exist
- **Watch** files for changes

### 📥 Importing the `fs` Module

The `fs` module gives you **three different APIs** (ways to use it):

```js
// 1. Callback-based API (original, oldest)
const fs = require('fs');

// 2. Promise-based API (modern, recommended)
const fs = require('fs').promises;
// OR
const fs = require('fs/promises'); // Node 14+

// 3. Synchronous API (blocks the thread — use sparingly)
const fs = require('fs');
// Same import, but you use methods like readFileSync()
```

We'll learn **all three** so you understand the evolution, but we'll settle on using the **Promise-based API with async/await** as the recommended approach.

---

### 📖 Reading Files

#### Synchronous — `fs.readFileSync()`

This **blocks** your entire program until the file is fully read:

```js
const fs = require('fs');

// Read a file synchronously (BLOCKING)
const content = fs.readFileSync('hello.txt', 'utf-8');
console.log(content);
// Nothing else runs until the file is completely read
```

**What is `'utf-8'`?** It's the **encoding** — it tells Node.js to return the content as a **readable string**. Without it, you get raw binary data (a `Buffer`):

```js
// Without encoding — returns a Buffer (raw bytes)
const raw = fs.readFileSync('hello.txt');
console.log(raw);
// <Buffer 48 65 6c 6c 6f 20 57 6f 72 6c 64>

// With encoding — returns a string
const text = fs.readFileSync('hello.txt', 'utf-8');
console.log(text);
// "Hello World"
```

> **When is `readFileSync` OK?**
> - Reading config files at application startup (before your server starts)
> - CLI tools that run once and exit
>
> **When is it BAD?**
> - Inside request handlers (blocks ALL other users)
> - In any code that runs repeatedly during the app's lifetime

---

#### Callback-based — `fs.readFile()`

This is **non-blocking** — your program continues running while the file is being read:

```js
const fs = require('fs');

console.log('1. Starting to read file...');

fs.readFile('hello.txt', 'utf-8', (error, data) => {
  if (error) {
    console.error('Failed to read file:', error.message);
    return;
  }
  console.log('3. File content:', data);
});

console.log('2. This runs BEFORE the file is read!');
```

**Output:**
```
1. Starting to read file...
2. This runs BEFORE the file is read!
3. File content: Hello World
```

**Why does "2" print before "3"?** Because `readFile` is **asynchronous**. It tells the OS "hey, read this file for me", then immediately moves to the next line. When the file is done being read, it calls the **callback function** with the result.

The callback always follows the **error-first pattern**: `(error, data) => { ... }`
- If something went wrong, `error` has the details
- If everything worked, `error` is `null` and `data` has the content

---

#### Promise-based — `fs.promises.readFile()` (Recommended)

```js
const fs = require('fs').promises;

async function readMyFile() {
  try {
    const content = await fs.readFile('hello.txt', 'utf-8');
    console.log('File content:', content);
  } catch (error) {
    console.error('Failed to read file:', error.message);
  }
}

readMyFile();
```

This is the cleanest approach. We'll dive deep into **how Promises and async/await work** in sections 6 and 7.

### ✍️ Writing Files

#### `fs.writeFileSync()` — Synchronous Write

```js
const fs = require('fs');

// Creates the file if it doesn't exist
// OVERWRITES the file if it does exist!
fs.writeFileSync('output.txt', 'Hello, this is my content!');
console.log('File written!');
```

#### `fs.writeFile()` — Callback Write

```js
const fs = require('fs');

fs.writeFile('output.txt', 'Hello, this is my content!', (error) => {
  if (error) {
    console.error('Write failed:', error.message);
    return;
  }
  console.log('File written successfully!');
});
```

#### Promise-based Write (Recommended)

```js
const fs = require('fs').promises;

async function writeMyFile() {
  try {
    await fs.writeFile('output.txt', 'Hello, this is my content!');
    console.log('File written successfully!');
  } catch (error) {
    console.error('Write failed:', error.message);
  }
}

writeMyFile();
```

**Writing JSON data (very common in backends):**

```js
const fs = require('fs').promises;

async function saveUserData() {
  const users = [
    { id: 1, name: 'Alice', email: 'alice@example.com' },
    { id: 2, name: 'Bob', email: 'bob@example.com' },
  ];

  // Convert JavaScript object/array to a JSON string
  const jsonString = JSON.stringify(users, null, 2);
  //                                       ↑     ↑
  //                                  replacer  spaces (pretty-print)

  await fs.writeFile('users.json', jsonString);
  console.log('Users saved!');
}

saveUserData();
```

The `JSON.stringify(data, null, 2)` creates nicely formatted JSON:

```json
[
  {
    "id": 1,
    "name": "Alice",
    "email": "alice@example.com"
  },
  {
    "id": 2,
    "name": "Bob",
    "email": "bob@example.com"
  }
]
```

### 📝 Appending to Files

Appending **adds** content to the end of a file instead of overwriting it:

```js
const fs = require('fs').promises;

async function logMessage(message) {
  const timestamp = new Date().toISOString();
  const logEntry = `[${timestamp}] ${message}\n`;

  await fs.appendFile('app.log', logEntry);
  console.log('Log entry added!');
}

logMessage('Server started');
logMessage('User logged in');
```

**Result in `app.log`:**
```
[2026-02-24T10:30:00.000Z] Server started
[2026-02-24T10:30:01.000Z] User logged in
```

### 🗑️ Deleting Files

```js
const fs = require('fs').promises;

async function deleteFile() {
  try {
    await fs.unlink('temp-file.txt');
    console.log('File deleted!');
  } catch (error) {
    if (error.code === 'ENOENT') {
      console.log('File does not exist, nothing to delete');
    } else {
      throw error; // Re-throw unexpected errors
    }
  }
}

deleteFile();
```

> **Why `unlink` and not `delete`?** It's a Unix heritage. In Unix, files are "linked" to the file system. "Unlinking" removes that link. When no links remain, the file is deleted.

## ⚡ Synchronous vs Asynchronous — Why It Matters

### 🍽️ The Restaurant Analogy

Imagine you run a restaurant:

**Synchronous approach (single-threaded, blocking):**
```
Customer 1 orders → Chef cooks meal → Customer 1 served
Customer 2 orders → Chef cooks meal → Customer 2 served
Customer 3 orders → Chef cooks meal → Customer 3 served
```
Each customer **waits for the previous customer's meal** to be fully cooked before anyone can even place an order. If one meal takes 30 minutes, everyone waits.

**Asynchronous approach (what Node.js does):**
```
Customer 1 orders → Chef starts cooking (sends to kitchen)
Customer 2 orders → Chef starts cooking (sends to kitchen)  
Customer 3 orders → Chef starts cooking (sends to kitchen)
← Customer 2's meal ready → Served!
← Customer 1's meal ready → Served!
← Customer 3's meal ready → Served!
```
The **waiter** (Node.js event loop) takes orders from everyone immediately. The **kitchen** (OS-level I/O threads) handles the actual cooking. When a meal is ready, it gets served.

### 🧠 Why This Matters in Node.js

Node.js is **single-threaded** — it has **one main thread** doing all the work. If you block that thread by reading a file synchronously, **nobody else can be served**:

```js
// ❌ BLOCKING — Your entire server freezes while reading the file
const data = fs.readFileSync('huge-file.txt', 'utf-8');
// ALL other requests wait until this finishes

// ✅ NON-BLOCKING — Your server keeps handling other requests
fs.readFile('huge-file.txt', 'utf-8', (err, data) => {
  // This runs later, when the file is ready
});
// The server continues handling other requests immediately
```

**Rule of thumb:**
- **Synchronous (`*Sync`) methods:** Only at application startup
- **Async methods:** Everywhere else (especially in request handlers)

## 📞 Callbacks — The Original Async Pattern

### 🎯 What Is a Callback?

A **callback** is a function you pass to another function, saying "run this when you're done."

```js
function doSomethingLater(callback) {
  console.log('Starting the work...');
  
  setTimeout(() => {
    console.log('Work is done!');
    callback('Here is the result');  // Call the callback with the result
  }, 2000); // Simulate 2 seconds of work
}

doSomethingLater((result) => {
  console.log('Received:', result);
});
```

**Output:**
```
Starting the work...
(2 seconds pass)
Work is done!
Received: Here is the result
```

### ⚠️ Error-First Callbacks (Node.js Convention)

In Node.js, callbacks always follow this pattern: `(error, result) => { ... }`

```js
const fs = require('fs');

fs.readFile('data.txt', 'utf-8', (error, data) => {
  // Always check for errors FIRST
  if (error) {
    console.error('Something went wrong:', error.message);
    return; // Stop here — don't try to use `data`
  }

  // If we get here, everything worked
  console.log('File content:', data);
});
```

### 🔥 The Callback Hell Problem

What if you need to do multiple async operations **in sequence**? Things get ugly fast:

```js
const fs = require('fs');

// Read a config file, then read the data file it points to,
// then process the data and write the result

fs.readFile('config.json', 'utf-8', (err, configData) => {
  if (err) {
    console.error('Failed to read config:', err);
    return;
  }

  const config = JSON.parse(configData);

  fs.readFile(config.dataFile, 'utf-8', (err, rawData) => {
    if (err) {
      console.error('Failed to read data:', err);
      return;
    }

    const processed = rawData.toUpperCase(); // some processing

    fs.writeFile('output.txt', processed, (err) => {
      if (err) {
        console.error('Failed to write output:', err);
        return;
      }

      console.log('Done!');

      // What if we need to do MORE things? Even more nesting...
      fs.readFile('another-file.txt', 'utf-8', (err, moreData) => {
        // 😱 This is getting out of control!
      });
    });
  });
});
```

This is called **"Callback Hell"** or the **"Pyramid of Doom"** — the code keeps indenting further and further to the right. It's:
- Hard to read
- Hard to debug
- Hard to add error handling
- Easy to make mistakes

**This problem is exactly why Promises were invented.**

## 🤝 Promises — The Modern Foundation

### 🎯 What Is a Promise?

A **Promise** is an object that represents the **eventual completion (or failure)** of an asynchronous operation and its resulting value.

Think of a Promise like an **online order receipt**:

1. You place an order (create the Promise) — **Pending** state
2. Either:
   - Your package arrives → **Fulfilled** (resolved) state
   - Something goes wrong → **Rejected** state

A Promise is always in one of three states:
- **Pending** — The operation is still happening
- **Fulfilled** — The operation completed successfully
- **Rejected** — The operation failed

```
         ┌──────────┐
         │ PENDING   │  ← Just created, waiting for result
         └─────┬─────┘
               │
      ┌────────┴────────┐
      ▼                 ▼
┌──────────┐     ┌──────────┐
│ FULFILLED │     │ REJECTED  │
│ (success) │     │ (failure) │
└──────────┘     └──────────┘
```

Once a Promise is fulfilled or rejected, it **never changes state again**. It's **settled**.

---

### 🏗️ Creating a Promise

```js
const myPromise = new Promise((resolve, reject) => {
  // resolve and reject are functions given to us by JavaScript

  const success = true; // Simulate success/failure

  if (success) {
    resolve('Operation completed!'); // Fulfill the promise
  } else {
    reject(new Error('Something went wrong!')); // Reject the promise
  }
});
```

**Real-world example — simulating a database query:**

```js
function getUserFromDatabase(userId) {
  return new Promise((resolve, reject) => {
    // Simulate a 1-second database query
    setTimeout(() => {
      const users = {
        1: { name: 'Alice', age: 25 },
        2: { name: 'Bob', age: 30 },
      };

      const user = users[userId];

      if (user) {
        resolve(user);                        // Found the user!
      } else {
        reject(new Error('User not found'));   // No such user
      }
    }, 1000);
  });
}
```

---

### 🔄 Using a Promise with `.then()` and `.catch()`

```js
getUserFromDatabase(1)
  .then((user) => {
    console.log('Found user:', user);
    // { name: 'Alice', age: 25 }
  })
  .catch((error) => {
    console.error('Error:', error.message);
  });
```

**How `.then()` and `.catch()` work:**

- `.then(callback)` — Runs when the Promise is **fulfilled**. Receives the resolved value.
- `.catch(callback)` — Runs when the Promise is **rejected**. Receives the error.
- `.finally(callback)` — Runs **regardless** of success or failure. No value received.

```js
getUserFromDatabase(1)
  .then((user) => {
    console.log('User:', user);
  })
  .catch((error) => {
    console.error('Error:', error.message);
  })
  .finally(() => {
    console.log('Query complete (success or failure)');
  });
```

### 🔄 Promise Chaining — Solving Callback Hell

The magic of Promises: **`.then()` returns a NEW Promise**, so you can chain them:

```js
const fs = require('fs').promises;

fs.readFile('config.json', 'utf-8')
  .then((configData) => {
    const config = JSON.parse(configData);
    return fs.readFile(config.dataFile, 'utf-8'); // Returns a new Promise
  })
  .then((rawData) => {
    const processed = rawData.toUpperCase();
    return fs.writeFile('output.txt', processed); // Returns a new Promise
  })
  .then(() => {
    console.log('Done!');
  })
  .catch((error) => {
    // ONE catch handles errors from ANY step above!
    console.error('Something failed:', error.message);
  });
```

Compare this to the callback hell version — **flat, readable, one error handler**.

### 🔄 Promise Utility Methods

#### `Promise.all()` — Run Multiple Promises in Parallel

When you have multiple independent operations, run them **at the same time**:

```js
const fs = require('fs').promises;

async function readAllFiles() {
  // All three files are read SIMULTANEOUSLY
  const results = await Promise.all([
    fs.readFile('file1.txt', 'utf-8'),
    fs.readFile('file2.txt', 'utf-8'),
    fs.readFile('file3.txt', 'utf-8'),
  ]);

  console.log('File 1:', results[0]);
  console.log('File 2:', results[1]);
  console.log('File 3:', results[2]);
}
```

**Important:** If **ANY** promise rejects, `Promise.all()` rejects immediately with that error.

```
file2 ░░░░░░░░████████░░░░░░░░
file3 ░░░░░░░░░░░░░░░░████████

Timeline with Promise.all:
file1 ████████
file2 ████████
file3 ████████
Total: ████████  (8 units — 3x faster!)
```

#### `Promise.allSettled()` — Run All, Don't Fail Fast

Unlike `Promise.all()`, this **waits for ALL promises** to settle, even if some fail:

```js
const results = await Promise.allSettled([
  fs.readFile('exists.txt', 'utf-8'),
  fs.readFile('missing.txt', 'utf-8'),   // This will fail
  fs.readFile('also-exists.txt', 'utf-8'),
]);

results.forEach((result, index) => {
  if (result.status === 'fulfilled') {
    console.log(`File ${index + 1}: ${result.value}`);
  } else {
    console.log(`File ${index + 1} failed: ${result.reason.message}`);
  }
});
```

---

#### `Promise.race()` — First One Wins

Returns the result of whichever promise settles **first**:

```js
// Timeout pattern — if the file takes more than 5 seconds, give up
const result = await Promise.race([
  fs.readFile('big-file.txt', 'utf-8'),
  new Promise((_, reject) =>
    setTimeout(() => reject(new Error('Timeout!')), 5000)
  ),
]);
```

---

#### `Promise.any()` — First Success Wins

Like `race`, but ignores rejections — returns the first **fulfilled** promise:

```js
// Try multiple API endpoints, use whichever responds first
const data = await Promise.any([
  fetch('https://api1.example.com/data'),
  fetch('https://api2.example.com/data'),
  fetch('https://api3.example.com/data'),
]);
```

## ✨ async/await — The Syntactic Sugar

### 🎯 What Is async/await?

`async/await` is **syntactic sugar on top of Promises**. It makes asynchronous code **look and behave like synchronous code**, making it much easier to read and write.

Under the hood, it's **still using Promises**. It just provides a cleaner syntax.

---

### 🔄 The `async` Keyword

Adding `async` before a function does **two things**:

1. Makes the function **always return a Promise**
2. Allows you to use `await` inside it

```js
// Regular function
function greet() {
  return 'Hello';
}
console.log(greet()); // 'Hello' (just a string)

// Async function
async function greetAsync() {
  return 'Hello';
}
console.log(greetAsync()); // Promise { 'Hello' } (it's a Promise!)

// To get the value from the Promise:
greetAsync().then((value) => console.log(value)); // 'Hello'
```

Even if you just return a regular value, `async` wraps it in a Promise automatically.

---

### 🔄 The `await` Keyword

`await` **pauses** the execution of an `async` function until the Promise settles:

```js
async function fetchData() {
  console.log('1. Starting...');

  // await PAUSES here until the Promise resolves
  const data = await someAsyncOperation();

  // This line ONLY runs after the above Promise resolves
  console.log('2. Got data:', data);

  return data;
}
```

**Without await:**
```js
const data = someAsyncOperation();
console.log(data); // Promise { <pending> } — NOT the actual data!
```

**With await:**
```js
const data = await someAsyncOperation();
console.log(data); // The actual resolved value!
```

`await` **unwraps** the Promise and gives you the resolved value directly.

---

### 🛡️ Error Handling with try/catch

With callbacks, you check `if (error)`. With async/await, you use **try/catch** — the same way you handle errors in synchronous code:

```js
async function readConfig() {
  try {
    const data = await fs.readFile('config.json', 'utf-8');
    const config = JSON.parse(data);
    console.log('Config loaded:', config);
    return config;
  } catch (error) {
    // Catches BOTH: file read errors AND JSON parse errors
    console.error('Failed to load config:', error.message);
    
    // Return a default config as fallback
    return { port: 3000, debug: false };
  } finally {
    // Runs no matter what (optional)
    console.log('Config loading attempt complete');
  }
}
```

### 🔀 Sequential vs Parallel Execution

#### Sequential (one after another)

```js
async function sequential() {
  // These run ONE AT A TIME
  const user = await fetchUser(1);      // Wait 1 second
  const posts = await fetchPosts(1);    // Wait 1 second
  const comments = await fetchComments(1); // Wait 1 second
  // Total time: ~3 seconds
}
```

#### Parallel (all at once)

```js
async function parallel() {
  // Start ALL three at the same time
  const [user, posts, comments] = await Promise.all([
    fetchUser(1),
    fetchPosts(1),
    fetchComments(1),
  ]);
  // Total time: ~1 second (they ran simultaneously!)
}
```

**When to use which:**

- **Sequential:** When each operation depends on the previous one's result
- **Parallel:** When operations are **independent** of each other

```js
// Must be SEQUENTIAL — each step needs the previous result
async function processOrder(orderId) {
  const order = await getOrder(orderId);              // Need order first
  const inventory = await checkInventory(order.items); // Need order.items
  const receipt = await chargeCustomer(order.total);   // Need order.total
  return receipt;
}

// Can be PARALLEL — these are independent
async function getDashboardData(userId) {
  const [profile, notifications, recentOrders] = await Promise.all([
    getProfile(userId),
    getNotifications(userId),
    getRecentOrders(userId),
  ]);
  return { profile, notifications, recentOrders };
}
```

### 💡 Common async/await Patterns

#### Looping with async/await

```js
// Process items one at a time (sequential)
async function processItemsSequentially(items) {
  for (const item of items) {
    await processItem(item);
    console.log(`Processed: ${item.name}`);
  }
}

// Process ALL items at once (parallel)
async function processItemsInParallel(items) {
  await Promise.all(
    items.map((item) => processItem(item))
  );
  console.log('All items processed!');
}

// Process in batches (controlled parallel)
async function processInBatches(items, batchSize = 5) {
  for (let i = 0; i < items.length; i += batchSize) {
    const batch = items.slice(i, i + batchSize);
    await Promise.all(batch.map((item) => processItem(item)));
    console.log(`Batch ${Math.floor(i / batchSize) + 1} complete`);
  }
}
```

> **Warning:** Don't use `forEach` with async/await — it doesn't wait for async operations!
> ```js
> // ❌ BROKEN — forEach doesn't await
> items.forEach(async (item) => {
>   await processItem(item); // These all fire at once, uncontrolled!
> });
>
> // ✅ CORRECT — for...of properly awaits each iteration
> for (const item of items) {
>   await processItem(item);
> }
> ```

---

#### Retry Pattern

```js
async function fetchWithRetry(url, maxRetries = 3) {
  for (let attempt = 1; attempt <= maxRetries; attempt++) {
    try {
      const response = await fetch(url);
      return response; // Success! Return immediately
    } catch (error) {
      console.log(`Attempt ${attempt} failed: ${error.message}`);

      if (attempt === maxRetries) {
        throw new Error(`Failed after ${maxRetries} attempts`);
      }

      // Wait before retrying (exponential backoff)
      const delay = Math.pow(2, attempt) * 1000; // 2s, 4s, 8s
      await new Promise((resolve) => setTimeout(resolve, delay));
    }
  }
}
```

---

#### Timeout Pattern

```js
function withTimeout(promise, ms) {
  const timeout = new Promise((_, reject) =>
    setTimeout(() => reject(new Error(`Timed out after ${ms}ms`)), ms)
  );
  return Promise.race([promise, timeout]);
}

// Usage:
async function getData() {
  try {
    const data = await withTimeout(fetchUser(1), 5000);
    console.log('Got data:', data);
  } catch (error) {
    console.error(error.message); // "Timed out after 5000ms"
  }
}
```

### 🔄 The Evolution of Async Code — A Comparison

Let's see the **same operation** written all three ways:

**Task:** Read a JSON file, parse it, modify it, and write it back.

#### 1. Callbacks (old way)

```js
const fs = require('fs');

fs.readFile('data.json', 'utf-8', (err, raw) => {
  if (err) {
    console.error('Read error:', err);
    return;
  }

  const data = JSON.parse(raw);
  data.updated = new Date().toISOString();

    if (err) {
      console.error('Write error:', err);
      return;
    }

  });
});
```

#### 2. Promises with .then() (better)

```js

    data.updated = new Date().toISOString();

    console.log('File updated!');
  })
  .catch((err) => {
    console.error('Error:', err);
  });
```

#### 3. async/await (best — recommended)

```js

    const data = JSON.parse(raw);
    data.updated = new Date().toISOString();

  }

updateFile();
```

The async/await version reads almost like synchronous code — line by line, top to bottom. That's its superpower.

### 📝 Project: A Simple JSON-based Data Store

```js
const fs = require('fs').promises;
const path = require('path');

class JsonStore {
  constructor(filename) {
    // Build the full path to the data file
    this.filePath = path.join(__dirname, 'data', filename);
  }

  // Ensure the data directory exists
  async init() {
    const dir = path.dirname(this.filePath);
    // { recursive: true } means "create parent dirs if needed, don't error if exists"
    await fs.mkdir(dir, { recursive: true });

    // Create the file with empty array if it doesn't exist
    try {
      await fs.access(this.filePath);
    } catch {
      await fs.writeFile(this.filePath, JSON.stringify([], null, 2));
    }
  }

  // Read all items from the store
  async getAll() {
    const raw = await fs.readFile(this.filePath, 'utf-8');
    return JSON.parse(raw);
  }

  // Find one item by ID
  async getById(id) {
    const items = await this.getAll();
    return items.find((item) => item.id === id) || null;
  }

  // Add a new item
  async add(item) {
    const items = await this.getAll();

    const newItem = {
      id: Date.now(),       // Simple unique ID
      ...item,
      createdAt: new Date().toISOString(),
    };

    items.push(newItem);
    await fs.writeFile(this.filePath, JSON.stringify(items, null, 2));
    return newItem;
  }

  // Update an existing item
  async update(id, updates) {
    const items = await this.getAll();
    const index = items.findIndex((item) => item.id === id);

    if (index === -1) {
      throw new Error(`Item with id ${id} not found`);
    }

    items[index] = {
      ...items[index],
      ...updates,
      updatedAt: new Date().toISOString(),
    };

    await fs.writeFile(this.filePath, JSON.stringify(items, null, 2));
    return items[index];
  }

  // Delete an item
  async delete(id) {
    const items = await this.getAll();
    const filtered = items.filter((item) => item.id !== id);

    if (filtered.length === items.length) {
      throw new Error(`Item with id ${id} not found`);
    }

    await fs.writeFile(this.filePath, JSON.stringify(filtered, null, 2));
    return true;
  }
}
```

### 📁 A File-based Logger Utility

```js
const fs = require('fs').promises;
const path = require('path');

class Logger {
  constructor(logDir = 'logs') {
    this.logDir = path.join(__dirname, logDir);
  }

  async init() {
    await fs.mkdir(this.logDir, { recursive: true });
  }

  // Get today's log file name (one file per day)
  _getLogFileName() {
    const date = new Date().toISOString().split('T')[0]; // '2026-02-24'
    return path.join(this.logDir, `${date}.log`);
  }

  // Format a log entry
  _formatEntry(level, message, data) {
    const timestamp = new Date().toISOString();
    let entry = `[${timestamp}] [${level.toUpperCase()}] ${message}`;
    if (data) {
      entry += ` | Data: ${JSON.stringify(data)}`;
    }
    return entry + '\n';
  }

  // Write a log entry
  async _log(level, message, data) {
    const entry = this._formatEntry(level, message, data);
    const logFile = this._getLogFileName();

    await fs.appendFile(logFile, entry);

    // Also print to console with colors
    const colors = {
      info: '\x1b[36m',    // Cyan
      warn: '\x1b[33m',    // Yellow
      error: '\x1b[31m',   // Red
      debug: '\x1b[90m',   // Gray
    };
    const reset = '\x1b[0m';
    console.log(`${colors[level] || ''}${entry.trim()}${reset}`);
  }

  async info(message, data) { await this._log('info', message, data); }
  async warn(message, data) { await this._log('warn', message, data); }
  async error(message, data) { await this._log('error', message, data); }
  async debug(message, data) { await this._log('debug', message, data); }

  // Read today's log
  async getTodaysLogs() {
    try {
      return await fs.readFile(this._getLogFileName(), 'utf-8');
    } catch {
      return 'No logs for today';
    }
  }

  // Clean up old logs (keep last N days)
  async cleanup(keepDays = 7) {
    const files = await fs.readdir(this.logDir);
    const cutoff = new Date();
    cutoff.setDate(cutoff.getDate() - keepDays);

    for (const file of files) {
      const dateStr = path.basename(file, '.log'); // '2026-02-17'
      const fileDate = new Date(dateStr);

      if (fileDate < cutoff) {
        await fs.unlink(path.join(this.logDir, file));
        console.log(`Deleted old log: ${file}`);
      }
    }
  }
}

// Usage:

  const logger = new Logger();
  await logger.init();

  await logger.info('Server started', { port: 3000 });
  await logger.warn('High memory usage', { usage: '85%' });
  await logger.error('Database connection failed', { host: 'localhost' });
  await logger.debug('Processing request', { url: '/api/users' });

  const logs = await logger.getTodaysLogs();
  console.log('\n--- Today\'s Logs ---\n', logs);
}

main().catch(console.error);
```

## 🌐 The `http` Module — Creating a Server from Scratch

### 🎯 What Is an HTTP Server?

An HTTP server is a program that:

1. **Listens** on a specific port (like a phone line)
2. **Receives** HTTP requests from clients (browsers, apps, other servers)
3. **Processes** those requests (reads data, talks to databases, computes things)
4. **Sends back** HTTP responses (HTML, JSON, files, etc.)

When you visit `http://localhost:3000` in your browser, your browser sends an HTTP request to **port 3000 on your own computer**. If a server is listening there, it receives the request and sends back a response that your browser displays.

```
  ┌──────────┐         HTTP Request          ┌──────────┐
  │          │  ──────────────────────────▶   │          │
  │  CLIENT  │  GET /api/users HTTP/1.1       │  SERVER  │
  │ (Browser)│                                │ (Node.js)│
  │          │  ◀──────────────────────────   │          │
  └──────────┘         HTTP Response          └──────────┘
                  200 OK
                  Content-Type: application/json
                  [{"name":"Alice"},{"name":"Bob"}]
```

---

### ▶️ The Simplest Possible HTTP Server

```js
const http = require('http');

// Create the server
const server = http.createServer((request, response) => {
  // This function runs for EVERY incoming request

  response.writeHead(200, { 'Content-Type': 'text/plain' });
  response.end('Hello, World!');
});

// Start listening for requests
server.listen(3000, () => {
  console.log('Server is running at http://localhost:3000');
});
```

**Let's break this down line by line:**

1. **`http.createServer(callback)`** — Creates a new server. The callback runs every time someone sends a request.

2. **`request` (often `req`)** — An object containing everything about the **incoming request**:
   - `request.url` — The path requested (e.g., `/api/users`)
   - `request.method` — The HTTP method (GET, POST, PUT, DELETE)
   - `request.headers` — The request headers

3. **`response` (often `res`)** — An object you use to **send back a response**:
   - `response.writeHead(statusCode, headers)` — Set the status code and headers
   - `response.write(data)` — Send data (can be called multiple times)
   - `response.end(data)` — End the response (optionally sending final data)

4. **`server.listen(port, callback)`** — Tells the server to start listening on the specified port

---

### 🧠 Understanding HTTP Methods and Status Codes

#### HTTP Methods (What the client wants to do)

| Method   | Purpose                    | Example                          |
|----------|----------------------------|----------------------------------|
| `GET`    | **Read** data              | Get a list of users              |
| `POST`   | **Create** new data        | Create a new user                |
| `PUT`    | **Replace** existing data   | Replace all user data            |
| `PATCH`  | **Update** part of data    | Change just the user's email     |
| `DELETE` | **Remove** data            | Delete a user                    |

#### HTTP Status Codes (How things went)

| Code | Meaning              | When to use                                  |
|------|----------------------|----------------------------------------------|
| 200  | OK                   | Request successful                           |
| 201  | Created              | New resource was created (POST)              |
| 204  | No Content           | Success, but nothing to return (DELETE)       |
| 400  | Bad Request          | Client sent invalid data                     |
| 401  | Unauthorized         | Client needs to log in                       |
| 403  | Forbidden            | Client is logged in but not allowed           |
| 404  | Not Found            | The requested resource doesn't exist          |
| 500  | Internal Server Error| Something broke on the server                |

```js
// Parse the JSON body from a request
function parseRequestBody(req) {
  return new Promise((resolve, reject) => {
    let body = '';

    // Data comes in chunks
    req.on('data', (chunk) => {
      body += chunk.toString();

      // Prevent memory attacks — limit body size to 1MB
      if (body.length > 1e6) {
        reject(new Error('Request body too large'));
      }
    });

    // All data received
    req.on('end', () => {
      try {
        resolve(body ? JSON.parse(body) : {});
      } catch {
        reject(new Error('Invalid JSON in request body'));
      }
    });

    req.on('error', reject);
  });
}

// Send a JSON response
function sendJson(res, statusCode, data) {
  res.writeHead(statusCode, {
    'Content-Type': 'application/json',
    'Access-Control-Allow-Origin': '*', // Allow cross-origin requests
  });
  res.end(JSON.stringify(data, null, 2));
}

// Send an error response
function sendError(res, statusCode, message) {
  sendJson(res, statusCode, { error: message });
}

  readUsers,
  writeUsers,
  parseRequestBody,
  sendJson,
  sendError,
```

```js
}

// GET /api/users/:id — Get a single user
async function getUserById(req, res, userId) {
  const users = await readUsers();
  const user = users.find((u) => u.id === userId);

  if (!user) {
    return sendError(res, 404, `User with id ${userId} not found`);
  }

  sendJson(res, 200, user);
}

// POST /api/users — Create a new user
async function createUser(req, res) {
  const body = await parseRequestBody(req);

  // Validation
  if (!body.name || !body.email) {
    return sendError(res, 400, 'Name and email are required');
  }

  const users = await readUsers();

  // Check for duplicate email
  if (users.some((u) => u.email === body.email)) {
    return sendError(res, 400, 'Email already exists');
  }

  const newUser = {
    id: Date.now(),
    name: body.name,
    email: body.email,
    age: body.age || null,
    createdAt: new Date().toISOString(),
  };

  users.push(newUser);
  await writeUsers(users);

  sendJson(res, 201, newUser);
}

// PUT /api/users/:id — Update a user
async function updateUser(req, res, userId) {
  const body = await parseRequestBody(req);
  const users = await readUsers();
  const index = users.findIndex((u) => u.id === userId);

  if (index === -1) {
    return sendError(res, 404, `User with id ${userId} not found`);
  }

  users[index] = {
    ...users[index],
    ...body,
    id: userId,   // Prevent changing the ID
    updatedAt: new Date().toISOString(),
  };

  await writeUsers(users);
  sendJson(res, 200, users[index]);
}

// DELETE /api/users/:id — Delete a user
async function deleteUser(req, res, userId) {
  const users = await readUsers();
  const index = users.findIndex((u) => u.id === userId);

  if (index === -1) {
    return sendError(res, 404, `User with id ${userId} not found`);
  }

  const deletedUser = users.splice(index, 1)[0];
  await writeUsers(users);

  sendJson(res, 200, { message: 'User deleted', user: deletedUser });
}

// ============================================
// ROUTER — Maps URLs to handlers
// ============================================

async function router(req, res) {
  // Parse the URL
  const baseURL = `http://${req.headers.host}`;
  const parsedUrl = new URL(req.url, baseURL);
  const pathname = parsedUrl.pathname;
  const method = req.method;

  console.log(`${method} ${pathname}`);

  // Handle CORS preflight requests
  if (method === 'OPTIONS') {
    res.writeHead(204, {
      'Access-Control-Allow-Origin': '*',
      'Access-Control-Allow-Methods': 'GET, POST, PUT, DELETE, OPTIONS',
      'Access-Control-Allow-Headers': 'Content-Type',
    });
    return res.end();
  }

  // Route: GET /
  if (method === 'GET' && pathname === '/') {
    return sendJson(res, 200, {
      message: 'Welcome to the Users API!',
      endpoints: {
        'GET /api/users': 'Get all users',
        'GET /api/users/:id': 'Get a user by ID',
        'POST /api/users': 'Create a new user',
        'PUT /api/users/:id': 'Update a user',
        'DELETE /api/users/:id': 'Delete a user',
      },
    });
  }

  // Route: GET /api/users
  if (method === 'GET' && pathname === '/api/users') {
    return getAllUsers(req, res);
  }

  // Route: POST /api/users
  if (method === 'POST' && pathname === '/api/users') {
    return createUser(req, res);
  }

  // Route: GET/PUT/DELETE /api/users/:id
  const userIdMatch = pathname.match(/^\/api\/users\/(\d+)$/);
  if (userIdMatch) {
    const userId = parseInt(userIdMatch[1]);

    if (method === 'GET') return getUserById(req, res, userId);
    if (method === 'PUT') return updateUser(req, res, userId);
    if (method === 'DELETE') return deleteUser(req, res, userId);
  }

  // Route not found
  sendError(res, 404, `Cannot ${method} ${pathname}`);
}

// ============================================
// CREATE AND START THE SERVER
// ============================================

const PORT = process.env.PORT || 3000;

const server = http.createServer(async (req, res) => {
  try {
    await router(req, res);
  } catch (error) {
    console.error('Server error:', error);
    sendError(res, 500, 'Internal Server Error');
  }
});

// Initialize data file, then start listening
initDataFile().then(() => {
  server.listen(PORT, () => {
    console.log(`
    ┌──────────────────────────────────────────┐
    │                                          │
    │   🚀 Server running on port ${PORT}         │
    │   http://localhost:${PORT}                   │
    │                                          │
    │   Endpoints:                             │
    │   GET    /api/users                      │
    │   GET    /api/users/:id                  │
    │   POST   /api/users                      │
    │   PUT    /api/users/:id                  │
    │   DELETE /api/users/:id                  │
    │                                          │
    └──────────────────────────────────────────┘
    `);
  });
});

// Handle server errors
server.on('error', (error) => {
  if (error.code === 'EADDRINUSE') {
    console.error(`Port ${PORT} is already in use!`);
  } else {
    console.error('Server error:', error);
  }
  process.exit(1);
});

// Graceful shutdown
process.on('SIGINT', () => {
  console.log('\nShutting down gracefully...');
  server.close(() => {
    console.log('Server closed');
    process.exit(0);
  });
});
```

### 🧠 Understanding `parseRequestBody` — How POST Data Arrives

When a client sends data (like a POST request with JSON), the data doesn't arrive all at once. It comes in **chunks** (small pieces). This is because of how networking works — data travels in packets.

```js
function parseRequestBody(req) {
  return new Promise((resolve, reject) => {
    let body = '';

    // 'data' event fires for EACH chunk of data that arrives
    req.on('data', (chunk) => {
      body += chunk.toString();
    });

    // 'end' event fires when ALL chunks have arrived
    req.on('end', () => {
      resolve(JSON.parse(body));
    });
  });
}
```

**Why a Promise?** Because receiving the body is **asynchronous** — we don't know when all the data will arrive. The Promise lets us `await` the complete body in our handler:

```js
const body = await parseRequestBody(req);
// Now body is a complete JavaScript object!
```

### ⚠️ Error Handling in the Server

Our server has **multiple layers** of error handling:

```js
// Layer 1: Individual handler validation
if (!body.name || !body.email) {
  return sendError(res, 400, 'Name and email are required');
}

// Layer 2: Try/catch in the router wrapper
const server = http.createServer(async (req, res) => {
  try {
    await router(req, res);
  } catch (error) {
    console.error('Server error:', error);
    sendError(res, 500, 'Internal Server Error');
  }
});

// Layer 3: Server error events
server.on('error', (error) => {
  console.error('Server error:', error);
});

// Layer 4: Process-level handling
process.on('SIGINT', () => {
  server.close(() => process.exit(0));
});
```

This ensures the server **never crashes** from an unhandled error — it always sends a proper error response to the client.

## 📋 Summary & Key Takeaways

### 📦 The `path` Module
- **Always** use `path.join()` or `path.resolve()` instead of string concatenation for file paths
- Use `__dirname` to build paths relative to the current file
- Use `path.parse()` to break a path into its components

### 📦 The `fs` Module
- Has **three APIs**: callback, synchronous, and promise-based
- **Always use the promise-based API** (`fs.promises` or `fs/promises`) with `async/await`
- Only use synchronous methods at startup (loading config files)
- **Never** use synchronous methods in request handlers

### 💡 Async Patterns
| Pattern    | Syntax               | Error Handling    | Readability |
|------------|----------------------|-------------------|-------------|
| Callbacks  | `fn(arg, callback)`  | `if (err)`        | Poor ❌     |
| Promises   | `.then().catch()`    | `.catch()`        | Good ✅     |
| async/await| `await fn(arg)`      | `try/catch`       | Best ✅✅   |

### 📦 The `http` Module
- `http.createServer()` creates a server with a request handler callback
- The handler receives `(request, response)` objects
- Parse the URL and method to route requests to the right handler
- Always send a response — if you don't, the client hangs forever
- Use `response.writeHead()` to set status codes and headers
- Use `response.end()` to send the response body and close the connection

### 📝 The Golden Rules
1. **Never block the event loop** — avoid `*Sync` methods in production code
2. **Always handle errors** — use `try/catch` with `async/await`
3. **Use `Promise.all()`** for independent parallel operations
4. **Use `for...of`** (not `forEach`) when iterating with `async/await`
5. **Send proper status codes** — 200 for success, 4xx for client errors, 5xx for server errors
6. **Always close the response** — call `res.end()` in every code path
