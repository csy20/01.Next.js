## What is Express?

Express is a **minimal and flexible web application framework** for Node.js. Think of it as a set of tools that makes building web servers incredibly easy compared to using raw Node.js.

### Why Do We Need Express?

With raw Node.js, creating a simple server that handles different URLs looks like this:

```js
// WITHOUT Express — raw Node.js HTTP server
const http = require("http");

const server = http.createServer((req, res) => {
  // We have to manually check the URL and method
  if (req.url === "/" && req.method === "GET") {
    res.writeHead(200, { "Content-Type": "text/plain" });
    res.end("Home Page");
  } else if (req.url === "/about" && req.method === "GET") {
    res.writeHead(200, { "Content-Type": "text/plain" });
    res.end("About Page");
  } else if (req.url === "/api/users" && req.method === "POST") {
    // Handling POST data manually is PAINFUL
    let body = "";
    req.on("data", (chunk) => {
      body += chunk.toString();
    });
    req.on("end", () => {
      const user = JSON.parse(body);
      res.writeHead(201, { "Content-Type": "application/json" });
      res.end(JSON.stringify({ message: "User created", user }));
    });
  } else {
    res.writeHead(404, { "Content-Type": "text/plain" });
    res.end("Not Found");
  }
});

server.listen(3000, () => {
  console.log("Server running on port 3000");
});
```

That's messy. Now imagine having 50 routes. It becomes unmanageable. With Express, the same thing looks like:

```js
// WITH Express — clean and organized
const express = require("express");
const app = express();

app.use(express.json()); // One line to parse JSON bodies

app.get("/", (req, res) => {
  res.send("Home Page");
});

app.get("/about", (req, res) => {
  res.send("About Page");
});

app.post("/api/users", (req, res) => {
  const user = req.body; // Body is already parsed!
  res.status(201).json({ message: "User created", user });
});

app.listen(3000, () => {
  console.log("Server running on port 3000");
});
```

**Express gives us:**

- Clean routing (matching URLs to code)
- Easy request/response handling
- Built-in body parsing
- Middleware support (we'll cover this in the next chapter)
- A massive ecosystem of plugins

## The Big Picture: How the Web Works Before Express

Before diving deeper, let's understand the **flow of a web request**:

```
┌──────────┐         HTTP Request          ┌──────────────┐
│          │  ──────────────────────────►   │              │
│  Client  │   GET /api/users?page=2       │    Server    │
│ (Browser │                               │  (Express)   │
│  or App) │  ◄──────────────────────────  │              │
│          │         HTTP Response          │              │
└──────────┘   { users: [...], page: 2 }   └──────────────┘
```

### Step-by-Step:

1. **Client sends a request** — The browser (or Postman, or a frontend app) sends an HTTP request. This request has:
   - A **method** (GET, POST, PUT, DELETE, etc.)
   - A **URL** (`/api/users?page=2`)
   - **Headers** (metadata like `Content-Type: application/json`)
   - A **body** (for POST/PUT — the data being sent)

2. **Server receives the request** — Express receives this raw HTTP request and wraps it in a nice `req` (request) object.

3. **Express matches the request to a route** — Express looks at the method + URL and finds the matching route handler you defined.

4. **Your route handler runs** — Your callback function executes, processes the request, and sends back a response.

5. **Client receives the response** — The client gets back the data, status code, and headers.

## Setting Up Express From Scratch

### Step 1: Initialize a Node.js Project

```bash
# Create a new folder and navigate into it
mkdir my-express-app
cd my-express-app

# Initialize a package.json file
# -y flag accepts all defaults automatically
npm init -y
```

This creates a `package.json` file — the "identity card" of your project. It tracks your project's name, version, dependencies, and scripts.

```json
{
  "name": "my-express-app",
  "version": "1.0.0",
  "description": "",
  "main": "index.js",
  "scripts": {
    "test": "echo \"Error: no test specified\" && exit 1"
  },
  "keywords": [],
  "author": "",
  "license": "ISC"
}
```

### Step 2: Install Express

```bash
npm install express
```

This does three things:

1. Downloads Express and all its dependencies into a `node_modules/` folder
2. Adds `"express": "^4.x.x"` to the `dependencies` section of `package.json`
3. Creates/updates a `package-lock.json` file (exact version snapshot)

### Step 3: Create Your Entry File

```bash
touch index.js
```

Now let's write the most basic Express server possible:

```js
// index.js

// Step 1: Import Express
// require() loads the Express module and returns a function
const express = require("express");

// Step 2: Create an Express application
// Calling express() creates a new application instance
// This 'app' object is the heart of your server
const app = express();

// Step 3: Define a port
// The port is like a "door number" on your computer
// Requests addressed to this port will reach your server
const PORT = 3000;

// Step 4: Start listening for requests
app.listen(PORT, () => {
  console.log(`Server is running on http://localhost:${PORT}`);
});
```

### Step 4: Run Your Server

```bash
node index.js
# Output: Server is running on http://localhost:3000
```

If you visit `http://localhost:3000` in your browser right now, you'll see:

```
Cannot GET /
```

This is Express telling you: "I received your GET request to `/`, but you haven't told me what to do with it." That's because we haven't defined any **routes** yet.

## Understanding the Express Application Object

When you call `express()`, you get back an **application object** (`app`). This object is the central piece of Express. Think of it as the **manager** of your web server.

```js
const app = express();
```

### What can `app` do?

| Method             | Purpose                                          |
| ------------------ | ------------------------------------------------ |
| `app.get()`        | Define a handler for GET requests                |
| `app.post()`       | Define a handler for POST requests               |
| `app.put()`        | Define a handler for PUT requests                |
| `app.delete()`     | Define a handler for DELETE requests             |
| `app.patch()`      | Define a handler for PATCH requests              |
| `app.all()`        | Handle ALL HTTP methods for a route              |
| `app.use()`        | Mount middleware (we'll cover this next chapter) |
| `app.listen()`     | Start the server on a specific port              |
| `app.set()`        | Set application-level settings                   |
| `app.route()`      | Create chainable route handlers                  |

## Starting a Server: `app.listen()`

```js
app.listen(PORT, callback);
```

### What Does `listen()` Actually Do?

Under the hood, `app.listen()` does this:

```js
// Simplified version of what Express does internally
app.listen = function (port, callback) {
  const server = http.createServer(this); // 'this' is the Express app
  return server.listen(port, callback);
};
```

So Express creates a raw Node.js HTTP server and passes itself as the request handler. Every incoming request goes through the Express app.

### Port Numbers Explained

- Ports range from **0 to 65535**
- Ports **0-1023** are "well-known" (reserved for system services — HTTP uses 80, HTTPS uses 443)
- Ports **1024-49151** are "registered" (common apps)
- Ports **49152-65535** are "dynamic" (temporary)
- For development, we typically use **3000, 5000, 8000, or 8080**

```js
// Common pattern: use environment variable or fallback to default
const PORT = process.env.PORT || 3000;

app.listen(PORT, () => {
  console.log(`Server running on port ${PORT}`);
});
```

> **Why `process.env.PORT`?** When you deploy to services like Heroku or Render, they assign a port dynamically via an environment variable. Your code needs to use that assigned port.

## What is a Route?

A **route** is a combination of:

1. **An HTTP method** (GET, POST, PUT, DELETE, etc.)
2. **A URL path** (`/`, `/about`, `/api/users`)
3. **A handler function** (the code that runs when the route is matched)

```js
app.METHOD(PATH, HANDLER);
```

Think of routes like a **restaurant menu**:

```
Route = Menu Item
─────────────────────────────────────────────
GET  /             → Serve the home page
GET  /about        → Serve the about page
GET  /api/users    → Return a list of users
POST /api/users    → Create a new user
GET  /api/users/5  → Return user with ID 5
PUT  /api/users/5  → Update user with ID 5
DELETE /api/users/5 → Delete user with ID 5
```

When a request comes in, Express goes through your defined routes **from top to bottom** and runs the **first match**.

## Handling GET Requests

GET requests are used to **retrieve data**. The browser makes GET requests when you type a URL, click a link, or load a page.

### Basic Syntax

```js
app.get(path, handlerFunction);
```

The handler function receives two objects:

- `req` — the **request** object (information about the incoming request)
- `res` — the **response** object (tools to send back a response)

### Examples

```js
// Example 1: Simple text response
app.get("/", (req, res) => {
  res.send("Hello, World!");
});

// Example 2: Sending JSON (most common for APIs)
app.get("/api/status", (req, res) => {
  res.json({
    status: "running",
    uptime: process.uptime(),
    timestamp: new Date().toISOString(),
  });
});

// Example 3: Sending HTML
app.get("/welcome", (req, res) => {
  res.send("<h1>Welcome to my website!</h1><p>This is HTML.</p>");
});

// Example 4: Sending a status code with the response
app.get("/not-found-example", (req, res) => {
  res.status(404).send("This page does not exist");
});

// Example 5: Returning a list of items (simulating a database)
const users = [
  { id: 1, name: "Alice", email: "alice@example.com" },
  { id: 2, name: "Bob", email: "bob@example.com" },
  { id: 3, name: "Charlie", email: "charlie@example.com" },
];

app.get("/api/users", (req, res) => {
  res.json(users);
});
```

### What Happens When Express Matches a GET Route

```
Browser types: http://localhost:3000/api/users

1. Browser creates HTTP request:
   ┌─────────────────────────────┐
   │ GET /api/users HTTP/1.1     │  ← Method + Path
   │ Host: localhost:3000        │  ← Header
   │ Accept: text/html, */*      │  ← Header
   └─────────────────────────────┘

2. Express receives the request
3. Express checks: "Do I have a route for GET /api/users?"
4. YES → Runs the handler function
5. Handler calls res.json(users)
6. Express sends back:
   ┌─────────────────────────────────────┐
   │ HTTP/1.1 200 OK                     │
   │ Content-Type: application/json      │
   │                                     │
   │ [{"id":1,"name":"Alice",...}, ...]  │
   └─────────────────────────────────────┘
```

## Handling POST Requests

POST requests are used to **send data to the server** — typically to create something new (a user, a blog post, an order, etc.).

### Basic Syntax

```js
app.post(path, handlerFunction);
```

### The Key Difference: POST Requests Have a Body

Unlike GET requests, POST requests carry **data in the body** of the request. For example, when you fill out a registration form and click "Submit," the browser sends a POST request with your form data in the body.

```
POST /api/users HTTP/1.1
Host: localhost:3000
Content-Type: application/json        ← Tells the server the format of the body

{                                      ← This is the BODY
  "name": "Diana",
  "email": "diana@example.com"
}
```

### Important: You Must Enable Body Parsing First!

Express does **NOT** parse the request body by default. The body arrives as a raw stream of bytes. You need to tell Express how to parse it:

```js
// Add this BEFORE your routes
app.use(express.json()); // Parse JSON bodies
```

We'll explain this in detail in the [Parsing Incoming Data](#parsing-incoming-data-why-we-need-body-parsing) section.

### Examples

```js
const express = require("express");
const app = express();

// Enable JSON body parsing
app.use(express.json());

// In-memory "database" (just an array for learning purposes)
let users = [
  { id: 1, name: "Alice", email: "alice@example.com" },
  { id: 2, name: "Bob", email: "bob@example.com" },
];
let nextId = 3; // Simple ID counter

// POST — Create a new user
app.post("/api/users", (req, res) => {
  // req.body contains the parsed JSON data sent by the client
  const { name, email } = req.body;

  // Validation: Check if required fields are provided
  if (!name || !email) {
    return res.status(400).json({
      error: "Both 'name' and 'email' are required",
    });
  }

  // Create the new user object
  const newUser = {
    id: nextId++,
    name,
    email,
  };

  // Add to our "database"
  users.push(newUser);

  // Send back the created user with 201 status
  // 201 means "Created" — something new was successfully made
  res.status(201).json({
    message: "User created successfully",
    user: newUser,
  });
});
```

### Testing POST Requests

You can't test POST requests from a browser URL bar (that only sends GET). Use one of these:

**Option 1: curl (command line)**

```bash
curl -X POST http://localhost:3000/api/users \
  -H "Content-Type: application/json" \
  -d '{"name": "Diana", "email": "diana@example.com"}'
```

**Option 2: Postman or Thunder Client (VS Code extension)**

- Set method to POST
- Set URL to `http://localhost:3000/api/users`
- In the Body tab, select "raw" → "JSON"
- Enter: `{"name": "Diana", "email": "diana@example.com"}`
- Click Send

**Option 3: JavaScript fetch (from a frontend or another script)**

```js
fetch("http://localhost:3000/api/users", {
  method: "POST",
  headers: {
    "Content-Type": "application/json",
  },
  body: JSON.stringify({
    name: "Diana",
    email: "diana@example.com",
  }),
})
  .then((res) => res.json())
  .then((data) => console.log(data));
```

## The Request Object (`req`) in Detail

The `req` object represents the **incoming HTTP request**. Express enhances the raw Node.js request with many useful properties and methods.

### Most Important `req` Properties

```js
app.get("/example", (req, res) => {
  console.log(req.method);    // "GET" — the HTTP method
  console.log(req.url);       // "/example?search=hello" — the full URL
  console.log(req.path);      // "/example" — just the path (no query string)
  console.log(req.params);    // {} — route parameters (covered below)
  console.log(req.query);     // { search: "hello" } — query string parsed
  console.log(req.body);      // {} or parsed body data (for POST/PUT)
  console.log(req.headers);   // { host: "localhost:3000", ... } — all headers
  console.log(req.ip);        // "127.0.0.1" — client's IP address
  console.log(req.hostname);  // "localhost" — hostname from Host header
  console.log(req.protocol);  // "http" — protocol used
});
```

### Visual Map of Where Data Lives in a Request

```
http://localhost:3000/api/users/42?sort=name&order=asc
│         │              │     │  │                   │
│         │              │     │  └───────────────────┘
│         │              │     │     req.query = { sort: "name", order: "asc" }
│         │              │     │
│         │              │     └── req.params = { id: "42" }  (if route is /api/users/:id)
│         │              │
│         │              └──────── req.path = "/api/users/42"
│         │
│         └──────────────────────── req.hostname = "localhost"
│
└────────────────────────────────── req.protocol = "http"


POST /api/users HTTP/1.1              ← req.method = "POST"
Content-Type: application/json        ← req.headers["content-type"]
Authorization: Bearer abc123          ← req.headers["authorization"]

{                                     ← req.body (after parsing)
  "name": "Diana",
  "email": "diana@example.com"
}
```

## The Response Object (`res`) in Detail

The `res` object represents the **outgoing HTTP response**. It provides methods to send data back to the client.

### Most Important `res` Methods

```js
app.get("/demo", (req, res) => {
  // Sending responses (use only ONE per request):

  res.send("Hello");           // Send a string (auto-sets Content-Type to text/html)
  res.json({ key: "value" });  // Send JSON (auto-sets Content-Type to application/json)
  res.sendFile("/path/file");  // Send a file
  res.redirect("/other-url");  // Redirect to another URL
  res.render("template");      // Render a template (EJS, Pug, etc.)

  // Setting status codes:
  res.status(200);             // OK (default)
  res.status(201);             // Created
  res.status(400);             // Bad Request
  res.status(404);             // Not Found
  res.status(500);             // Internal Server Error

  // Chaining status with response:
  res.status(404).json({ error: "User not found" });
  res.status(201).json({ message: "Created!" });

  // Setting headers:
  res.set("X-Custom-Header", "my-value");
  res.set("Cache-Control", "no-store");
});
```

### Common HTTP Status Codes

| Code | Name                  | When to Use                              |
| ---- | --------------------- | ---------------------------------------- |
| 200  | OK                    | Request succeeded (default)              |
| 201  | Created               | New resource was created (POST success)  |
| 204  | No Content            | Success but nothing to return (DELETE)   |
| 400  | Bad Request           | Client sent invalid data                 |
| 401  | Unauthorized          | Authentication required                  |
| 403  | Forbidden             | Authenticated but no permission          |
| 404  | Not Found             | Resource doesn't exist                   |
| 409  | Conflict              | Resource already exists (duplicate)      |
| 500  | Internal Server Error | Something went wrong on the server       |

### `res.send()` vs `res.json()`

```js
// res.send() — Smart and flexible
res.send("Hello");            // Sends text/html
res.send({ name: "Alice" });  // Detects object → sends application/json
res.send(Buffer.from("hi"));  // Sends application/octet-stream

// res.json() — Explicitly for JSON
res.json({ name: "Alice" });  // Always sends application/json
res.json(null);               // Sends "null" as JSON
res.json([1, 2, 3]);          // Sends an array as JSON
```

> **Best Practice:** Use `res.json()` when sending JSON data. It's more explicit and also handles edge cases like `null` and `undefined` better than `res.send()`.

## Route Parameters (`req.params`)

### What Are Route Parameters?

Route parameters are **dynamic segments** in a URL path. They act as placeholders that capture values from the actual URL.

Think of it like a **fill-in-the-blank** in a URL:

```
Route:   /api/users/:id
                     ^^^
                     This is a parameter — it will match anything

URL:     /api/users/42
                    ^^
                    This value "42" gets captured as req.params.id
```

### Syntax: Use a Colon (`:`) to Define Parameters

```js
// :id is a route parameter
app.get("/api/users/:id", (req, res) => {
  console.log(req.params); // { id: "42" }   ← always a STRING
  console.log(req.params.id); // "42"
});
```

### Important: Route Parameter Values Are Always Strings

Even if the URL contains a number like `/users/42`, `req.params.id` will be the **string** `"42"`, not the number `42`. You need to convert it yourself if you need a number:

```js
app.get("/api/users/:id", (req, res) => {
  const id = parseInt(req.params.id, 10); // Convert string to number
  // Or: const id = Number(req.params.id);

  if (isNaN(id)) {
    return res.status(400).json({ error: "Invalid user ID" });
  }

  const user = users.find((u) => u.id === id);

  if (!user) {
    return res.status(404).json({ error: "User not found" });
  }

  res.json(user);
});
```

### Multiple Route Parameters

You can have **multiple parameters** in a single route:

```js
// Two parameters: userId and postId
app.get("/api/users/:userId/posts/:postId", (req, res) => {
  console.log(req.params);
  // If URL is /api/users/5/posts/12:
  // { userId: "5", postId: "12" }

  const { userId, postId } = req.params;

  res.json({
    message: `Fetching post ${postId} from user ${userId}`,
  });
});
```

### How Express Matches Route Parameters

```
Defined Route:     /api/users/:id/posts/:postId

Incoming URL:      /api/users/7/posts/99

Matching Process:
  /api     →  /api       ✓ literal match
  /users   →  /users     ✓ literal match
  /:id     →  /7         ✓ parameter captured → id = "7"
  /posts   →  /posts     ✓ literal match
  /:postId →  /99        ✓ parameter captured → postId = "99"

Result: req.params = { id: "7", postId: "99" }
```

### Full CRUD Example with Route Parameters

```js
const express = require("express");
const app = express();
app.use(express.json());

let users = [
  { id: 1, name: "Alice", email: "alice@example.com" },
  { id: 2, name: "Bob", email: "bob@example.com" },
  { id: 3, name: "Charlie", email: "charlie@example.com" },
];
let nextId = 4;

// GET all users
app.get("/api/users", (req, res) => {
  res.json(users);
});

// GET a single user by ID
app.get("/api/users/:id", (req, res) => {
  const id = parseInt(req.params.id, 10);
  const user = users.find((u) => u.id === id);

  if (!user) {
    return res.status(404).json({ error: "User not found" });
  }

  res.json(user);
});

// POST — Create a new user
app.post("/api/users", (req, res) => {
  const { name, email } = req.body;

  if (!name || !email) {
    return res.status(400).json({ error: "Name and email are required" });
  }

  const newUser = { id: nextId++, name, email };
  users.push(newUser);
  res.status(201).json(newUser);
});

// PUT — Update an entire user (replace)
app.put("/api/users/:id", (req, res) => {
  const id = parseInt(req.params.id, 10);
  const index = users.findIndex((u) => u.id === id);

  if (index === -1) {
    return res.status(404).json({ error: "User not found" });
  }

  const { name, email } = req.body;

  if (!name || !email) {
    return res.status(400).json({ error: "Name and email are required" });
  }

  users[index] = { id, name, email }; // Replace entirely
  res.json(users[index]);
});

// PATCH — Partially update a user
app.patch("/api/users/:id", (req, res) => {
  const id = parseInt(req.params.id, 10);
  const user = users.find((u) => u.id === id);

  if (!user) {
    return res.status(404).json({ error: "User not found" });
  }

  // Only update fields that were provided
  if (req.body.name) user.name = req.body.name;
  if (req.body.email) user.email = req.body.email;

  res.json(user);
});

// DELETE — Remove a user
app.delete("/api/users/:id", (req, res) => {
  const id = parseInt(req.params.id, 10);
  const index = users.findIndex((u) => u.id === id);

  if (index === -1) {
    return res.status(404).json({ error: "User not found" });
  }

  const deleted = users.splice(index, 1); // Remove from array
  res.json({ message: "User deleted", user: deleted[0] });
});

app.listen(3000, () => {
  console.log("Server running on port 3000");
});
```

## Query Strings (`req.query`)

### What Are Query Strings?

Query strings are **key-value pairs** appended to the end of a URL after a `?` symbol. They are used to pass **optional, extra information** along with a request.

```
https://example.com/api/users?sort=name&order=asc&limit=10
                              │                           │
                              └───────── query string ────┘

?sort=name    → key: "sort",  value: "name"
&order=asc    → key: "order", value: "asc"
&limit=10     → key: "limit", value: "10"
```

### Important Rules

- Query strings start with `?`
- Multiple pairs are separated by `&`
- Format is `key=value`
- **All values are strings** (even numbers like `10` come as `"10"`)
- They are **optional** — the route matches with or without them

### How Express Parses Query Strings

Express **automatically** parses query strings. No setup needed!

```js
// URL: http://localhost:3000/api/users?sort=name&order=asc&limit=10

app.get("/api/users", (req, res) => {
  console.log(req.query);
  // { sort: "name", order: "asc", limit: "10" }

  console.log(req.query.sort);   // "name"
  console.log(req.query.order);  // "asc"
  console.log(req.query.limit);  // "10"  ← It's a STRING, not a number!
});
```

### Practical Example: Filtering, Sorting, and Pagination

```js
const users = [
  { id: 1, name: "Alice", age: 30, role: "admin" },
  { id: 2, name: "Bob", age: 25, role: "user" },
  { id: 3, name: "Charlie", age: 35, role: "user" },
  { id: 4, name: "Diana", age: 28, role: "admin" },
  { id: 5, name: "Eve", age: 22, role: "user" },
  { id: 6, name: "Frank", age: 40, role: "moderator" },
];

app.get("/api/users", (req, res) => {
  let result = [...users]; // Copy the array so we don't modify the original

  // ── FILTERING ──
  // URL: /api/users?role=admin
  if (req.query.role) {
    result = result.filter((u) => u.role === req.query.role);
  }

  // URL: /api/users?minAge=25
  if (req.query.minAge) {
    const minAge = parseInt(req.query.minAge, 10);
    result = result.filter((u) => u.age >= minAge);
  }

  // ── SORTING ──
  // URL: /api/users?sort=name&order=desc
  if (req.query.sort) {
    const sortField = req.query.sort;    // e.g., "name" or "age"
    const order = req.query.order || "asc"; // default to ascending

    result.sort((a, b) => {
      if (a[sortField] < b[sortField]) return order === "asc" ? -1 : 1;
      if (a[sortField] > b[sortField]) return order === "asc" ? 1 : -1;
      return 0;
    });
  }

  // ── PAGINATION ──
  // URL: /api/users?page=2&limit=3
  const page = parseInt(req.query.page, 10) || 1;  // default page 1
  const limit = parseInt(req.query.limit, 10) || 10; // default 10 per page
  const startIndex = (page - 1) * limit;
  const endIndex = page * limit;

  const paginatedResult = result.slice(startIndex, endIndex);

  // Send response with metadata
  res.json({
    total: result.length,
    page,
    limit,
    totalPages: Math.ceil(result.length / limit),
    data: paginatedResult,
  });
});
```

### Example Requests and Responses

```bash
# Get all admin users
GET /api/users?role=admin
# → Returns Alice and Diana

# Get users older than 25, sorted by age descending
GET /api/users?minAge=25&sort=age&order=desc
# → Returns Frank(40), Charlie(35), Alice(30), Diana(28), Bob(25)

# Get page 2 with 2 users per page
GET /api/users?page=2&limit=2
# → Returns Charlie and Diana (items 3-4)

# Combine everything: admin users, sorted by name, page 1, limit 5
GET /api/users?role=admin&sort=name&page=1&limit=5
# → Returns Alice, Diana
```

```js
// URL: /search?featured
app.get("/search", (req, res) => {
  console.log(req.query.featured); // "" (empty string, but it IS present)

  // Check if a flag exists (even without a value)
  if ("featured" in req.query) {
    console.log("Featured filter is active");
  }
});
```

### Query Strings with Arrays

```js
// URL: /api/users?tags=javascript&tags=nodejs
app.get("/api/users", (req, res) => {
  console.log(req.query.tags); // ["javascript", "nodejs"]
  // Express automatically creates an array when the same key appears multiple times

// You can also use bracket notation:
// URL: /api/users?tags[]=javascript&tags[]=nodejs
```

## Route Parameters vs Query Strings

This is a common source of confusion. Here's how to decide which to use:

### When to Use Route Parameters (`:param`)

Use route parameters for **required, identifying** information — things that define **which specific resource** you're looking for.

```
GET /api/users/42         → Get user #42 (the ID identifies the resource)
GET /api/posts/7/comments → Get all comments for post #7
DELETE /api/users/42      → Delete user #42
```

### When to Use Query Strings (`?key=value`)

Use query strings for **optional, modifying** information — things that change **how** you get the data, not **which** data.

```
GET /api/users?sort=name&order=asc     → Get users, sorted by name
GET /api/users?page=2&limit=10         → Get users, page 2
GET /api/users?role=admin              → Get users, filtered by role
GET /api/products?minPrice=10&maxPrice=50 → Products in price range
GET /api/search?q=javascript           → Search for "javascript"
```

### Side-by-Side Comparison

```
┌────────────────────────────────────────────────────────────────┐
│           Feature         │  Route Params  │  Query Strings    │
├───────────────────────────┼────────────────┼───────────────────┤
│  Purpose                  │  Identify      │  Filter/Modify    │
│  Required?                │  Yes           │  No (optional)    │
│  Where in URL             │  In the path   │  After ? symbol   │
│  Syntax                   │  /users/:id    │  /users?key=val   │
│  Access in Express        │  req.params    │  req.query        │
│  Example URL              │  /users/42     │  /users?sort=name │
│  Without it, route works? │  No — 404      │  Yes — uses defaults│
└────────────────────────────────────────────────────────────────┘
```

### Real-World URL Examples

```
Spotify-like API:

GET /api/artists/abc123                     ← params: artist ID
GET /api/artists/abc123/albums              ← params: artist ID
GET /api/artists/abc123/albums?year=2024    ← params + query
GET /api/search?q=taylor+swift&type=artist  ← all query strings
GET /api/playlists/xyz789/tracks?limit=50   ← params + query
```

## Parsing Incoming Data: Why We Need Body Parsing

### The Problem

When a client sends data in a POST or PUT request, the data arrives at the server as a **raw stream of bytes** (think of it like a fire hose of characters). Express does NOT automatically read and decode this stream:

```js
// WITHOUT body parsing middleware
app.post("/api/users", (req, res) => {
  console.log(req.body); // undefined!! 😱
  // Express didn't parse the body, so it's undefined
});
```

### What Is a Request Body?

The body is the **payload** of an HTTP request. Not all requests have bodies:

| Method | Has Body? | Typical Use                     |
| ------ | --------- | ------------------------------- |
| GET    | No*       | Retrieve data                   |
| POST   | Yes       | Create new data                 |
| PUT    | Yes       | Replace existing data           |
| PATCH  | Yes       | Partially update existing data  |
| DELETE | No*       | Delete data                     |

> *Technically GET and DELETE CAN have bodies, but it's unconventional and discouraged.

### How Data Is Sent

Clients can send data in different **formats** (indicated by the `Content-Type` header):

```
1. JSON (most common for APIs):
   Content-Type: application/json
   Body: {"name": "Alice", "email": "alice@example.com"}

2. URL-encoded (HTML forms):
   Content-Type: application/x-www-form-urlencoded
   Body: name=Alice&email=alice%40example.com

3. Multipart (file uploads):
   Content-Type: multipart/form-data
   Body: (binary data with boundaries)

4. Plain text:
   Content-Type: text/plain
   Body: Hello, this is plain text
```

Express provides **built-in middleware** to handle the two most common formats: JSON and URL-encoded.

## Parsing JSON Bodies (`express.json()`)

### What It Does

`express.json()` is a built-in middleware that:

1. Checks if the incoming request has `Content-Type: application/json`
2. If yes, reads the raw body stream
3. Parses the JSON string into a JavaScript object
4. Attaches it to `req.body`

### How to Use

```js
const express = require("express");
const app = express();

// Add this line BEFORE any routes that need to read req.body
app.use(express.json());
```

### What `app.use()` Does Here

`app.use()` tells Express: "Run this function on **every incoming request** before it reaches any route handler." It's like a **checkpoint** that every request must pass through.

```
Request comes in → express.json() checks it → Your route handler
                   ├── Has JSON body? Parse it, attach to req.body
                   └── No JSON body? Just move on (req.body = undefined)
```

### Step-by-Step: What Happens Under the Hood

```js
app.use(express.json());

app.post("/api/users", (req, res) => {
  console.log(req.body); // { name: "Alice", email: "alice@example.com" }
});
```

When a POST request arrives:

```
Step 1: Client sends request
   POST /api/users HTTP/1.1
   Content-Type: application/json
   
   {"name":"Alice","email":"alice@example.com"}

Step 2: express.json() middleware kicks in
   - Sees Content-Type is application/json  ✓
   - Reads the raw body: '{"name":"Alice","email":"alice@example.com"}'
   - Runs JSON.parse() on it
   - Attaches result to req.body
   - req.body = { name: "Alice", email: "alice@example.com" }

Step 3: Your route handler runs
   - req.body is now a nice JavaScript object
   - You can use req.body.name, req.body.email, etc.
```

### Options for `express.json()`

```js
app.use(
  express.json({
    limit: "10kb", // Maximum body size (default: 100kb)
    // Rejects any body larger than 10KB
    // Protects against someone sending a massive payload

    strict: true, // Only accept arrays and objects (default: true)
    // If false, also accepts primitives like "hello" or 42

    type: "application/json", // Which Content-Type to parse (default)
    // You usually don't need to change this
  })
);
```

### What Happens If JSON Is Invalid?

If someone sends malformed JSON, `express.json()` will throw a `SyntaxError`:

```js
// Client sends: { name: "Alice"  ← missing closing brace
// Express responds with 400 Bad Request automatically

// You can also handle this with an error handler:
app.use((err, req, res, next) => {
  if (err instanceof SyntaxError && err.status === 400 && "body" in err) {
    return res.status(400).json({ error: "Invalid JSON in request body" });
  }
  next(err);
});
```

## Parsing URL-Encoded Bodies (`express.urlencoded()`)

### What Is URL-Encoded Data?

When you submit an HTML form (without JavaScript), the browser sends data in **URL-encoded** format:

```html
<form action="/api/register" method="POST">
  <input type="text" name="username" value="alice_123" />
  <input type="email" name="email" value="alice@example.com" />
  <input type="password" name="password" value="secret123" />
  <button type="submit">Register</button>
</form>
```

When submitted, the browser sends:

```
POST /api/register HTTP/1.1
Content-Type: application/x-www-form-urlencoded

username=alice_123&email=alice%40example.com&password=secret123
```

Notice:

- Data is formatted as `key=value` pairs separated by `&`
- Special characters are **encoded** (`@` becomes `%40`, spaces become `+` or `%20`)
- It looks exactly like a query string, but it's in the **body**, not the URL

### How to Use

```js
app.use(express.urlencoded({ extended: true }));
```

### The `extended` Option Explained

```js
// extended: false — uses the 'querystring' library (simpler)
// Only supports flat key-value pairs
app.use(express.urlencoded({ extended: false }));
// Body: name=Alice&age=30
// req.body = { name: "Alice", age: "30" }

// extended: true — uses the 'qs' library (more powerful)
// Supports nested objects and arrays
app.use(express.urlencoded({ extended: true }));
// Body: user[name]=Alice&user[age]=30&tags[]=js&tags[]=node
// req.body = { user: { name: "Alice", age: "30" }, tags: ["js", "node"] }
```

> **Best Practice:** Use `extended: true` in most cases. It's more flexible and handles complex form data.

### Example: Handling Form Submissions

```js
const express = require("express");
const app = express();

// Parse both JSON and URL-encoded bodies
app.use(express.json());
app.use(express.urlencoded({ extended: true }));

// Serve a simple HTML form
app.get("/register", (req, res) => {
  res.send(`
    <html>
      <body>
        <h1>Register</h1>
        <form action="/register" method="POST">
          <label>Username: <input type="text" name="username" /></label><br/>
          <label>Email: <input type="email" name="email" /></label><br/>
          <label>Password: <input type="password" name="password" /></label><br/>
          <button type="submit">Register</button>
        </form>
      </body>
    </html>
  `);
});

// Handle form submission
app.post("/register", (req, res) => {
  // req.body is parsed automatically by express.urlencoded()
  const { username, email, password } = req.body;

  console.log("Form data received:");
  console.log("Username:", username);
  console.log("Email:", email);
  console.log("Password:", password);

  // In real apps, you'd hash the password and save to a database
  res.send(`
    <html>
      <body>
        <h1>Welcome, ${username}!</h1>
        <p>Your account has been created with email: ${email}</p>
      </body>
    </html>
  `);
});

app.listen(3000, () => {
  console.log("Server running on port 3000");
});
```

### Using Both Parsers Together

In most applications, you want **both** parsers active since you'll receive data from both APIs (JSON) and HTML forms (URL-encoded):

```js
const express = require("express");
const app = express();

// Parse JSON bodies (from API clients like fetch, axios, Postman)
app.use(express.json());

// Parse URL-encoded bodies (from HTML forms)
app.use(express.urlencoded({ extended: true }));

// Now req.body works for both formats!
app.post("/api/data", (req, res) => {
  console.log(req.body);
  // Works whether client sent JSON or URL-encoded data
  res.json({ received: req.body });
});
```

### How Express Knows Which Parser to Use

```
Request arrives with Content-Type header:

  Content-Type: application/json
  → express.json() handles it

  Content-Type: application/x-www-form-urlencoded
  → express.urlencoded() handles it

  Content-Type: text/plain
  → Neither handles it (req.body remains undefined)

  No Content-Type header
  → Neither handles it (req.body remains undefined)
```

## Putting It All Together: A Complete Example

Here's a complete Express server that demonstrates everything we've covered:

```js
const express = require("express");
const app = express();
const PORT = process.env.PORT || 3000;

// ────────────────────────────────────────────────
// MIDDLEWARE — Body Parsing
// ────────────────────────────────────────────────
app.use(express.json());                         // Parse JSON bodies
app.use(express.urlencoded({ extended: true })); // Parse form bodies

// ────────────────────────────────────────────────
// "DATABASE" — In-memory storage for demo
// ────────────────────────────────────────────────
let books = [
  { id: 1, title: "The Great Gatsby", author: "F. Scott Fitzgerald", year: 1925, genre: "fiction" },
  { id: 2, title: "To Kill a Mockingbird", author: "Harper Lee", year: 1960, genre: "fiction" },
  { id: 3, title: "1984", author: "George Orwell", year: 1949, genre: "dystopian" },
  { id: 4, title: "Sapiens", author: "Yuval Noah Harari", year: 2011, genre: "non-fiction" },
  { id: 5, title: "Clean Code", author: "Robert C. Martin", year: 2008, genre: "technical" },
];
let nextId = 6;

// ────────────────────────────────────────────────
// ROUTES
// ────────────────────────────────────────────────

// Home route
app.get("/", (req, res) => {
  res.json({
    message: "Welcome to the Bookstore API",
    endpoints: {
      "GET /api/books": "Get all books (supports ?genre, ?sort, ?page, ?limit)",
      "GET /api/books/:id": "Get a specific book",
      "POST /api/books": "Create a new book",
      "PUT /api/books/:id": "Update a book",
      "DELETE /api/books/:id": "Delete a book",
      "GET /api/search": "Search books by title (?q=keyword)",
    },
  });
});

// GET all books with filtering, sorting, and pagination
app.get("/api/books", (req, res) => {
  let result = [...books]; // Don't mutate the original array

  // ── Filter by genre (query string) ──
  if (req.query.genre) {
    result = result.filter(
      (book) => book.genre.toLowerCase() === req.query.genre.toLowerCase()
    );
  }

  // ── Filter by year range (query strings) ──
  if (req.query.yearFrom) {
    const yearFrom = parseInt(req.query.yearFrom, 10);
    result = result.filter((book) => book.year >= yearFrom);
  }
  if (req.query.yearTo) {
    const yearTo = parseInt(req.query.yearTo, 10);
    result = result.filter((book) => book.year <= yearTo);
  }

  // ── Sort (query string) ──
  if (req.query.sort) {
    const sortField = req.query.sort;
    const order = req.query.order === "desc" ? -1 : 1;

    result.sort((a, b) => {
      if (a[sortField] < b[sortField]) return -1 * order;
      if (a[sortField] > b[sortField]) return 1 * order;
      return 0;
    });
  }

  // ── Pagination (query strings) ──
  const page = parseInt(req.query.page, 10) || 1;
  const limit = parseInt(req.query.limit, 10) || 10;
  const startIndex = (page - 1) * limit;
  const paginatedResult = result.slice(startIndex, startIndex + limit);

  res.json({
    total: result.length,
    page,
    limit,
    totalPages: Math.ceil(result.length / limit),
    data: paginatedResult,
  });
});

// GET a single book by ID (route parameter)
app.get("/api/books/:id", (req, res) => {
  const id = parseInt(req.params.id, 10);

  if (isNaN(id)) {
    return res.status(400).json({ error: "Invalid book ID — must be a number" });
  }

  const book = books.find((b) => b.id === id);

  if (!book) {
    return res.status(404).json({ error: `Book with ID ${id} not found` });
  }

  res.json(book);
});

// SEARCH books by title (query string)
app.get("/api/search", (req, res) => {
  const query = req.query.q;

  if (!query) {
    return res.status(400).json({ error: "Search query 'q' is required" });
  }

  const results = books.filter((book) =>
    book.title.toLowerCase().includes(query.toLowerCase())
  );

  res.json({
    query,
    count: results.length,
    results,
  });
});

// POST — Create a new book (JSON body)
app.post("/api/books", (req, res) => {
  const { title, author, year, genre } = req.body;

  // Validate required fields
  const errors = [];
  if (!title) errors.push("'title' is required");
  if (!author) errors.push("'author' is required");
  if (!year) errors.push("'year' is required");
  if (!genre) errors.push("'genre' is required");

  if (errors.length > 0) {
    return res.status(400).json({ errors });
  }

  // Validate year is a number
  if (typeof year !== "number" || year < 0) {
    return res.status(400).json({ error: "'year' must be a positive number" });
  }

  const newBook = {
    id: nextId++,
    title,
    author,
    year,
    genre: genre.toLowerCase(),
  };

  books.push(newBook);

  res.status(201).json({
    message: "Book created successfully",
    book: newBook,
  });
});

// PUT — Update (replace) a book by ID
app.put("/api/books/:id", (req, res) => {
  const id = parseInt(req.params.id, 10);
  const index = books.findIndex((b) => b.id === id);

  if (index === -1) {
    return res.status(404).json({ error: `Book with ID ${id} not found` });
  }

  const { title, author, year, genre } = req.body;

  if (!title || !author || !year || !genre) {
    return res.status(400).json({
      error: "All fields (title, author, year, genre) are required for PUT",
    });
  }

  books[index] = { id, title, author, year, genre: genre.toLowerCase() };
  res.json({ message: "Book updated", book: books[index] });
});

// PATCH — Partially update a book
app.patch("/api/books/:id", (req, res) => {
  const id = parseInt(req.params.id, 10);
  const book = books.find((b) => b.id === id);

  if (!book) {
    return res.status(404).json({ error: `Book with ID ${id} not found` });
  }

  // Only update provided fields
  if (req.body.title) book.title = req.body.title;
  if (req.body.author) book.author = req.body.author;
  if (req.body.year) book.year = req.body.year;
  if (req.body.genre) book.genre = req.body.genre.toLowerCase();

  res.json({ message: "Book partially updated", book });
});

// DELETE — Remove a book
app.delete("/api/books/:id", (req, res) => {
  const id = parseInt(req.params.id, 10);
  const index = books.findIndex((b) => b.id === id);

  if (index === -1) {
    return res.status(404).json({ error: `Book with ID ${id} not found` });
  }

  const deleted = books.splice(index, 1);
  res.json({ message: "Book deleted", book: deleted[0] });
});

// ────────────────────────────────────────────────
// 404 Handler — Must be AFTER all other routes
// ────────────────────────────────────────────────
app.use((req, res) => {
  res.status(404).json({
    error: "Route not found",
    hint: "Visit GET / to see available endpoints",
  });
});

// ────────────────────────────────────────────────
// START SERVER
// ────────────────────────────────────────────────
app.listen(PORT, () => {
  console.log(`📚 Bookstore API running on http://localhost:${PORT}`);
});
```

## Common Patterns and Best Practices

### 1. Route Order Matters

Express matches routes **top to bottom**. Put specific routes before general ones:

```js
// ❌ BAD — /api/users/search will match /:id first, with id = "search"
app.get("/api/users/:id", (req, res) => { /* ... */ });
app.get("/api/users/search", (req, res) => { /* ... */ });

// ✅ GOOD — specific routes before parameterized ones
app.get("/api/users/search", (req, res) => { /* ... */ });
app.get("/api/users/:id", (req, res) => { /* ... */ });
```

### 2. Always Return After Sending a Response

```js
// ❌ BAD — Express will crash with "Cannot set headers after they are sent"
app.get("/api/users/:id", (req, res) => {
  const user = users.find((u) => u.id === parseInt(req.params.id));
  if (!user) {
    res.status(404).json({ error: "Not found" });
    // Code continues executing after this! 😰
  }
  res.json(user); // This will also try to send a response — ERROR!
});

// ✅ GOOD — Use 'return' to stop execution
app.get("/api/users/:id", (req, res) => {
  const user = users.find((u) => u.id === parseInt(req.params.id));
  if (!user) {
    return res.status(404).json({ error: "Not found" }); // return stops here
  }
  res.json(user); // Only runs if user exists
});
```

### 3. Validate Input Data

Never trust data from the client:

```js
app.post("/api/users", (req, res) => {
  const { name, email, age } = req.body;

  // Check required fields
  if (!name || typeof name !== "string") {
    return res.status(400).json({ error: "Valid name is required" });
  }

  if (!email || !email.includes("@")) {
    return res.status(400).json({ error: "Valid email is required" });
  }

  if (age !== undefined && (typeof age !== "number" || age < 0 || age > 150)) {
    return res.status(400).json({ error: "Age must be a number between 0 and 150" });
  }

  // ... proceed with creating the user
});
```

### 4. Use `app.route()` for Clean Grouped Routes

```js
// Instead of writing each method separately:
app.get("/api/books", getAllBooks);
app.post("/api/books", createBook);

app.get("/api/books/:id", getBook);
app.put("/api/books/:id", updateBook);
app.delete("/api/books/:id", deleteBook);

// You can chain them with app.route():
app.route("/api/books")
  .get(getAllBooks)
  .post(createBook);

app.route("/api/books/:id")
  .get(getBook)
  .put(updateBook)
  .delete(deleteBook);
```

### 5. Use Environment Variables for Configuration

```js
// Instead of hardcoding:
const PORT = 3000; // ❌

// Use environment variables:
const PORT = process.env.PORT || 3000; // ✅
```

### 6. Consistent Response Format

Pick a format and stick with it:

```js
// Success response
{
  "success": true,
  "data": { /* ... */ },
  "message": "User created successfully"
}

// Error response
{
  "success": false,
  "error": "User not found",
  "statusCode": 404
}
```

## Summary

### Key Concepts Recap

```
Express Setup:
  npm init -y              → Create package.json
  npm install express      → Install Express
  const app = express()    → Create the application
  app.listen(PORT)         → Start the server

Route Definition:
  app.get(path, handler)   → Handle GET requests
  app.post(path, handler)  → Handle POST requests
  app.put(path, handler)   → Handle PUT requests
  app.delete(path, handler)→ Handle DELETE requests

Request Data Sources:
  req.params               → Dynamic URL segments     → /users/:id → { id: "42" }
  req.query                → URL query strings         → ?sort=name → { sort: "name" }
  req.body                 → Request body (POST/PUT)   → { name: "Alice" }

Body Parsing Middleware:
  app.use(express.json())                    → Parse JSON bodies
  app.use(express.urlencoded({extended:true}))→ Parse form bodies

Response Methods:
  res.send()               → Send any type of response
  res.json()               → Send JSON response
  res.status(code)         → Set HTTP status code
  res.status(201).json()   → Chain status + response
```
