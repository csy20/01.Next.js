## What is Mongoose and Why Do We Need It?

**Mongoose** is an **Object Data Modeling (ODM)** library for MongoDB and Node.js. It provides a structured way to interact with MongoDB by defining schemas, models, validation, and more.

### The Translation Analogy 🌐

Think of Mongoose as a **translator** between your JavaScript code and MongoDB:

```
┌──────────────────────────────────────────────────────────────────┐
│                    MONGOOSE = THE TRANSLATOR                      │
├──────────────────────────────────────────────────────────────────┤
│                                                                   │
│   Your JavaScript Code                                            │
│   (speaks JavaScript)                                             │
│        │                                                          │
│        │  "Create a user with name Alice and email alice@..."     │
│        ▼                                                          │
│   ┌──────────────┐                                                │
│   │   MONGOOSE    │  Translates your JS objects into MongoDB       │
│   │  (Translator) │  documents and vice versa. Also enforces      │
│   │              │  rules, validates data, and provides a          │
│   │              │  clean API to work with.                        │
│   └──────┬───────┘                                                │
│          │                                                        │
│          │  db.users.insertOne({ name: "Alice", email: "..." })  │
│          ▼                                                        │
│   ┌──────────────┐                                                │
│   │   MongoDB     │  Stores raw BSON documents                    │
│   │  (Database)   │  No schema enforcement by default             │
│   └──────────────┘                                                │
│                                                                   │
└──────────────────────────────────────────────────────────────────┘
```

### What Does Mongoose Give Us?

| Feature              | Without Mongoose (raw driver)     | With Mongoose                         |
| -------------------- | --------------------------------- | ------------------------------------- |
| Schema definition    | None — MongoDB accepts anything   | Strict schemas define data structure  |
| Validation           | Manual — write your own checks    | Built-in + custom validators          |
| Type casting         | Manual — numbers stay as strings  | Automatic — "42" becomes 42           |
| Default values       | Manual                            | Defined in schema                     |
| Relationships        | Manual ObjectId handling          | `populate()` for easy joins           |
| Middleware/Hooks      | None                              | Pre/post save, validate, remove, etc. |
| Query building       | Raw MongoDB query objects         | Chainable query API                   |
| Error handling       | Raw MongoDB errors                | Clean, structured error objects       |

> **Bottom line:** MongoDB is flexible but WILD. Mongoose adds **structure, safety, and convenience** on top of it.

## MongoDB Without Mongoose — The Raw Way

Before appreciating Mongoose, let's see what working with MongoDB directly looks like using the native `mongodb` driver:

```js
// WITHOUT Mongoose — using the raw MongoDB driver
const { MongoClient } = require("mongodb");

async function main() {
  const client = new MongoClient("mongodb://localhost:27017");
  await client.connect();

  const db = client.db("myapp");
  const users = db.collection("users");

  // Insert — ANYTHING goes! No validation!
  await users.insertOne({ name: "Alice", email: "alice@example.com", age: 25 });
  await users.insertOne({ name: 42 }); // Name is a number? MongoDB: "Sure!"
  await users.insertOne({ x: true }); // Random fields? MongoDB: "No problem!"
  await users.insertOne({}); // Empty document? MongoDB: "Go ahead!"

  // Find — returns raw cursor, need to convert to array
  const allUsers = await users.find({}).toArray();

  // Update — no validation again
  await users.updateOne({ name: "Alice" }, { $set: { age: "not a number" } });
  // Age is now a string instead of a number? MongoDB doesn't care.

  await client.close();
}
```

### The Problem

MongoDB is **schema-less** by design — it stores whatever you throw at it. This is great for flexibility but terrible for reliability:

```
Without Mongoose, your "users" collection might contain:
──────────────────────────────────────────────────────

Document 1: { name: "Alice", email: "alice@example.com", age: 25 }
Document 2: { name: 42 }                          ← name is a number??
Document 3: { x: true }                           ← no name, no email??
Document 4: { }                                    ← completely empty??
Document 5: { Name: "Bob", EMAIL: "bob@..." }      ← inconsistent casing??

How would your frontend handle this chaos?
It expects { name: String, email: String } every time!
```

Mongoose solves this by letting you define **exactly** what your data should look like.

## Connecting Express to MongoDB via Mongoose

### The Basic Connection

```js
// The simplest possible connection
const mongoose = require("mongoose");

mongoose.connect("mongodb://localhost:27017/myapp");
```

That's it at the most basic level. But in a real application, you want to handle success, failure, and configuration properly.

### Production-Ready Connection Setup

Here's the recommended way to set up the connection in a real project:

```js
// config/db.js — Database connection module

const mongoose = require("mongoose");

const connectDB = async () => {
  try {
    const conn = await mongoose.connect(process.env.DATABASE_URL);

    console.log(`✅ MongoDB Connected: ${conn.connection.host}`);
  } catch (error) {
    console.error(`❌ MongoDB Connection Error: ${error.message}`);
    process.exit(1); // Exit the process with failure code
  }
};

module.exports = connectDB;
```

```js
// server.js — Using the connection module

const dotenv = require("dotenv");
dotenv.config(); // Load .env variables FIRST

const app = require("./app");
const connectDB = require("./config/db");

const PORT = process.env.PORT || 3000;

// Connect to DB, then start server
connectDB().then(() => {
  app.listen(PORT, () => {
    console.log(`🚀 Server running on port ${PORT}`);
  });
});
```

```bash
# .env file
DATABASE_URL=mongodb://localhost:27017/myapp
PORT=3000
```

### What Happens During Connection — Step by Step

```
mongoose.connect("mongodb://localhost:27017/myapp")
         │              │              │       │
         │              │              │       └── Database name
         │              │              └────────── Port (MongoDB default)
         │              └───────────────────────── Host (localhost = your machine)
         └──────────────────────────────────────── Protocol (MongoDB wire protocol)

Step-by-Step:
─────────────────────────────────────────────────────────

1. Mongoose creates a connection pool
   (a set of reusable connections to MongoDB)

2. Mongoose sends a "hello" to the MongoDB server
   "Hey, I want to connect to the 'myapp' database"

3. MongoDB responds:
   "Welcome! Here's your connection. The database 'myapp'
    is ready (I'll create it when you first write to it)"

4. Mongoose caches this connection globally
   All models you define will use THIS connection

5. Your app can now use Models to read/write data

┌─────────────────────────────────────────────┐
│            CONNECTION POOL                   │
│                                              │
│   ┌────────┐  ┌────────┐  ┌────────┐       │
│   │ Conn 1 │  │ Conn 2 │  │ Conn 3 │  ...  │
│   └────┬───┘  └────┬───┘  └────┬───┘       │
│        │           │           │             │
│        └───────────┼───────────┘             │
│                    │                         │
│                    ▼                         │
│              MongoDB Server                  │
│                                              │
│   Multiple connections are maintained so     │
│   your app can handle many requests at once  │
│   without waiting for each query to finish.  │
│                                              │
└─────────────────────────────────────────────┘
```

## Understanding the Connection String

The connection string (URI) tells Mongoose WHERE and HOW to connect to MongoDB.

### Format

```
mongodb://username:password@host:port/database?options
│          │        │        │    │     │         │
│          │        │        │    │     │         └── Query parameters
│          │        │        │    │     └──────────── Database name
│          │        │        │    └────────────────── Port number
│          │        │        └─────────────────────── Hostname or IP
│          │        └──────────────────────────────── Password
│          └───────────────────────────────────────── Username
└──────────────────────────────────────────────────── Protocol
```

### Common Connection Examples

```js
// 1. Local MongoDB (no authentication)
mongoose.connect("mongodb://localhost:27017/myapp");

// 2. Local MongoDB (with authentication)
mongoose.connect("mongodb://admin:secret123@localhost:27017/myapp");

// 3. MongoDB Atlas (cloud) — the most common for deployment
mongoose.connect(
  "mongodb+srv://username:password@cluster0.abc123.mongodb.net/myapp"
);

// 4. Using environment variables (RECOMMENDED for production)
mongoose.connect(process.env.DATABASE_URL);
// Where .env contains:
// DATABASE_URL=mongodb+srv://user:pass@cluster0.abc123.mongodb.net/myapp
```

### MongoDB Atlas vs Local MongoDB

```
┌─────────────────────────────────────────────────────────────┐
│                LOCAL vs ATLAS (CLOUD)                         │
├────────────────────────────┬────────────────────────────────┤
│     LOCAL MongoDB          │     MongoDB Atlas (Cloud)       │
├────────────────────────────┼────────────────────────────────┤
│ Runs on YOUR computer      │ Runs on MongoDB's servers      │
│ URL: localhost:27017       │ URL: cluster0.abc123.net       │
│ Protocol: mongodb://       │ Protocol: mongodb+srv://       │
│ Free, no signup needed     │ Free tier available (512MB)    │
│ Data lost if PC breaks     │ Automatic backups              │
│ Good for development       │ Good for production            │
│ Must install MongoDB       │ No installation needed         │
│ Only accessible from       │ Accessible from anywhere       │
│ your machine               │ (with credentials)             │
└────────────────────────────┴────────────────────────────────┘
```

## Handling Connection Events

Mongoose emits events during the connection lifecycle. You can listen to these for monitoring:

```js
const mongoose = require("mongoose");

// ─── Connection Events ─────────────────────────────────

// Fires when connected successfully
mongoose.connection.on("connected", () => {
  console.log("✅ Mongoose connected to MongoDB");
});

// Fires on connection error
mongoose.connection.on("error", (err) => {
  console.error("❌ Mongoose connection error:", err.message);
});

// Fires when disconnected
mongoose.connection.on("disconnected", () => {
  console.log("⚠️  Mongoose disconnected from MongoDB");
});

// Fires when reconnecting
mongoose.connection.on("reconnected", () => {
  console.log("🔄 Mongoose reconnected to MongoDB");
});

// ─── Graceful Shutdown ──────────────────────────────────
// When you stop your app (Ctrl+C), close the connection properly

process.on("SIGINT", async () => {
  await mongoose.connection.close();
  console.log("🛑 MongoDB connection closed due to app termination");
  process.exit(0);
});
```

### Connection Lifecycle

```
App starts
    │
    ▼
mongoose.connect()
    │
    ├── Success → "connected" event fires
    │                │
    │                ▼
    │          App is running, queries work
    │                │
    │                ├── Network issue? → "disconnected" event
    │                │                      │
    │                │                      ▼
    │                │                Mongoose auto-reconnects
    │                │                      │
    │                │                      ▼
    │                │              "reconnected" event
    │                │
    │                ├── Ctrl+C pressed? → SIGINT handler
    │                │                      │
    │                │                      ▼
    │                │              connection.close()
    │                │                      │
    │                │                      ▼
    │                │              "disconnected" event
    │                │                      │
    │                │                      ▼
    │                │              process.exit(0)
    │
    └── Failure → "error" event fires
                     │
                     ▼
               process.exit(1)
```

## Connection Best Practices

```js
// config/db.js — Production-ready connection

const mongoose = require("mongoose");

const connectDB = async () => {
  try {
    const conn = await mongoose.connect(process.env.DATABASE_URL, {
      // Mongoose 8+ has sensible defaults, but you can customize:
      // maxPoolSize: 10,       // Maximum number of connections in the pool
      // serverSelectionTimeoutMS: 5000,  // Timeout for finding a server
      // socketTimeoutMS: 45000,          // Timeout for socket inactivity
    });

    console.log(`✅ MongoDB Connected: ${conn.connection.host}`);
    console.log(`📦 Database: ${conn.connection.name}`);
  } catch (error) {
    console.error(`❌ Error: ${error.message}`);
    process.exit(1);
  }
};

module.exports = connectDB;
```

```
Connection Best Practices:
──────────────────────────────────────────────────────

1. ALWAYS use environment variables for the connection string
   → Never hardcode credentials in your source code

2. Connect BEFORE starting the server
   → Ensures the database is ready before accepting requests

3. Handle connection errors gracefully
   → Don't let the app run without a database

4. Use a separate config/db.js file
   → Keeps connection logic separate from app logic

5. Enable graceful shutdown
   → Close connections when the app stops (SIGINT handler)
```

## What is a Schema?

A **Schema** is a **blueprint** that defines the **shape** of documents within a MongoDB collection. It specifies:

- What **fields** (properties) a document can have
- What **type** each field is (String, Number, Date, etc.)
- What **rules** (validations) each field must follow
- What **defaults** to use when a value isn't provided

### The Blueprint Analogy 📐

```
┌──────────────────────────────────────────────────────────────┐
│                  SCHEMA = ARCHITECTURAL BLUEPRINT             │
├──────────────────────────────────────────────────────────────┤
│                                                               │
│   Building a house without a blueprint:                       │
│   • The kitchen might end up on the roof                     │
│   • The door might be 2 feet tall                            │
│   • Some rooms might not have floors                         │
│   • Total chaos!                                              │
│                                                               │
│   Building a house WITH a blueprint:                          │
│   • Every room has a defined purpose and size                │
│   • Doors are 7 feet tall                                    │
│   • Everything follows the plan                              │
│   • Predictable and reliable!                                 │
│                                                               │
│   Schema = Blueprint for your MongoDB documents               │
│   • Every field has a defined type and purpose               │
│   • Data follows the rules you set                           │
│   • Predictable and reliable!                                 │
│                                                               │
└──────────────────────────────────────────────────────────────┘
```

## Why Schemas Matter — The Chaos Without Them

```js
// WITHOUT a schema — MongoDB accepts ANYTHING

// These ALL go into the same "users" collection:
db.users.insertOne({ name: "Alice", email: "alice@gmail.com" });
db.users.insertOne({ username: "bob", mail: "bob@yahoo.com" });  // Wrong field names!
db.users.insertOne({ name: 12345 });                              // Name is a number!
db.users.insertOne({ name: "", email: "" });                      // Empty strings!
db.users.insertOne({});                                            // Nothing at all!
db.users.insertOne({ name: "Eve", age: -5, email: "not-an-email" }); // Negative age!
```

```js
// WITH a Mongoose schema — ONLY valid data gets through

const userSchema = new mongoose.Schema({
  name:  { type: String, required: true, minlength: 2 },
  email: { type: String, required: true, match: /^\S+@\S+\.\S+$/ },
  age:   { type: Number, min: 0, max: 150 },
});

// Now:
await User.create({ name: "Alice", email: "alice@gmail.com" }); // ✅ Works
await User.create({ username: "bob" });     // ❌ Error: name is required
await User.create({ name: 12345 });         // ✅ Auto-cast to string "12345"
await User.create({ name: "", email: "" }); // ❌ Error: validation failed
await User.create({});                       // ❌ Error: name and email required
await User.create({ name: "Eve", age: -5 }); // ❌ Error: age below minimum
```

## Defining Your First Schema

### Basic Syntax

```js
const mongoose = require("mongoose");

// Method 1: Shorthand (just specify types)
const userSchema = new mongoose.Schema({
  name: String,
  email: String,
  age: Number,
  isActive: Boolean,
  joinedAt: Date,
});

// Method 2: Detailed (specify types + options)
// This is what you'll use in real projects
const userSchema = new mongoose.Schema({
  name: {
    type: String,
    required: true,
  },
  email: {
    type: String,
    required: true,
    unique: true,
  },
  age: {
    type: Number,
    min: 0,
  },
  isActive: {
    type: Boolean,
    default: true,
  },
  joinedAt: {
    type: Date,
    default: Date.now,
  },
});
```

### Anatomy of a Schema Field

```
fieldName: {
  type: String,              ← What TYPE of data (required)
  required: true,            ← Must this field exist?
  unique: true,              ← Must values be unique across all documents?
  default: "some value",     ← Default value if not provided
  trim: true,                ← Remove whitespace from start/end (String only)
  lowercase: true,           ← Convert to lowercase (String only)
  uppercase: true,           ← Convert to uppercase (String only)
  minlength: 2,              ← Minimum length (String only)
  maxlength: 100,            ← Maximum length (String only)
  min: 0,                    ← Minimum value (Number/Date only)
  max: 150,                  ← Maximum value (Number/Date only)
  enum: ["a", "b", "c"],     ← Must be one of these values
  match: /regex/,            ← Must match this regex pattern (String only)
  validate: Function,        ← Custom validation function
  index: true,               ← Create a database index on this field
  select: false,             ← Exclude from query results by default
  immutable: true,           ← Cannot be changed after creation
}
```

## Mongoose Data Types — The Complete Reference

Mongoose supports the following data types, each mapping to a MongoDB BSON type:

```
┌──────────────────┬───────────────────┬──────────────────────────────┐
│  Mongoose Type    │  JavaScript Type  │  Example Values              │
├──────────────────┼───────────────────┼──────────────────────────────┤
│  String           │  String           │  "hello", "Alice"            │
│  Number           │  Number           │  42, 3.14, -10               │
│  Boolean          │  Boolean          │  true, false                 │
│  Date             │  Date             │  2024-01-15T10:30:00Z        │
│  ObjectId         │  ObjectId         │  507f1f77bcf86cd799439011    │
│  Array            │  Array            │  [1, 2, 3], ["a", "b"]      │
│  Buffer           │  Buffer           │  Binary data (images, files) │
│  Mixed            │  Any              │  Anything — no type checking │
│  Map              │  Map              │  Key-value pairs             │
│  Decimal128       │  Decimal128       │  High-precision decimals     │
│  BigInt           │  BigInt           │  Very large integers         │
└──────────────────┴───────────────────┴──────────────────────────────┘
```

Let's explore each type in detail:

---

### String

The most common data type. Used for names, emails, descriptions, etc.

```js
const productSchema = new mongoose.Schema({
  // Basic string
  name: String,

  // String with all available options
  description: {
    type: String,
    required: [true, "A product must have a description"],
    trim: true, // "  hello  " → "hello"
    minlength: [10, "Description must be at least 10 characters"],
    maxlength: [500, "Description cannot exceed 500 characters"],
    lowercase: true, // "HELLO" → "hello"
    // uppercase: true,    // "hello" → "HELLO"  (can't use both)
  },

  // String with enum — must be one of these values
  category: {
    type: String,
    enum: {
      values: ["electronics", "clothing", "books", "food"],
      message: "{VALUE} is not a valid category",
    },
    default: "electronics",
  },

  // String with regex pattern matching
  sku: {
    type: String,
    match: [/^[A-Z]{3}-\d{4}$/, "SKU must follow format: ABC-1234"],
    // Valid:   "ABC-1234"
    // Invalid: "abc-1234", "AB-123", "ABCD-12345"
  },
});
```

### Type Casting for Strings

Mongoose automatically converts non-string values to strings:

```js
// Mongoose auto-casts these to strings:
await Product.create({ name: 42 }); // name becomes "42"
await Product.create({ name: true }); // name becomes "true"
await Product.create({ name: { toString: () => "hello" } }); // calls toString()
```

---

### Number

Used for ages, prices, quantities, ratings, etc.

```js
const productSchema = new mongoose.Schema({
  // Basic number
  quantity: Number,

  // Number with options
  price: {
    type: Number,
    required: [true, "A product must have a price"],
    min: [0, "Price cannot be negative"],
    max: [1000000, "Price cannot exceed 1,000,000"],
  },

  // Number with custom validation
  rating: {
    type: Number,
    min: [1, "Rating must be at least 1"],
    max: [5, "Rating cannot be more than 5"],
    set: (val) => Math.round(val * 10) / 10, // Round to 1 decimal
    // 4.666 → 4.7
  },

  // Integer-only (using custom validation)
  stock: {
    type: Number,
    validate: {
      validator: Number.isInteger,
      message: "Stock must be a whole number",
    },
    min: 0,
  },
});
```

### Type Casting for Numbers

```js
// Mongoose auto-casts:
await Product.create({ price: "29.99" }); // string "29.99" → number 29.99
await Product.create({ price: true }); //   boolean true → number 1
await Product.create({ price: false }); //  boolean false → number 0

// But these will FAIL:
await Product.create({ price: "not a number" }); // ❌ CastError
await Product.create({ price: {} }); //              ❌ CastError
```

---

### Boolean

Used for flags, toggles, active/inactive states.

```js
const userSchema = new mongoose.Schema({
  isActive: {
    type: Boolean,
    default: true,
  },
  isVerified: {
    type: Boolean,
    default: false,
  },
  acceptedTerms: {
    type: Boolean,
    required: [true, "You must accept the terms and conditions"],
  },
});
```

### Type Casting for Booleans

```js
// Mongoose is smart about casting to Boolean:
// These become TRUE:
true, "true", 1, "1", "yes"

// These become FALSE:
false, "false", 0, "0", "no"
```

---

### Date

Used for timestamps, deadlines, birthdates, etc.

```js
const eventSchema = new mongoose.Schema({
  // Basic date
  createdAt: Date,

  // Date with default (current time when document is created)
  registeredAt: {
    type: Date,
    default: Date.now, // Note: Date.now (without parentheses) = function reference
    // Date.now() would set the time when the SCHEMA is defined, not the document
  },

  // Date with min/max
  eventDate: {
    type: Date,
    required: true,
    min: ["2024-01-01", "Event date must be after 2024"],
    validate: {
      validator: function (date) {
        return date > Date.now(); // Must be in the future
      },
      message: "Event date must be in the future",
    },
  },

  // Immutable date — can never be changed after creation
  accountCreated: {
    type: Date,
    default: Date.now,
    immutable: true, // Cannot be modified after the document is created
  },
});
```

### `Date.now` vs `Date.now()`

```
CRITICAL DIFFERENCE:
──────────────────────────────────────────────

  default: Date.now      ← CORRECT
  • This is a FUNCTION REFERENCE
  • Mongoose calls it fresh each time a new document is created
  • Each document gets the CURRENT timestamp

  default: Date.now()    ← WRONG (usually)
  • This IMMEDIATELY calls the function
  • Returns the timestamp when the schema file is LOADED
  • ALL documents would get the SAME timestamp
  • Only the timestamp from when your server started!
```

### ObjectId (References)

Used to create **relationships** between documents. Think of it as a **foreign key** in relational databases.

```js
const commentSchema = new mongoose.Schema({
  text: {
    type: String,
    required: true,
  },

  // Reference to the User who wrote this comment
  author: {
    type: mongoose.Schema.Types.ObjectId,
    ref: "User", // Name of the model this ID refers to
    required: true,
  },

  // Reference to the Post this comment belongs to
  post: {
    type: mongoose.Schema.Types.ObjectId,
    ref: "Post",
    required: true,
  },
});

// When querying, you can "populate" these references
// to get the full user/post objects instead of just IDs:
const comment = await Comment.findById(commentId)
  .populate("author") // Replaces author ID with full User document
  .populate("post"); // Replaces post ID with full Post document
```

```
How ObjectId References Work:
──────────────────────────────────────────────

  comments collection:                    users collection:
  ┌──────────────────────────┐          ┌──────────────────────┐
  │ _id: "c001"              │          │ _id: "u001"          │
  │ text: "Great post!"     │ ──ref──► │ name: "Alice"        │
  │ author: "u001"           │          │ email: "alice@..."   │
  └──────────────────────────┘          └──────────────────────┘

  Before populate():
  { text: "Great post!", author: "u001" }   ← Just an ID string

  After populate("author"):
  {
    text: "Great post!",
    author: { _id: "u001", name: "Alice", email: "alice@..." }
  }                                          ← Full user object!
```

### Mixed (Schema.Types.Mixed)

Used when you want a field to accept **any** data type. Use sparingly — it defeats the purpose of schemas!

```js
const logSchema = new mongoose.Schema({
  level: String,
  message: String,

  // Mixed type — can be anything
  metadata: {
    type: mongoose.Schema.Types.Mixed,
  },

  // metadata: {}
  // Also shorthand:
  // metadata: mongoose.Schema.Types.Mixed

await Log.create({ level: "warn", message: "Slow", metadata: 42 });
```

> **⚠️ Warning:** Mongoose **cannot** detect changes to Mixed fields. You must call `doc.markModified('fieldName')` before `doc.save()` to save changes to Mixed fields.

```js
const log = await Log.findById(id);
log.metadata.newField = "added";
log.markModified("metadata"); // REQUIRED for Mixed types!
```

## Default Values — Pre-Filling Your Data

Defaults let you specify what value a field should have when the user doesn't provide one.

### Basic Defaults

```js
const userSchema = new mongoose.Schema({
  name: { type: String, required: true },
  email: { type: String, required: true },

  // Static defaults — fixed values
  role: { type: String, default: "user" },
  isActive: { type: Boolean, default: true },
  loginCount: { type: Number, default: 0 },
  bio: { type: String, default: "No bio provided" },

  // Dynamic defaults — function that runs each time
  joinedAt: { type: Date, default: Date.now },

  // Default array
  permissions: { type: [String], default: ["read"] },
});
```

### Static vs Dynamic Defaults

```js
// STATIC default — same value every time
role: { type: String, default: "user" }
// Every new user gets role = "user"

// DYNAMIC default — function called each time
joinedAt: { type: Date, default: Date.now }
// Each user gets the current timestamp when THEY are created

// DYNAMIC default with custom function
referralCode: {
  type: String,
  default: function() {
    // Generate a random 8-character code
    return Math.random().toString(36).substring(2, 10).toUpperCase();
  }
}
// Each user gets a UNIQUE random code
```

```
How Defaults Work:
──────────────────────────────────────────────

  await User.create({ name: "Alice", email: "alice@example.com" });

  What gets saved to MongoDB:
  {
    _id: ObjectId("..."),              ← Auto-generated by MongoDB
    name: "Alice",                      ← Provided by user
    email: "alice@example.com",         ← Provided by user
    role: "user",                       ← DEFAULT applied
    isActive: true,                     ← DEFAULT applied
    loginCount: 0,                      ← DEFAULT applied
    bio: "No bio provided",            ← DEFAULT applied
    joinedAt: 2024-01-15T10:30:00Z,    ← DEFAULT function called
    permissions: ["read"],              ← DEFAULT applied
    referralCode: "A3F8KM2N",          ← DEFAULT function called
    __v: 0                              ← Version key (Mongoose adds this)
  }
```

### Default with `this` Context

```js
const orderSchema = new mongoose.Schema({
  items: [{ name: String, price: Number, quantity: Number }],

  // Default that depends on other fields
  totalItems: {
    type: Number,
    default: function () {
      // 'this' refers to the document being created
      return this.items ? this.items.length : 0;
    },
  },
});
```

> **Note:** Arrow functions (`() =>`) do NOT have their own `this`. If you need to access the document in a default function, use a regular function (`function()`).

## Validations — Enforcing Rules on Your Data

Validation is the **security guard** at the door of your database. It checks every piece of data before it's allowed in.

```
┌──────────────────────────────────────────────────────────────┐
│                VALIDATION = SECURITY GUARD                    │
├──────────────────────────────────────────────────────────────┤
│                                                               │
│   Incoming Data                                               │
│   { name: "", email: "not-an-email", age: -5 }              │
│        │                                                      │
│        ▼                                                      │
│   ┌──────────────────┐                                        │
│   │  SCHEMA VALIDATOR │                                       │
│   │  (Security Guard) │                                       │
│   │                    │                                       │
│   │  ✓ name required  │  → ❌ FAIL: name is empty            │
│   │  ✓ email format   │  → ❌ FAIL: not a valid email        │
│   │  ✓ age >= 0       │  → ❌ FAIL: -5 is negative           │
│   │                    │                                       │
│   │  REJECTED! ✋      │                                       │
│   │  ValidationError   │                                       │
│   └──────────────────┘                                        │
│                                                               │
│   None of the invalid data reaches the database!              │
│                                                               │
└──────────────────────────────────────────────────────────────┘
```

### Built-in Validators

Mongoose provides validators out of the box for common checks:

#### Validators Available for ALL Types

```js
const schema = new mongoose.Schema({
  // required — field MUST be present and not empty
  name: {
    type: String,
    required: true,
    // Or with custom error message:
    // required: [true, "Name is required"]
  },

  // enum — value must be one of the specified options
  status: {
    type: String,
    enum: ["active", "inactive", "suspended"],
    // Or with custom error message:
    // enum: {
    //   values: ["active", "inactive", "suspended"],
    //   message: "{VALUE} is not a valid status"
    // }
  },
});
```

#### String-Specific Validators

```js
const userSchema = new mongoose.Schema({
  username: {
    type: String,
    required: [true, "Username is required"],
    minlength: [3, "Username must be at least 3 characters"],
    maxlength: [30, "Username cannot exceed 30 characters"],
    match: [/^[a-zA-Z0-9_]+$/, "Username can only contain letters, numbers, and underscores"],
    trim: true,       // Remove leading/trailing whitespace
    lowercase: true,  // Convert to lowercase before saving
  },

  email: {
    type: String,
    required: [true, "Email is required"],
    unique: true,     // Note: 'unique' is NOT a validator, it's a MongoDB index
    match: [
      /^\w+([.-]?\w+)*@\w+([.-]?\w+)*(\.\w{2,3})+$/,
      "Please provide a valid email address",
    ],
    lowercase: true,
  },
});
```

> **Important:** `unique: true` is **NOT** a Mongoose validator. It creates a MongoDB **unique index**. If a duplicate value is inserted, MongoDB (not Mongoose) throws an error with code `11000`. You handle this in your error-handling middleware, not in validation.

#### Number-Specific Validators

```js
const productSchema = new mongoose.Schema({
  price: {
    type: Number,
    required: [true, "Price is required"],
    min: [0, "Price cannot be negative"],
    max: [999999.99, "Price cannot exceed $999,999.99"],
  },

  rating: {
    type: Number,
    min: [1, "Rating must be at least 1.0"],
    max: [5, "Rating cannot exceed 5.0"],
    default: 3,
  },

  quantity: {
    type: Number,
    min: [0, "Quantity cannot be negative"],
    validate: {
      validator: Number.isInteger,
      message: "Quantity must be a whole number",
    },
  },
});
```

#### Summary of Built-in Validators

```
┌──────────────────┬──────────────────────────────────────────┐
│   Validator       │   Works On                               │
├──────────────────┼──────────────────────────────────────────┤
│   required        │   All types                              │
│   enum            │   All types (most useful for String)     │
│   min             │   Number, Date                           │
│   max             │   Number, Date                           │
│   minlength       │   String                                 │
│   maxlength       │   String                                 │
│   match           │   String (regex pattern)                 │
│   trim            │   String (not a validator, a setter)     │
│   lowercase       │   String (not a validator, a setter)     │
│   uppercase       │   String (not a validator, a setter)     │
└──────────────────┴──────────────────────────────────────────┘
```

### Custom Validators

When built-in validators aren't enough, you can write your own:

```js
const userSchema = new mongoose.Schema({
  // Custom validator with a function
  email: {
    type: String,
    required: true,
    validate: {
      validator: function (value) {
        // Return true if valid, false if invalid
        return /^\S+@\S+\.\S+$/.test(value);
      },
      message: "Please provide a valid email address",
    },
  },

  // Custom validator — password strength
  password: {
    type: String,
    required: [true, "Password is required"],
    minlength: [8, "Password must be at least 8 characters"],
    validate: {
      validator: function (value) {
        // Must contain at least one uppercase, one lowercase, one number
        return /^(?=.*[a-z])(?=.*[A-Z])(?=.*\d).+$/.test(value);
      },
      message:
        "Password must contain at least one uppercase letter, one lowercase letter, and one number",
    },
  },

  // Custom validator using 'this' to access other fields
  passwordConfirm: {
    type: String,
    required: [true, "Please confirm your password"],
    validate: {
      validator: function (value) {
        // 'this' refers to the current document
        // Check this field matches the 'password' field
        return value === this.password;
      },
      message: "Passwords do not match",
    },
    // NOTE: This only works on .create() and .save()
    // NOT on .findOneAndUpdate()!
  },

  // Custom validator — phone number format
  phone: {
    type: String,
    validate: {
      validator: function (v) {
        return /^\+?[\d\s-()]{10,}$/.test(v);
      },
      message: (props) => `${props.value} is not a valid phone number`,
    },
  },

  // Async custom validator (e.g., check external API)
  username: {
    type: String,
    validate: {
      validator: async function (value) {
        // Check if username contains banned words (could be an API call)
        const bannedWords = ["admin", "root", "superuser"];
        return !bannedWords.includes(value.toLowerCase());
      },
      message: "This username is not allowed",
    },
  },
});
```

### Conditional Validation

Sometimes a field should only be required based on another field's value:

```js
const shipmentSchema = new mongoose.Schema({
  type: {
    type: String,
    enum: ["standard", "express", "pickup"],
    required: true,
  },

  // Address is required ONLY if type is NOT "pickup"
  address: {
    type: String,
    required: function () {
      return this.type !== "pickup";
    },
  },

  // Pickup location required ONLY if type IS "pickup"
  pickupLocation: {
    type: String,
    required: function () {
      return this.type === "pickup";
    },
  },

  // Express tracking number required only for express shipments
  trackingNumber: {
    type: String,
    required: function () {
      return this.type === "express";
    },
  },
});
```

### When Do Validators Run?

```
Validators run during:
──────────────────────────────────────────────

  ✅ Model.create()            → All validators run
  ✅ document.save()           → All validators run
  ✅ Model.findOneAndUpdate()  → ONLY if { runValidators: true }
  ✅ Model.updateOne()         → ONLY if { runValidators: true }

  ❌ Model.findOneAndUpdate()  → Validators DO NOT run by default!
  ❌ Model.updateMany()        → Validators DO NOT run by default!

  // To enable validators on update operations:
  User.findByIdAndUpdate(id, data, {
    new: true,
    runValidators: true    // ← ADD THIS!
  });
```

> **This is a common gotcha!** By default, update operations SKIP validation. Always add `runValidators: true` to your update options.

## What is a Model?

A **Model** is a **compiled version** of a Schema. If the Schema is the blueprint, the Model is the **factory** that builds houses from that blueprint.

```
┌──────────────────────────────────────────────────────────────┐
│               SCHEMA vs MODEL                                 │
├──────────────────────────────────────────────────────────────┤
│                                                               │
│   SCHEMA (Blueprint)           MODEL (Factory)                │
│   ──────────────────          ─────────────────               │
│   • Defines the structure     • Creates documents             │
│   • Sets validation rules     • Queries the database          │
│   • Configures options        • Updates documents             │
│   • Just a definition         • Deletes documents             │
│   • Cannot interact with DB   • CAN interact with DB          │
│                                                               │
│   const userSchema = ...      const User = mongoose.model(    │
│                                 "User", userSchema            │
│                               );                              │
│                                                               │
│   Schema CANNOT do this:      Model CAN do this:              │
│   ✗ userSchema.find()         ✓ User.find()                  │
│   ✗ userSchema.create()       ✓ User.create()                │
│   ✗ userSchema.deleteOne()    ✓ User.deleteOne()             │
│                                                               │
└──────────────────────────────────────────────────────────────┘
```

---

## Schema vs Model — The Difference

```js
const mongoose = require("mongoose");

// ──── STEP 1: Define the Schema (the blueprint) ────
const userSchema = new mongoose.Schema({
  name: { type: String, required: true },
  email: { type: String, required: true, unique: true },
  age: { type: Number, min: 0 },
});

// At this point, you CANNOT do anything with the database.
// userSchema.find() ← This doesn't exist!

// ──── STEP 2: Compile the Model (the factory) ──────
const User = mongoose.model("User", userSchema);
//                           │         │
//                           │         └── The schema to use
//                           └──────────── The model name (SINGULAR)
//                                         Mongoose creates a "users"
//                                         collection (lowercase, plural)

// NOW you can interact with the database:
const alice = await User.create({ name: "Alice", email: "alice@example.com" });
const users = await User.find({});
```

### How Mongoose Names Collections

```
Model Name → Collection Name (automatic conversion)
──────────────────────────────────────────────

mongoose.model("User", schema)      → "users" collection
mongoose.model("Product", schema)   → "products" collection
mongoose.model("Category", schema)  → "categories" collection
mongoose.model("Person", schema)    → "people" collection  (smart pluralization!)
mongoose.model("Mouse", schema)     → "mice" collection    (even handles irregulars!)

// If you want to override the automatic naming:
const userSchema = new mongoose.Schema({ ... }, {
  collection: "my_custom_users"    // Force a specific collection name
});
```

## Compiling a Model from a Schema

### The Standard Pattern

```js
// models/User.js

const mongoose = require("mongoose");

// 1. Define the schema
const userSchema = new mongoose.Schema({
  name: {
    type: String,
    required: [true, "Name is required"],
    trim: true,
  },
  email: {
    type: String,
    required: [true, "Email is required"],
    unique: true,
    lowercase: true,
  },
  role: {
    type: String,
    enum: ["user", "admin"],
    default: "user",
  },
  createdAt: {
    type: Date,
    default: Date.now,
  },
});

// 2. Compile the model and export it
module.exports = mongoose.model("User", userSchema);
```

```js
// controllers/userController.js — Using the Model

const User = require("../models/User");

// Now User can do everything:
const allUsers = await User.find(); // Query all documents
const oneUser = await User.findById(id); // Find by ID
const newUser = await User.create(data); // Create new document
await User.findByIdAndUpdate(id, data); // Update
await User.findByIdAndDelete(id); // Delete
```

### What Happens When You Call `mongoose.model()`

```
mongoose.model("User", userSchema)
         │
         ├── 1. Compiles the schema into a constructor function
         │      (adds query methods, validation logic, etc.)
         │
         ├── 2. Links the model to the "users" collection in MongoDB
         │      (auto-pluralizes "User" → "users")
         │
         ├── 3. Registers the model globally in Mongoose
         │      (so you can reference it with "User" in populate())
         │
         └── 4. Returns the Model class
                (the thing you use to find, create, update, delete)
```

## Schema Options

You can pass a second argument to `new mongoose.Schema()` for configuration:

```js
const userSchema = new mongoose.Schema(
  {
    name: String,
    email: String,
  },
  {
    // ─── Commonly Used Options ─────────────────────────

    timestamps: true,
    // Automatically adds 'createdAt' and 'updatedAt' fields
    // createdAt: set once when document is created
    // updatedAt: updated every time the document is modified

    collection: "custom_users",
    // Override the automatic collection name
    // Default: "users" (lowercase, pluralized model name)

    versionKey: "__v",
    // The field Mongoose uses for document versioning
    // Set to false to disable: versionKey: false

    strict: true,
    // DEFAULT: true — only fields defined in schema are saved
    // If you try to save a field NOT in the schema, it's silently ignored
    // Set to false to allow any fields (defeats purpose of schema!)

    toJSON: { virtuals: true },
    // Include virtual properties when converting to JSON
    // (res.json(user) triggers toJSON)

    toObject: { virtuals: true },
    // Include virtual properties when converting to Object
    // (console.log(user) triggers toObject)

    id: true,
    // Creates a virtual 'id' property (string version of _id)
    // user.id === user._id.toString()
  }
);
```

### The `strict` Option Explained

```js
// strict: true (DEFAULT) — Only schema-defined fields are saved

const userSchema = new mongoose.Schema(
  { name: String, email: String },
  { strict: true }
);

await User.create({
  name: "Alice",
  email: "alice@example.com",
  hackerField: "malicious data", // ← This field is NOT in the schema
});

// What gets saved to MongoDB:
// { name: "Alice", email: "alice@example.com" }
// 'hackerField' was SILENTLY IGNORED! ✅ Safe!

// strict: false — ANYTHING gets saved (dangerous!)
// { name: "Alice", email: "alice@example.com", hackerField: "malicious data" }
```

> **Always keep `strict: true`** (the default). It prevents unexpected fields from being injected into your database.

## Instance Methods vs Static Methods

### Instance Methods — Methods on Individual Documents

Instance methods are functions you can call on a **specific document** (an instance of the model):

```js
const userSchema = new mongoose.Schema({
  name: String,
  email: String,
  password: String, // Assume hashed
  loginAttempts: { type: Number, default: 0 },
  isLocked: { type: Boolean, default: false },
});

// ──── Define Instance Methods ────────────────────────

// Check if the provided password matches the stored password
userSchema.methods.correctPassword = async function (candidatePassword) {
  // 'this' refers to the CURRENT DOCUMENT
  const bcrypt = require("bcryptjs");
  return await bcrypt.compare(candidatePassword, this.password);
};

// Increment login attempts
userSchema.methods.incrementLoginAttempts = async function () {
  this.loginAttempts += 1;
  if (this.loginAttempts >= 5) {
    this.isLocked = true;
  }
  return this.save();
};

// Get a greeting (simple example)
userSchema.methods.getGreeting = function () {
  return `Hello, my name is ${this.name}!`;
};

const User = mongoose.model("User", userSchema);

// ──── Using Instance Methods ──────────────────────────

const user = await User.findById(userId);

// Call methods on the specific document:
const isCorrect = await user.correctPassword("myPassword123");
const greeting = user.getGreeting(); // "Hello, my name is Alice!"
await user.incrementLoginAttempts();
```

### Static Methods — Methods on the Model Itself

Static methods are called on the **Model** (not on individual documents):

```js
const userSchema = new mongoose.Schema({
  name: String,
  email: String,
  role: { type: String, enum: ["user", "admin"], default: "user" },
  isActive: { type: Boolean, default: true },
});

// ──── Define Static Methods ──────────────────────────

// Find all active users
userSchema.statics.findActive = function () {
  // 'this' refers to the MODEL (User)
  return this.find({ isActive: true });
};

// Find all admins
userSchema.statics.findAdmins = function () {
  return this.find({ role: "admin" });
};

// Search users by name (case-insensitive)
userSchema.statics.searchByName = function (name) {
  return this.find({ name: new RegExp(name, "i") });
};

const User = mongoose.model("User", userSchema);

// ──── Using Static Methods ────────────────────────────

// Call methods on the Model itself (not on a document):
const activeUsers = await User.findActive();
const admins = await User.findAdmins();
const results = await User.searchByName("alice");
```

### Instance vs Static — When to Use Which

```
┌──────────────────────────┬───────────────────────────────┐
│    Instance Methods       │    Static Methods             │
├──────────────────────────┼───────────────────────────────┤
│ Called on a DOCUMENT     │ Called on the MODEL            │
│ user.checkPassword()     │ User.findActive()             │
│                          │                               │
│ 'this' = the document    │ 'this' = the Model (User)     │
│                          │                               │
│ Use when you need to     │ Use when you need to          │
│ work with ONE specific   │ query/aggregate MANY          │
│ document's data          │ documents                     │
│                          │                               │
│ Examples:                │ Examples:                     │
│ • Check password         │ • Find all active users       │
│ • Generate auth token    │ • Search by name              │
│ • Calculate something    │ • Get statistics              │
│   for this document      │ • Custom query helpers        │
└──────────────────────────┴───────────────────────────────┘
```

## Virtual Properties

Virtuals are fields that are **computed on-the-fly** and NOT stored in the database. They're derived from other fields.

```js
const userSchema = new mongoose.Schema(
  {
    firstName: { type: String, required: true },
    lastName: { type: String, required: true },
    birthDate: Date,
  },
  {
    toJSON: { virtuals: true }, // Include virtuals in JSON output
    toObject: { virtuals: true }, // Include virtuals in object output
  }
);

// ──── Virtual: Full Name ────────────────────────────

// Getter — computed when you ACCESS the property
userSchema.virtual("fullName").get(function () {
  return `${this.firstName} ${this.lastName}`;
});

// Setter — splits a full name into first/last when you SET it
userSchema.virtual("fullName").set(function (name) {
  const [first, ...rest] = name.split(" ");
  this.firstName = first;
  this.lastName = rest.join(" ");
});

// ──── Virtual: Age (calculated from birthDate) ──────

userSchema.virtual("age").get(function () {
  if (!this.birthDate) return null;
  const ageDiff = Date.now() - this.birthDate.getTime();
  return Math.floor(ageDiff / (1000 * 60 * 60 * 24 * 365.25));
});

const User = mongoose.model("User", userSchema);

// ──── Using Virtuals ──────────────────────────────────

const user = await User.create({
  firstName: "Alice",
  lastName: "Johnson",
  birthDate: new Date("1995-06-15"),
});

console.log(user.fullName); // "Alice Johnson" (computed, not stored)
console.log(user.age); // 30 (calculated from birthDate)

// Using the setter:
user.fullName = "Bob Smith";
console.log(user.firstName); // "Bob"
console.log(user.lastName); // "Smith"
```

```
How Virtuals Work:
──────────────────────────────────────────────

  What's stored in MongoDB:
  {
    firstName: "Alice",
    lastName: "Johnson",
    birthDate: 1995-06-15T00:00:00Z
  }

  What your app sees (with virtuals):
  {
    firstName: "Alice",
    lastName: "Johnson",
    birthDate: 1995-06-15T00:00:00Z,
    fullName: "Alice Johnson",     ← Virtual (computed)
    age: 30                         ← Virtual (computed)
  }

  Virtuals save storage space and always stay in sync!
  If firstName changes, fullName automatically updates.
```

> **Important:** Virtuals are NOT stored in the database, so you **cannot** query by them. `User.find({ fullName: "Alice Johnson" })` will NOT work.

## Middleware (Hooks) — Pre and Post

Middleware (also called **hooks**) are functions that run **before** or **after** certain operations. They're like interceptors.

### Pre Middleware — Runs BEFORE the Operation

```js
const userSchema = new mongoose.Schema({
  name: String,
  email: String,
  password: String,
  slug: String,
  passwordChangedAt: Date,
});

// ──── Pre 'save' — runs before document.save() ──────

// Hash the password before saving
userSchema.pre("save", async function (next) {
  // Only run if password was actually modified
  if (!this.isModified("password")) return next();

  const bcrypt = require("bcryptjs");
  this.password = await bcrypt.hash(this.password, 12);

  // Don't persist passwordConfirm to the database
  this.passwordConfirm = undefined;

  next(); // MUST call next() to continue to the next middleware
});

// Create a URL-friendly slug from the name
userSchema.pre("save", function (next) {
  if (!this.isModified("name")) return next();
  this.slug = this.name.toLowerCase().replace(/\s+/g, "-");
  next();
});

// Set passwordChangedAt when password is modified
userSchema.pre("save", function (next) {
  if (!this.isModified("password") || this.isNew) return next();
  this.passwordChangedAt = Date.now() - 1000; // Subtract 1 second for JWT timing
  next();
});

// ──── Pre 'find' — runs before any find query ────────

// Automatically exclude inactive users from all queries
userSchema.pre(/^find/, function (next) {
  // /^find/ matches find, findOne, findById, findOneAndUpdate, etc.
  // 'this' refers to the QUERY object, not a document
  this.find({ isActive: { $ne: false } });
  next();
});
```

### Post Middleware — Runs AFTER the Operation

```js
// ──── Post 'save' — runs after document.save() ──────

userSchema.post("save", function (doc, next) {
  console.log(`📝 New user saved: ${doc.name} (${doc.email})`);
  next();
});

// ──── Post 'find' — runs after find queries ──────────

userSchema.post(/^find/, function (docs, next) {
  console.log(`📊 Query returned ${docs.length} results`);
  next();
});
```

### Common Middleware Use Cases

```
┌──────────────────────────┬───────────────────────────────────┐
│   Hook                    │   Common Use Cases                │
├──────────────────────────┼───────────────────────────────────┤
│   pre('save')             │ • Hash passwords                 │
│                           │ • Generate slugs                 │
│                           │ • Set timestamps                 │
│                           │ • Validate related data          │
│                           │                                   │
│   post('save')            │ • Send welcome email             │
│                           │ • Log creation events            │
│                           │ • Trigger notifications          │
│                           │                                   │
│   pre('find')             │ • Filter out deleted/inactive    │
│   pre(/^find/)            │ • Add default sort order         │
│                           │ • Auto-populate references       │
│                           │                                   │
│   pre('remove')           │ • Delete related documents       │
│   pre('deleteOne')        │ • Clean up references            │
│                           │ • Remove uploaded files          │
│                           │                                   │
│   pre('validate')         │ • Complex cross-field validation │
│   post('validate')        │ • Log validation results         │
└──────────────────────────┴───────────────────────────────────┘
```

### The `next()` Flow

```
  Pre Middleware 1              Pre Middleware 2              Actual Operation
  (hash password)               (create slug)                 (.save())
       │                             │                             │
       ├── next() ──────────────────►├── next() ──────────────────►│
       │                             │                             │
       │                             │                             ▼
       │                             │                        Saved to DB
       │                             │                             │
       │                             │                             ▼
       │                             │                    Post Middleware
       │                             │                    (send email)
```

> **Critical:** Always call `next()` in your middleware! If you forget, the request will **hang forever** and never complete.

## Indexes for Performance

Indexes speed up database queries dramatically. Without an index, MongoDB scans **every document** in a collection.

```js
const userSchema = new mongoose.Schema({
  name: String,
  email: {
    type: String,
    unique: true, // Creates a unique index automatically
    index: true, // Creates a regular index (for fast lookups)
  },
  role: String,
  createdAt: Date,
  location: {
    type: { type: String, default: "Point" },
    coordinates: [Number],
  },
});

// ──── Compound Index (multiple fields) ────────────────
// Useful when you frequently query by both fields together
userSchema.index({ role: 1, createdAt: -1 });
// 1 = ascending, -1 = descending
// Optimizes: User.find({ role: "admin" }).sort({ createdAt: -1 })

// ──── Text Index (for full-text search) ───────────────
userSchema.index({ name: "text", email: "text" });
// Optimizes: User.find({ $text: { $search: "alice" } })

// ──── Geospatial Index ─────────────────────────────
userSchema.index({ location: "2dsphere" });
// Optimizes: location-based queries (find nearby users)
```

```
Without Index:                     With Index:
──────────────────────            ──────────────────────

Query: Find user with             Query: Find user with
email = "alice@..."               email = "alice@..."

MongoDB scans ALL docs:           MongoDB uses index:
Doc 1 → No match                  Index lookup → found!
Doc 2 → No match                  Jump directly to doc
Doc 3 → No match                  
...                                Time: ~1ms ✅
Doc 999 → No match
Doc 1000 → MATCH!                 

Time: ~100ms ❌                   
(gets worse as data grows)        
```

> **Warning:** Don't add indexes to every field! Each index takes up memory and slows down write operations (inserts/updates). Only index fields you frequently query or sort by.

## Putting It All Together — Complete User Model

Here's a comprehensive, production-ready User model that combines everything we've learned:

```js
// models/User.js — Production-ready User Model

const mongoose = require("mongoose");
const bcrypt = require("bcryptjs");

const userSchema = new mongoose.Schema(
  {
    // ─── Basic Info ──────────────────────────────────
    firstName: {
      type: String,
      required: [true, "First name is required"],
      trim: true,
      minlength: [2, "First name must be at least 2 characters"],
      maxlength: [50, "First name cannot exceed 50 characters"],
    },
    lastName: {
      type: String,
      required: [true, "Last name is required"],
      trim: true,
      minlength: [2, "Last name must be at least 2 characters"],
    },

    // ─── Authentication ──────────────────────────────
    email: {
      type: String,
      required: [true, "Email is required"],
      unique: true,
      lowercase: true,
      match: [/^\S+@\S+\.\S+$/, "Please provide a valid email"],
    },
    password: {
      type: String,
      required: [true, "Password is required"],
      minlength: [8, "Password must be at least 8 characters"],
      select: false, // NEVER include password in query results
    },
    passwordChangedAt: Date,

    // ─── Profile ─────────────────────────────────────
    role: {
      type: String,
      enum: {
        values: ["user", "admin", "moderator"],
        message: "{VALUE} is not a valid role",
      },
      default: "user",
    },
    avatar: {
      type: String,
      default: "default-avatar.png",
    },
    bio: {
      type: String,
      maxlength: [500, "Bio cannot exceed 500 characters"],
      default: "",
    },

    // ─── Status ──────────────────────────────────────
    isActive: {
      type: Boolean,
      default: true,
      select: false, // Hide from normal queries
    },
    isVerified: {
      type: Boolean,
      default: false,
    },
  },
  {
    timestamps: true, // Adds createdAt and updatedAt
    toJSON: { virtuals: true },
    toObject: { virtuals: true },
  }
);

// ─── Indexes ─────────────────────────────────────────
userSchema.index({ email: 1 }); // Fast email lookups
userSchema.index({ role: 1, isActive: 1 }); // Fast role-based queries

// ─── Virtual: Full Name ─────────────────────────────
userSchema.virtual("fullName").get(function () {
  return `${this.firstName} ${this.lastName}`;
});

// ─── Pre-save: Hash Password ────────────────────────
userSchema.pre("save", async function (next) {
  if (!this.isModified("password")) return next();
  this.password = await bcrypt.hash(this.password, 12);
  next();
});

// ─── Pre-save: Update passwordChangedAt ─────────────
userSchema.pre("save", function (next) {
  if (!this.isModified("password") || this.isNew) return next();
  this.passwordChangedAt = Date.now() - 1000;
  next();
});

// ─── Pre-find: Exclude inactive users ───────────────
userSchema.pre(/^find/, function (next) {
  this.find({ isActive: { $ne: false } });
  next();
});

// ─── Instance Method: Check Password ────────────────
userSchema.methods.correctPassword = async function (candidatePassword) {
  return await bcrypt.compare(candidatePassword, this.password);
};

// ─── Instance Method: Changed Password After Token ──
userSchema.methods.changedPasswordAfter = function (JWTTimestamp) {
  if (this.passwordChangedAt) {
    const changedTimestamp = parseInt(
      this.passwordChangedAt.getTime() / 1000,
      10
    );
    return JWTTimestamp < changedTimestamp;
  }
  return false; // Not changed
};

// ─── Static Method: Find Active Users ───────────────
userSchema.statics.findActive = function () {
  return this.find({ isActive: true });
};

// ─── Export the Model ───────────────────────────────
module.exports = mongoose.model("User", userSchema);
```

## Common Patterns and Best Practices

### 1. One Model Per File

```
models/
├── User.js       ← One model per file
├── Product.js
├── Order.js
└── Review.js
```

### 2. Always Use `select: false` for Sensitive Fields

```js
password: {
  type: String,
  select: false,   // Won't appear in User.find() results
}

// To explicitly include it when needed:
const user = await User.findById(id).select("+password");
```

### 3. Use `timestamps: true` Instead of Manual Date Fields

```js
// ❌ Don't do this manually
const schema = new mongoose.Schema({
  createdAt: { type: Date, default: Date.now },
  updatedAt: { type: Date, default: Date.now },
});

// ✅ Let Mongoose handle it
const schema = new mongoose.Schema({ ... }, {
  timestamps: true    // Auto-creates createdAt and updatedAt
});
```

### 4. Validate on Updates Too

```js
// ❌ Validators don't run by default on updates
User.findByIdAndUpdate(id, data);

// ✅ Always include runValidators
User.findByIdAndUpdate(id, data, {
  new: true,
  runValidators: true,
});
```

### 5. Use `immutable` for Fields That Should Never Change

```js
email: {
  type: String,
  immutable: true,  // Cannot be changed after creation
}
```

### 6. Pre-find Hooks for Soft Deletes

```js
// Instead of actually deleting documents, mark them as inactive
userSchema.pre(/^find/, function (next) {
  this.find({ isActive: { $ne: false } });  // Filter out "deleted" users
  next();
});
```
