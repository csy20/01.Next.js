# MongoDB Fundamentals


## 🎯 What is a Database?

Before diving into MongoDB, let's understand what a database actually is.

```
Analogy — Filing Cabinet:
──────────────────────────────────────────────

  Imagine you run a business. You have:
  • Customer information
  • Product inventory
  • Sales records

  WITHOUT a database (paper files):
  ┌─────────────────────────────────┐
  │ 📁 Folder: Customers            │
  │   📄 Alice.txt                  │
  │   📄 Bob.txt                    │
  │   📄 Charlie.txt                │
  │ 📁 Folder: Products             │
  │   📄 Laptop.txt                 │
  │   📄 Phone.txt                  │
  │ 📁 Folder: Sales                │
  │   📄 sale_001.txt               │
  │   📄 sale_002.txt               │
  └─────────────────────────────────┘

  Problems:
  • Finding Alice's sales takes minutes of searching
  • Two people can't read the same file at once
  • No backup if the cabinet catches fire
  • Searching by "sales > $100" is impossible

  WITH a database:
  • Instant search by any field
  • Multiple users can access simultaneously
  • Automatic backups
  • Complex queries in milliseconds

  A DATABASE is organized, searchable, persistent storage
  for your application's data.
```

## 🔀 SQL vs NoSQL — The Two Worlds

### 📝 What is SQL?

**SQL** (Structured Query Language) databases store data in **tables** with fixed rows and columns — like a spreadsheet. Every row has exactly the same columns.

```
SQL Database (e.g., MySQL, PostgreSQL):
──────────────────────────────────────────────

  users table:
  ┌────┬──────────┬──────────────────┬─────┬────────────┐
  │ id │ name     │ email            │ age │ country    │
  ├────┼──────────┼──────────────────┼─────┼────────────┤
  │ 1  │ Alice    │ alice@gmail.com  │ 28  │ USA        │
  │ 2  │ Bob      │ bob@yahoo.com    │ 35  │ UK         │
  │ 3  │ Charlie  │ charlie@mail.com │ 22  │ India      │
  └────┴──────────┴──────────────────┴─────┴────────────┘

  Key Characteristics:
  • Fixed SCHEMA — every row has the same columns
  • You must define the schema BEFORE adding data
  • Adding a new column? EVERY row gets it (even if null)
  • Relationships via FOREIGN KEYS and JOINs
  • Uses the SQL language for queries
  • ACID transactions (Atomicity, Consistency, Isolation, Durability)

  Query: SELECT * FROM users WHERE age > 25;
  Result: Alice (28), Bob (35)
```

---

### 📝 What is NoSQL?

**NoSQL** (Not Only SQL) databases store data in **documents** (like JSON objects). Each document can have different fields — no fixed schema required.

```
NoSQL Database (e.g., MongoDB):
──────────────────────────────────────────────

  users collection:
  ┌─────────────────────────────────────────────┐
  │ {                                            │
  │   _id: ObjectId("507f1f77bcf86cd7994390"),  │
  │   name: "Alice",                            │
  │   email: "alice@gmail.com",                 │
  │   age: 28,                                  │
  │   country: "USA",                           │
  │   hobbies: ["reading", "hiking"]            │ ← Arrays! SQL can't do this easily
  │ }                                            │
  ├─────────────────────────────────────────────┤
  │ {                                            │
  │   _id: ObjectId("507f1f77bcf86cd7994391"),  │
  │   name: "Bob",                              │
  │   email: "bob@yahoo.com",                   │
  │   age: 35                                   │ ← No hobbies or country field — that's OK!
  │ }                                            │
  ├─────────────────────────────────────────────┤
  │ {                                            │
  │   _id: ObjectId("507f1f77bcf86cd7994392"),  │
  │   name: "Charlie",                          │
  │   email: "charlie@mail.com",                │
  │   age: 22,                                  │
  │   country: "India",                         │
  │   address: {                                │ ← Nested objects! SQL can't do this natively
  │     city: "Mumbai",                         │
  │     zip: "400001"                           │
  │   }                                         │
  │ }                                            │
  └─────────────────────────────────────────────┘

  Key Characteristics:
  • FLEXIBLE schema — each document can be different
  • No need to define schema before adding data (but Mongoose helps)
  • Adding a new field? Just add it to the new documents
  • Relationships via embedding or referencing
  • Uses its own query language (not SQL)
  • Horizontal scaling (easy to distribute across servers)
```

MongoDB is a **document-oriented NoSQL database** built for modern applications. The name comes from "hu**mongo**us" — it's designed to handle huge amounts of data.

### 🌟 Key Features of MongoDB

```
1. DOCUMENT MODEL
   Data stored as JSON-like documents (BSON internally)
   Natural mapping to JavaScript objects
   No ORM mismatch like with SQL

2. FLEXIBLE SCHEMA
   Each document can have different fields
   Easy to evolve your data model
   Add fields without migration scripts

3. RICH QUERY LANGUAGE
   Filtering, sorting, projection, aggregation
   Regular expressions, geospatial queries
   Full-text search

4. INDEXING
   Any field can be indexed (including nested fields)
   Compound indexes, text indexes, geospatial indexes
   Dramatically speeds up queries

5. AGGREGATION PIPELINE
   Multi-stage data processing
   Group, sort, filter, reshape
   Like SQL GROUP BY but more powerful

6. REPLICATION
   Multiple copies of your data across servers
   Automatic failover if primary server dies
   High availability out of the box

7. SHARDING (Horizontal Scaling)
   Split data across multiple machines
   Handle billions of documents
   Add more servers as you grow
```

## 📄 Documents — MongoDB's Building Blocks

### 📝 What is a Document?

A document is a **record** in MongoDB. It's a JSON-like data structure made up of **field-value pairs**. Think of it as a JavaScript object stored in the database.

```js
// This is a MongoDB document:
{
  _id: ObjectId("507f1f77bcf86cd799439011"),
  name: "Alice Johnson",
  email: "alice@example.com",
  age: 28,
  isActive: true,
  hobbies: ["reading", "hiking", "photography"],
  address: {
    street: "123 Main St",
    city: "Springfield",
    state: "IL",
    zip: "62701"
  },
  createdAt: ISODate("2024-01-15T10:30:00Z")
}
```

```
Document Anatomy:
──────────────────────────────────────────────

  {                                     ← Document starts
    _id: ObjectId("507f..."),           ← Unique identifier (required)
    name: "Alice Johnson",              ← String field
    age: 28,                            ← Number field
    isActive: true,                     ← Boolean field
    hobbies: ["reading", "hiking"],     ← Array field
    address: {                          ← Nested document (subdocument)
      city: "Springfield",
      state: "IL"
    },
    createdAt: ISODate("2024-...")      ← Date field
  }                                     ← Document ends

  Each line is a FIELD (key-value pair).
  Fields can hold any data type.
  Documents in the same collection can have different fields.
```

---

### 📐 Document Structure and Rules

```
┌──────────────────────────────────────────────────────────────┐
│              DOCUMENT RULES                                    │
├──────────────────────────────────────────────────────────────┤
│                                                               │
│  1. Maximum size: 16 MB per document                         │
│     That's about 16 million characters of text.              │
│     More than enough for most use cases.                     │
│     If you need more, use GridFS for large files.            │
│                                                               │
│  2. Every document MUST have an _id field                    │
│     MongoDB auto-generates one if you don't provide it.      │
│     The _id is always the primary key.                       │
│     It must be unique within the collection.                 │
│                                                               │
│  3. Field names are strings                                  │
│     Cannot start with $ (reserved for operators)             │
│     Cannot contain null characters                           │
│     Are case-sensitive ("Name" ≠ "name")                    │
│     Can contain dots (.) but avoid it (used for nested access)│
│                                                               │
│  4. Field order matters for comparison                       │
│     { a: 1, b: 2 } ≠ { b: 2, a: 1 } in strict comparison │
│     But queries work regardless of order.                    │
│                                                               │
│  5. Duplicate field names are NOT allowed                    │
│     Each field name must be unique within a document.        │
│                                                               │
└──────────────────────────────────────────────────────────────┘
```

### 🔑 The _id Field

Every MongoDB document has a unique `_id` field. If you don't provide one, MongoDB generates an **ObjectId** automatically.

```
ObjectId Anatomy:
──────────────────────────────────────────────

  ObjectId("507f1f77bcf86cd799439011")
  │          │          │
  │          └──────────┴── 24 hex characters = 12 bytes
  │
  └── ObjectId type wrapper

  The 12 bytes encode:
  ┌──────────┬────────┬──────┬──────────┐
  │ 4 bytes  │ 5 bytes│ 3 bytes│         │
  │ Timestamp│ Random │Counter │         │
  │ (seconds)│ value  │       │          │
  └──────────┴────────┴──────┴──────────┘

  • Timestamp: When the ObjectId was created (Unix time)
  • Random: Machine and process identifier
  • Counter: Incrementing value for uniqueness

  This means:
  ✅ ObjectIds are (roughly) sortable by creation time
  ✅ They're unique across machines (no coordination needed)
  ✅ You can extract the creation time:
     ObjectId("507f1f77bcf86cd799439011").getTimestamp()
     → 2012-10-17T20:46:00.000Z
```

```js
// You CAN provide your own _id:
db.users.insertOne({
  _id: "custom-id-123",
  name: "Alice"
});

// But usually, let MongoDB generate it:
db.users.insertOne({
  name: "Alice"
  // _id will be auto-generated as ObjectId
});

// ObjectId is the default and recommended type for _id
```

### 📦 Nested Documents (Embedded Documents)

Documents can contain other documents — called **embedded documents** or **subdocuments**:

```js
{
  _id: ObjectId("..."),
  name: "Alice",

  // Level 1 nesting
  address: {
    street: "123 Main St",
    city: "Springfield",
    state: "IL",

    // Level 2 nesting
    coordinates: {
      lat: 39.7817,
      lng: -89.6501
    }
  },

  // Another embedded document
  preferences: {
    theme: "dark",
    language: "en",
    notifications: {
      email: true,
      push: false,
      sms: false
    }
  }
}

// Accessing nested fields (dot notation):
// address.city → "Springfield"
// address.coordinates.lat → 39.7817
// preferences.notifications.email → true
```

```
Nesting Limits:
──────────────────────────────────────────────

  MongoDB supports up to 100 levels of nesting.
  But in practice, keep it to 2-3 levels max.

  ✅ Good:  user.address.city
  ✅ OK:    user.address.coordinates.lat
  ❌ Bad:   user.company.department.team.lead.preferences.theme
             (too deep — hard to query and maintain)
```

---

### 📊 Arrays in Documents

Documents can contain arrays of any type — this is one of MongoDB's most powerful features:

```js
{
  _id: ObjectId("..."),
  name: "Alice",

  // Array of strings
  tags: ["developer", "javascript", "nodejs"],

  // Array of numbers
  scores: [95, 87, 92, 88],

  // Array of embedded documents
  education: [
    {
      school: "MIT",
      degree: "B.S. Computer Science",
      year: 2018
    },
    {
      school: "Stanford",
      degree: "M.S. Machine Learning",
      year: 2020
    }
  ],

  // Array of ObjectIds (references to other documents)
  friendIds: [
    ObjectId("507f1f77bcf86cd799439012"),
    ObjectId("507f1f77bcf86cd799439013"),
    ObjectId("507f1f77bcf86cd799439014")
  ]
}
```

```
Arrays vs SQL — A Huge Advantage:
──────────────────────────────────────────────

  In SQL, to store a user's hobbies:
  You need a SEPARATE table and a JOIN!

  users:                    user_hobbies:
  ┌────┬───────┐           ┌─────────┬────────────┐
  │ id │ name  │           │ user_id │ hobby      │
  ├────┼───────┤           ├─────────┼────────────┤
  │ 1  │ Alice │           │ 1       │ reading    │
  └────┴───────┘           │ 1       │ hiking     │
                           │ 1       │ photography│
                           └─────────┴────────────┘

  Query: SELECT h.hobby FROM users u
         JOIN user_hobbies h ON u.id = h.user_id
         WHERE u.name = 'Alice';

  In MongoDB, it's just an array field:

  { name: "Alice", hobbies: ["reading", "hiking", "photography"] }

  Query: db.users.findOne({ name: "Alice" }).hobbies
  → ["reading", "hiking", "photography"]

  ONE query. No JOINs. Much simpler!
```

## 💾 BSON — How MongoDB Stores Data

### 📝 What is JSON?

**JSON** (JavaScript Object Notation) is a text format for representing structured data. It's what you write and read:

```json
{
  "name": "Alice",
  "age": 28,
  "isActive": true,
  "hobbies": ["reading", "hiking"],
  "address": {
    "city": "Springfield"
  }
}
```

```
JSON Data Types (only 6!):
──────────────────────────────────────────────

  1. String:   "hello"
  2. Number:   42, 3.14
  3. Boolean:  true, false
  4. Array:    [1, 2, 3]
  5. Object:   { "key": "value" }
  6. null:     null

  That's it! JSON is simple but limited.
  No dates, no binary data, no integers vs floats.
```

---

### 📝 What is BSON?

**BSON** (Binary JSON) is what MongoDB ACTUALLY stores on disk. It extends JSON with more data types and is designed for efficiency.

```
JSON vs BSON:
──────────────────────────────────────────────

  You write JSON:                  MongoDB stores BSON:
  ┌──────────────────────┐        ┌──────────────────────┐
  │ {                    │        │ \x16\x00\x00\x00     │
  │   "name": "Alice",  │  ───►  │ \x02name\x00\x06\x00 │
  │   "age": 28         │        │ \x00Alice\x00\x10age  │
  │ }                    │        │ \x00\x1c\x00\x00\x00 │
  └──────────────────────┘        └──────────────────────┘
  Human-readable text              Binary format (faster!)

  Think of it like:
  • JSON = English (you can read it)
  • BSON = Machine code (the computer reads it faster)

  MongoDB automatically converts:
  JSON → BSON when you INSERT data
  BSON → JSON when you READ data
  You never deal with BSON directly!
```

```
├────────────────────┼──────────────────────┼──────────────────────┤
│ Format             │ Text (human-readable)│ Binary (machine-opt) │

```

### 📋 BSON Data Types

```
┌──────────────────────────────────────────────────────────────┐
│              BSON DATA TYPES                                   │
├──────────────────────────────────────────────────────────────┤
│                                                               │
│  Type         │ Example                    │ Notes            │
│  ─────────────┼───────────────────────────┼─────────────────  │
│  Double       │ 3.14                      │ 64-bit float     │
│  String       │ "hello"                   │ UTF-8 encoded    │
│  Object       │ { key: "value" }          │ Embedded doc     │
│  Array        │ [1, 2, 3]                 │ Ordered list     │
│  Binary Data  │ BinData(0, "...")         │ Images, files    │
│  ObjectId     │ ObjectId("507f...")       │ 12-byte unique ID│
│  Boolean      │ true, false               │                  │
│  Date         │ ISODate("2024-01-15")    │ Millisecond prec │
│  Null         │ null                      │ Absent value     │
│  RegExp       │ /pattern/i               │ Regular expression│
│  32-bit Int   │ NumberInt(42)             │ Integer          │
│  64-bit Int   │ NumberLong(9007199254740) │ Large integer    │
│  Decimal128   │ NumberDecimal("9.99")     │ Precise decimals │
│  Timestamp    │ Timestamp(...)            │ Internal use     │
│  Min/Max Key  │ MinKey, MaxKey            │ Comparison bounds│
│                                                               │

│                                                               │
└──────────────────────────────────────────────────────────────┘
```

```
Why BSON Data Types Matter:
──────────────────────────────────────────────

  JSON:  { "price": 9.99 }
  → Stored as a floating-point number
  → 9.99 might become 9.990000000000002 (floating point error!)

  BSON:  { price: NumberDecimal("9.99") }
  → Stored as exact decimal
  → 9.99 stays exactly 9.99

  This matters for financial applications!

  JSON:  { "birthday": "2024-01-15" }
  → Just a string. Can't do date math.

  BSON:  { birthday: ISODate("2024-01-15") }
  → Real date. Can sort, compare, calculate age, etc.
```

## 📁 Collections — Groups of Documents

A **collection** is a group of documents — like a table in SQL, but without a fixed schema.

```
Collection:
──────────────────────────────────────────────

  "users" collection:
  ┌─────────────────────────────────────────┐
  │ { _id: ..., name: "Alice", age: 28 }   │  ← Document 1
  │ { _id: ..., name: "Bob", age: 35 }     │  ← Document 2
  │ { _id: ..., name: "Charlie" }           │  ← Document 3 (no age — that's OK!)
  └─────────────────────────────────────────┘

  "products" collection:
  ┌──────────────────────────────────────────────┐
  │ { _id: ..., title: "Laptop", price: 999 }   │
  │ { _id: ..., title: "Phone", price: 699 }    │
  └──────────────────────────────────────────────┘

  Each collection holds documents of a SIMILAR type.
  But MongoDB doesn't enforce this — Mongoose does.
```

### 🏗️ Creating Collections

```js
// Collections are created AUTOMATICALLY when you insert the first document:
db.users.insertOne({ name: "Alice" });
// The "users" collection is created automatically if it doesn't exist!

// Or create explicitly (rarely needed):
db.createCollection("products");

// Create with options (validation rules):
db.createCollection("orders", {
  validator: {
    $jsonSchema: {
      bsonType: "object",
      required: ["customer", "total"],
      properties: {
        customer: { bsonType: "string" },
        total: { bsonType: "number", minimum: 0 }
      }
    }
  }
});
```

### 📝 Collection Rules and Naming

```
Collection Naming Rules:
──────────────────────────────────────────────

  ✅ Use lowercase:          users, products, orders
  ✅ Use plural names:       users (not user)
  ✅ Use camelCase or        userProfiles or user_profiles
     snake_case:
  ❌ Cannot start with:      system. (reserved)
  ❌ Cannot contain:         null character (\0)
  ❌ Cannot be empty string
  ❌ Max length:             120 bytes (usually ~100 chars)

  Common conventions:
  • users          (user accounts)
  • products       (items for sale)
  • orders         (purchase orders)
  • comments       (user comments)
  • sessions       (login sessions)
  • logs           (application logs)
```

### 🔍 Read Operations (Find)

```js
// ──── Find ALL documents ────────────────────────────

db.users.find()
// Returns all documents in the users collection

// Pretty print:
db.users.find().pretty()

// ──── Find with a filter ────────────────────────────

db.users.find({ age: 28 })
// Find users where age is exactly 28

db.users.find({ name: "Alice" })
// Find users named Alice

// ──── Find ONE document ─────────────────────────────

db.users.findOne({ email: "alice@example.com" })
// Returns the FIRST matching document

// ──── Comparison operators ──────────────────────────

db.users.find({ age: { $gt: 25 } })      // age > 25
db.users.find({ age: { $gte: 25 } })     // age >= 25
db.users.find({ age: { $lt: 30 } })      // age < 30
db.users.find({ age: { $lte: 30 } })     // age <= 30
db.users.find({ age: { $ne: 28 } })      // age != 28
db.users.find({ age: { $in: [22, 28] } })// age is 22 OR 28

// ──── Logical operators ─────────────────────────────

// AND (implicit — just add more fields):
db.users.find({ age: { $gt: 25 }, name: "Alice" })
// age > 25 AND name is Alice

// OR:
db.users.find({
  $or: [
    { age: { $lt: 25 } },
    { age: { $gt: 30 } }
  ]
})
// age < 25 OR age > 30

// ──── Projection (select specific fields) ───────────

db.users.find({}, { name: 1, email: 1 })
// Only return name and email (and _id)
// { _id: ..., name: "Alice", email: "alice@..." }

db.users.find({}, { password: 0, __v: 0 })
// Return everything EXCEPT password and __v

// ──── Sorting ───────────────────────────────────────

db.users.find().sort({ age: 1 })    // Sort by age ascending
db.users.find().sort({ age: -1 })   // Sort by age descending
db.users.find().sort({ name: 1, age: -1 })  // Sort by name ASC, then age DESC

// ──── Limit and Skip (pagination) ───────────────────

db.users.find().limit(2)            // Only return 2 documents
db.users.find().skip(2)             // Skip the first 2
db.users.find().skip(2).limit(2)    // Page 2 (2 items per page)

// ──── Count ─────────────────────────────────────────

db.users.countDocuments()           // Count all
db.users.countDocuments({ age: { $gt: 25 } })  // Count matching
```

### ✏️ Update Operations

```js
// ──── Update ONE document ───────────────────────────

db.users.updateOne(
  { name: "Alice" },               // Filter: which document
  { $set: { age: 29, city: "NYC" } }  // Update: what to change
);

// $set only changes the specified fields.
// Other fields remain untouched!

// ──── Update MANY documents ────────────────────────

db.users.updateMany(
  { age: { $lt: 25 } },            // All users under 25
  { $set: { status: "young" } }    // Add/update the status field
);

// ──── Common update operators ───────────────────────

// $set — Set a field's value (or create it)
db.users.updateOne({ name: "Alice" }, { $set: { age: 30 } });

// $unset — Remove a field entirely
db.users.updateOne({ name: "Alice" }, { $unset: { city: "" } });

// $inc — Increment a number
db.users.updateOne({ name: "Alice" }, { $inc: { age: 1 } });

// $push — Add to an array
db.users.updateOne({ name: "Alice" }, { $push: { hobbies: "yoga" } });

// $pull — Remove from an array
db.users.updateOne({ name: "Alice" }, { $pull: { hobbies: "hiking" } });

// $addToSet — Add to array only if not already present
db.users.updateOne({ name: "Alice" }, { $addToSet: { hobbies: "yoga" } });

// ──── Replace a document entirely ───────────────────

db.users.replaceOne(
  { name: "Bob" },
  { name: "Bob", email: "newbob@example.com", age: 36 }
  // Replaces the ENTIRE document (except _id)
);

// ──── Find and update (return the document) ─────────

db.users.findOneAndUpdate(
  { name: "Alice" },
  { $set: { age: 30 } },
  { returnDocument: "after" }  // Return the updated document
);
```

## 🏗️ MongoDB Data Modeling Concepts

```
The Big Decision — Embed vs Reference:
──────────────────────────────────────────────

  EMBEDDING (Denormalization):
  Put related data INSIDE the document

  {
    name: "Alice",
    address: {          ← Embedded!
      city: "NYC",
      zip: "10001"
    }
  }

  ✅ One query gets everything
  ✅ Fast reads
  ❌ Document can get too large
  ❌ Data duplication

  Use for: Data always accessed together, 1:1, 1:Few

  REFERENCING (Normalization):
  Store a pointer (ObjectId) to another collection

  // users collection:
  { _id: "user1", name: "Alice" }

  // addresses collection:
  { _id: "addr1", userId: "user1", city: "NYC" }

  ✅ No duplication
  ✅ Documents stay small
  ❌ Multiple queries needed
  ❌ More complex code

  Use for: Data queried independently, 1:Many, Many:Many
```

## 💡 Common Patterns and Best Practices

```
┌──────────────────────────────────────────────────────────────┐
│              BEST PRACTICES                                    │
├──────────────────────────────────────────────────────────────┤
│                                                               │
│  1. Design your schema around your QUERIES                   │
│     "How will I access this data?" drives your design        │
│                                                               │
│  2. Use Mongoose in Node.js for schema enforcement           │
│     MongoDB is schema-less; Mongoose adds structure          │
│                                                               │

│     db.users.createIndex({ email: 1 })                       │
│     Without indexes, MongoDB scans EVERY document            │
│                                                               │
│  4. Keep documents under 16 KB when possible                 │
│     The hard limit is 16 MB, but smaller is faster           │
│                                                               │
│  5. Use appropriate data types                               │
│     Dates → ISODate (not strings)                            │
│     Money → NumberDecimal (not floats)                        │
│     IDs → ObjectId (not strings)                             │
│                                                               │
│  6. Name collections with plural lowercase                   │
│     users, products, orders (not User, Product)              │
│                                                               │
│  7. Use Atlas for production                                 │
│     Local MongoDB is fine for development only               │
│                                                               │
│  8. Back up your data regularly                              │
│     Atlas does this automatically                            │
│     locally: mongodump / mongorestore                        │
│                                                               │
└──────────────────────────────────────────────────────────────┘
```
