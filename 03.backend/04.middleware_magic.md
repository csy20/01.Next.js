# Middleware Magic


## 🎯 What is Middleware?

Middleware is a **function that runs between the time a request arrives at your server and the time a response is sent back**. The word itself gives it away: "middle" + "ware" — software in the middle.

Every middleware function has access to three things:

1. **`req`** — the request object (incoming data)
2. **`res`** — the response object (tools to send data back)
3. **`next`** — a function that passes control to the next middleware

```js
function myMiddleware(req, res, next) {
  // Do something with req and/or res
  // Then call next() to move on
  next();
}
```

That's it. **Every middleware function is just a regular function with `(req, res, next)`.**

### 🧠 A Simple Mental Model

Think of middleware like a **conveyor belt in a factory**:

```
Raw Request  →  [Middleware 1]  →  [Middleware 2]  →  [Middleware 3]  →  [Route Handler]  →  Response
                 (parse body)      (log request)      (check auth)       (your code)
```

Each station on the conveyor belt:
- Can **inspect** the request
- Can **modify** the request (add data, change properties)
- Can **end the request** early (send a response and stop)
- Can **pass it along** to the next station using `next()`

## ✈️ The Analogy: Airport Security Checkpoints

Imagine you're at an airport trying to board a flight. Before you reach the gate (your route handler), you must pass through several checkpoints (middleware):

```
You (the Request) arrive at the airport

  Checkpoint 1: Ticket Check (Body Parser)
  ┌─────────────────────────────────────────┐
  │ "Do you have a valid ticket?"           │
  │  ✓ Yes → Stamp your ticket → next()    │
  │  ✗ No  → "Sorry, you can't enter"      │
  │           res.status(400).send("No ticket") │
  └─────────────────────────────────────────┘
                    │ next()
                    ▼
  Checkpoint 2: Security Scanner (Logger)
  ┌─────────────────────────────────────────┐
  │ "Let me scan and log your details"      │
  │  📝 Records: time, name, destination    │
  │  Always lets you through → next()       │
  └─────────────────────────────────────────┘
                    │ next()
                    ▼
  Checkpoint 3: Passport Control (Auth)
  ┌─────────────────────────────────────────┐
  │ "Show me your passport and visa"        │
  │  ✓ Valid  → "Welcome!" → next()        │
  │  ✗ Invalid → "Access denied"            │
  │              res.status(401).send("Unauthorized") │
  └─────────────────────────────────────────┘
                    │ next()
                    ▼
  🛫 Gate (Route Handler)
  ┌─────────────────────────────────────────┐
  │ "Welcome aboard!"                       │
  │  res.json({ message: "Flight boarded" })│
  └─────────────────────────────────────────┘
```

Key takeaways from this analogy:

- **Each checkpoint is a middleware function**
- **`next()` is like walking to the next checkpoint** — it passes you along
- **Any checkpoint can stop you** — send a response and NOT call `next()`
- **The order of checkpoints matters** — you can't check the passport before the ticket
- **Some checkpoints just observe** (the logger) — they always call `next()`
- **Some checkpoints can block** (auth) — they may NOT call `next()`

## 📝 Middleware Function Signature

Every middleware function has the same basic shape:

```js
// Named function
function myMiddleware(req, res, next) {
  // Your logic here
  next(); // Pass to the next middleware
}

// Arrow function
const myMiddleware = (req, res, next) => {
  // Your logic here
  next(); // Pass to the next middleware
};

// Using it
app.use(myMiddleware);
```

### 🔑 The Three Parameters

```js
function middleware(req, res, next) {
  //                 │    │    │
  //                 │    │    └── Function: call this to move to the next middleware
  //                 │    └─────── Object: the response you'll send back
  //                 └──────────── Object: the incoming request with all its data
}
```

**`req` (Request Object):**
- Contains everything about the incoming request
- You can READ from it: `req.method`, `req.url`, `req.headers`, `req.body`
- You can WRITE to it: `req.customProperty = "something"` — this is how middleware shares data with later middleware or route handlers
- Whatever you attach to `req` will be available downstream

**`res` (Response Object):**
- Contains methods to send a response back to the client
- `res.send()`, `res.json()`, `res.status()`, etc.
- Once you send a response, **no other middleware or route handler should try to send another response**

**`next` (Next Function):**
- A callback function provided by Express
- When called, it tells Express: "I'm done, move to the next middleware or route handler"
- If you don't call it, the request **hangs** forever (the client just waits and waits...)

## 🔥 The `next()` Function — The Most Important Concept

The `next()` function is what makes the middleware chain work. It's the **mechanism that moves the request from one middleware to the next**. Without `next()`, the chain breaks.

### 🔍 How `next()` Works Internally (Simplified)

Imagine Express maintains an **ordered list (array)** of all your middleware and routes:

```js
// Internally, Express keeps something like this:
const middlewareStack = [
  { path: "*", handler: express.json() },           // Index 0
  { path: "*", handler: loggerMiddleware },          // Index 1
  { path: "*", handler: authMiddleware },            // Index 2
  { path: "GET /api/users", handler: getUsersRoute },// Index 3
];
```

When a request comes in, Express starts at index 0. The `next()` function is essentially **"go to the next item in the array"**:

```
Request arrives
     │
     ▼
[Index 0] express.json()
  - Parses the body
  - Calls next()  ─────────────►  Express moves to index 1
                                        │
                                        ▼
                                  [Index 1] loggerMiddleware
                                    - Logs the request
                                    - Calls next()  ──────►  Express moves to index 2
                                                                    │
                                                                    ▼
                                                              [Index 2] authMiddleware
                                                                - Checks token
                                                                - Calls next()  ──►  Index 3
                                                                                        │
                                                                                        ▼
                                                                                  [Index 3] Route
                                                                                    - res.json(data)
                                                                                    - Response sent!
```

### 📝 Three Things `next()` Can Do

```js
// 1. Move to the next middleware (most common)
function middleware(req, res, next) {
  console.log("I ran!");
  next(); // → Go to whatever is next in the stack
}

// 2. Skip to error-handling middleware
function middleware(req, res, next) {
  try {
    // some code that might fail
    doSomethingRisky();
    next();
  } catch (error) {
    next(error); // → Passing an argument to next() triggers ERROR handlers
  }
}

// 3. Skip to the next ROUTE (not the next middleware)
function middleware(req, res, next) {
  next("route"); // → Skip remaining middleware for THIS route, go to next route
}
```

### ⚠️ `next()` vs `next(error)` — Critical Difference

```js
// next() — everything is fine, move forward
next();
// Express: "OK, let me run the next normal middleware"

// next(error) — something went wrong, jump to error handlers
next(new Error("Something broke!"));
// Express: "Oh no! Let me SKIP all normal middleware and jump to error handlers"
```

This is a **massive** distinction:

```
Normal flow (next()):
  [Middleware 1] → [Middleware 2] → [Middleware 3] → [Route Handler]

Error flow (next(error)):
  [Middleware 1] → ✗ SKIP ✗ → ✗ SKIP ✗ → ✗ SKIP ✗ → [Error Handler]
```

## ⚠️ What Happens If You Forget `next()`?

This is one of the **most common bugs** in Express applications. If you forget to call `next()` and you also don't send a response, the request **hangs forever**.

```js
// ❌ BUG: This middleware never calls next() and never sends a response
app.use((req, res, next) => {
  console.log("Request received!");
  // Oops! Forgot next()!
  // The request just... hangs. The client waits forever.
  // Eventually it times out with no response.
});

app.get("/api/users", (req, res) => {
  // This NEVER runs because the middleware above never called next()
  res.json({ users: [] });
});
```

### 🔄 The Two Valid Endings for Any Middleware

Every middleware function MUST do **one of these two things**:

```js
// Option 1: Call next() to pass control forward
function middlewareA(req, res, next) {
  console.log("Doing something...");
  next(); // ✅ Passes to next middleware/route
}

// Option 2: Send a response to end the cycle
function middlewareB(req, res, next) {
  if (!req.headers.authorization) {
    return res.status(401).json({ error: "No token!" }); // ✅ Sends response, stops chain
  }
  next(); // If we get here, pass forward
}
```

> **Golden Rule:** Every middleware must either call `next()` OR send a response. Never neither. Never both.

### 📬 What About Calling Both `next()` AND Sending a Response?

```js
// ❌ BAD: Calling next() AND sending a response
app.use((req, res, next) => {
  res.send("Hello from middleware!");
  next(); // This will cause the next handler to ALSO try to send a response
  // ERROR: "Cannot set headers after they are sent to the client"
});
```

## 🔄 The Middleware Pipeline: How Express Processes Requests

Let's trace a request through a complete middleware pipeline:

```js
const express = require("express");
const app = express();

// Middleware 1: Body parser
app.use(express.json());

// Middleware 2: Custom logger
app.use((req, res, next) => {
  console.log(`[${new Date().toISOString()}] ${req.method} ${req.url}`);
  next();
});

// Middleware 3: Add a custom property to req
app.use((req, res, next) => {
  req.requestTime = Date.now();
  next();
});

// Route handler
app.get("/api/hello", (req, res) => {
  res.json({
    message: "Hello!",
    requestedAt: req.requestTime, // This was set by Middleware 3!
  });
});

app.listen(3000);
```

### 📨 Trace of `GET /api/hello`:

```
Client sends: GET /api/hello

  Step 1: express.json()
  ┌──────────────────────────────────────┐
  │ Checks Content-Type header           │
  │ No body to parse (it's a GET)        │
  │ Calls next()                         │
  └──────────────┬───────────────────────┘
                 │
  Step 2: Custom Logger
  ┌──────────────┴───────────────────────┐
  │ Prints: [2026-02-25T10:30:00Z] GET /api/hello │
  │ Calls next()                         │
  └──────────────┬───────────────────────┘
                 │
  Step 3: requestTime
  ┌──────────────┴───────────────────────┐
  │ Sets req.requestTime = 1740480600000 │
  │ Calls next()                         │
  └──────────────┬───────────────────────┘
                 │
  Step 4: Route Handler (GET /api/hello)
  ┌──────────────┴───────────────────────┐
  │ Reads req.requestTime (set earlier!) │
  │ Sends res.json({ message, time })   │
  │ Response sent → cycle complete ✓     │
  └──────────────────────────────────────┘

Client receives: { "message": "Hello!", "requestedAt": 1740480600000 }
```

## 🔧 `app.use()` — Mounting Middleware

`app.use()` is the method that **registers middleware** in Express. It tells Express: "Run this function for incoming requests."

### 📝 Syntax

```js
// Run for ALL routes and ALL methods
app.use(middlewareFunction);

// Run only for routes starting with /api
app.use("/api", middlewareFunction);

// Run multiple middleware at once
app.use(middleware1, middleware2, middleware3);

// Mount a router (we'll cover this in a later chapter)
app.use("/api/users", userRouter);
```

### 🔄 Global vs Path-Specific Middleware

```js
// GLOBAL — runs on EVERY request regardless of URL
app.use((req, res, next) => {
  console.log("This runs for every request");
  next();
});

// PATH-SPECIFIC — only runs for URLs starting with /api
app.use("/api", (req, res, next) => {
  console.log("This only runs for /api/* routes");
  next();
});

// Examples of what matches /api:
// /api           ✓
// /api/users     ✓
// /api/users/42  ✓
// /about         ✗ (doesn't start with /api)
// /              ✗
```

### 💡 Middleware Mounting with Paths — Detailed Examples

```js
app.use("/admin", (req, res, next) => {
  console.log("Admin area middleware");
  next();
});

// This triggers the admin middleware:
// GET /admin
// GET /admin/dashboard
// POST /admin/users
// DELETE /admin/users/5

// This does NOT trigger the admin middleware:
// GET /
// GET /api/users
// GET /administrator  (doesn't start with exactly "/admin")
```

> **Note:** When you mount middleware on a path, Express **strips that path prefix** from `req.url` inside the middleware. So for `GET /admin/dashboard`, inside the middleware `req.url` would be `/dashboard`, not `/admin/dashboard`. The original full URL is in `req.originalUrl`.

## 📋 Types of Middleware in Express

Express recognizes **five categories** of middleware:

```
┌─────────────────────────────────────────────────────────────────┐
│                     Types of Middleware                         │
├─────────────────────┬───────────────────────────────────────────┤
│ 1. Application-level │ Bound to app using app.use() or app.METHOD() │
│ 2. Router-level      │ Bound to express.Router()               │
│ 3. Error-handling    │ Has 4 args: (err, req, res, next)       │
│ 4. Built-in          │ Ships with Express (json, urlencoded, static) │
│ 5. Third-party       │ Installed via npm (morgan, cors, helmet)│
└─────────────────────┴───────────────────────────────────────────┘
```

Let's explore each in detail.

## 🏗️ Built-in Middleware

Express ships with **three built-in middleware** functions. You don't need to install anything — they come with Express.

### 📬 `express.json()`

We covered this in the previous chapter, but let's see it through the middleware lens:

```js
app.use(express.json());
```

**What it does:**

1. Checks if `Content-Type` header is `application/json`
2. If yes, reads the raw body stream (the bytes coming in)
3. Runs `JSON.parse()` on the raw string
4. Attaches the result to `req.body`
5. Calls `next()` so the next middleware/route can access `req.body`

**What it is under the hood:**

```js
// Simplified version of what express.json() does internally
function jsonParser(req, res, next) {
  // Skip if not JSON content type
  if (req.headers["content-type"] !== "application/json") {
    return next(); // Not my job, pass along
  }

  let rawBody = "";

  // Read raw bytes from the request stream
  req.on("data", (chunk) => {
    rawBody += chunk.toString();
  });

  req.on("end", () => {
    try {
      req.body = JSON.parse(rawBody); // Parse and attach to req
      next(); // Success! Move on
    } catch (err) {
      // Invalid JSON
      res.status(400).json({ error: "Invalid JSON" });
    }
  });
}
```

---

### 📬 `express.urlencoded()`

```js
app.use(express.urlencoded({ extended: true }));
```

**What it does:**

1. Checks if `Content-Type` header is `application/x-www-form-urlencoded`
2. Reads the body: `name=Alice&age=30`
3. Parses it into an object: `{ name: "Alice", age: "30" }`
4. Attaches it to `req.body`
5. Calls `next()`

**The `extended` option:**

```js
// extended: false → Uses Node's built-in 'querystring' module
// Handles flat key-value pairs only
// name=Alice&age=30 → { name: "Alice", age: "30" }

// extended: true → Uses the 'qs' library (more powerful)
// Handles nested objects and arrays
// user[name]=Alice&user[age]=30 → { user: { name: "Alice", age: "30" } }
// colors[]=red&colors[]=blue → { colors: ["red", "blue"] }
```

---

### 📬 `express.static()`

This middleware serves **static files** (HTML, CSS, JavaScript, images) from a directory on your server:

```js
// Serve files from the 'public' folder
app.use(express.static("public"));
```

**What it does:**

1. Request comes in: `GET /style.css`
2. Express looks for `public/style.css` on disk
3. If found → sends the file with proper Content-Type header
4. If not found → calls `next()` (lets other middleware/routes handle it)

**Example folder structure:**

```
my-app/
├── public/
│   ├── index.html
│   ├── style.css
│   ├── script.js
│   └── images/
│       └── logo.png
├── index.js
└── package.json
```

```js
const express = require("express");
const app = express();

// Serve everything in the 'public' folder
app.use(express.static("public"));

// Now these URLs work automatically:
// http://localhost:3000/index.html  → serves public/index.html
// http://localhost:3000/style.css   → serves public/style.css
// http://localhost:3000/images/logo.png → serves public/images/logo.png

app.listen(3000);
```

**With a URL prefix:**

```js
// Files are now served under /static/...
app.use("/static", express.static("public"));

// http://localhost:3000/static/style.css → serves public/style.css
// http://localhost:3000/style.css        → won't work anymore (no prefix)
```

**Using absolute path (recommended for production):**

```js
const path = require("path");

// __dirname = absolute path of the current file's directory
app.use(express.static(path.join(__dirname, "public")));
```

> **Why `path.join(__dirname, "public")`?** If you run your server from a different directory (e.g., during deployment), a relative path like `"public"` might resolve to the wrong location. Using `__dirname` ensures it always resolves relative to where your code file actually lives.

## 📦 Third-Party Middleware

Third-party middleware are **npm packages** built by the community. You install them and plug them into your Express app. Let's explore the most important ones.

### 📨 `morgan` — HTTP Request Logger

Morgan automatically **logs every HTTP request** that hits your server. It's invaluable for debugging and monitoring.

#### Installation

```bash
npm install morgan
```

#### Basic Usage

```js
const express = require("express");
const morgan = require("morgan");
const app = express();

// Use the 'dev' format for pretty, colorful output
app.use(morgan("dev"));

app.get("/api/users", (req, res) => {
  res.json([{ id: 1, name: "Alice" }]);
});

app.listen(3000);
```

#### What You See in the Terminal

When someone makes a request:

```bash
GET /api/users 200 12.305 ms - 32
│   │          │   │           │
│   │          │   │           └── Response size (32 bytes)
│   │          │   └────────────── Response time (12.305 milliseconds)
│   │          └─────────────────── Status code (200 OK)
│   └────────────────────────────── URL path
└────────────────────────────────── HTTP method

POST /api/users 201 3.211 ms - 65
GET /api/users/999 404 1.502 ms - 38
```

The color coding in `dev` format:

- **Green** = 2xx (success)
- **Cyan** = 3xx (redirect)
- **Yellow** = 4xx (client error)
- **Red** = 5xx (server error)

#### Morgan Formats

```js
// 'dev' — Concise colored output for development
app.use(morgan("dev"));
// Output: GET /api/users 200 5.123 ms - 42

// 'tiny' — Minimal output
app.use(morgan("tiny"));
// Output: GET /api/users 200 42 - 5.123 ms

// 'short' — Shorter than default, includes remote address
app.use(morgan("short"));
// Output: ::1 - GET /api/users HTTP/1.1 200 42 - 5.123 ms

// 'combined' — Apache combined log format (good for production)
app.use(morgan("combined"));
// Output: ::1 - - [25/Feb/2026:10:30:00 +0000] "GET /api/users HTTP/1.1" 200 42 "-" "Mozilla/5.0..."

// 'common' — Apache common log format
app.use(morgan("common"));
// Output: ::1 - - [25/Feb/2026:10:30:00 +0000] "GET /api/users HTTP/1.1" 200 42

// Custom format string
app.use(morgan(":method :url :status :response-time ms"));
// Output: GET /api/users 200 5.123 ms
```

#### Logging to a File

```js
const fs = require("fs");
const path = require("path");

// Create a write stream (in append mode)
const accessLogStream = fs.createWriteStream(
  path.join(__dirname, "access.log"),
  { flags: "a" } // 'a' = append (don't overwrite)
);

// Log to file using 'combined' format
app.use(morgan("combined", { stream: accessLogStream }));
```

#### Using Different Formats for Dev and Production

```js
if (process.env.NODE_ENV === "production") {
  // In production: log to a file, Apache format
  const logStream = fs.createWriteStream(path.join(__dirname, "access.log"), {
    flags: "a",
  });
  app.use(morgan("combined", { stream: logStream }));
} else {
  // In development: log to console, colorful format
  app.use(morgan("dev"));
}
```

#### How Morgan Works As Middleware (Under the Hood)

```js
// Simplified version of what morgan does
function simpleMorgan(req, res, next) {
  const start = Date.now();

  // Hook into the response's 'finish' event
  // This fires AFTER the response has been sent
  res.on("finish", () => {
    const duration = Date.now() - start;
    console.log(
      `${req.method} ${req.originalUrl} ${res.statusCode} ${duration}ms`
    );
  });

  next(); // Don't block the request! Let it continue.
}
```

Notice: Morgan calls `next()` immediately — it doesn't wait for the response. Instead, it hooks into the response's `finish` event to log the details AFTER the response is sent.

### 📬 `cors` — Cross-Origin Resource Sharing

CORS is one of the **most confusing yet essential** concepts in web development. Let's break it down completely.

#### The Problem: Same-Origin Policy

Browsers have a security rule called the **Same-Origin Policy**. It says:

> "A web page at `http://example.com` can only make requests to `http://example.com`. Requests to any other domain are BLOCKED."

Two URLs have the **same origin** if they share the same:
- **Protocol** (`http` vs `https`)
- **Domain** (`example.com` vs `api.example.com`)
- **Port** (`:3000` vs `:5000`)

```
Same origin? Let's check:

http://localhost:3000  ↔  http://localhost:3000/api     ✓ Same origin
http://localhost:3000  ↔  http://localhost:5000         ✗ Different port!
http://localhost:3000  ↔  https://localhost:3000        ✗ Different protocol!
http://mysite.com      ↔  http://api.mysite.com        ✗ Different subdomain!
```

#### When Does This Matter?

Imagine you have:
- A **React frontend** running on `http://localhost:3000`
- An **Express API** running on `http://localhost:5000`

When your React frontend tries to `fetch("http://localhost:5000/api/users")`, the browser says:

```
❌ Access to fetch at 'http://localhost:5000/api/users' from origin
   'http://localhost:3000' has been blocked by CORS policy:
   No 'Access-Control-Allow-Origin' header is present on the requested resource.
```

The browser **physically blocks** the response. The request actually reaches the server and the server responds, but the **browser refuses to give the response to your JavaScript code**.

#### The Solution: CORS Headers

The server needs to say: "Hey browser, it's OK — I trust requests from that origin." It does this by including special headers in the response:

```
Access-Control-Allow-Origin: http://localhost:3000
```

The `cors` middleware adds these headers automatically.

#### Installation

```bash
npm install cors
```

#### Basic Usage — Allow All Origins

```js
const express = require("express");
const cors = require("cors");
const app = express();

// Allow requests from ANY origin (good for development)
app.use(cors());
// This adds: Access-Control-Allow-Origin: *
```

#### Configuring Specific Origins

```js
// Allow only specific origins (RECOMMENDED for production)
app.use(
  cors({
    origin: "http://localhost:3000", // Only allow this one origin
  })
);

// Allow multiple specific origins
app.use(
  cors({
    origin: ["http://localhost:3000", "https://myapp.com", "https://admin.myapp.com"],
  })
);

// Dynamic origin (check against a whitelist)
const allowedOrigins = [
  "http://localhost:3000",
  "https://myapp.com",
  "https://staging.myapp.com",
];

app.use(
  cors({
    origin: function (origin, callback) {
      // Allow requests with no origin (like mobile apps or curl)
      if (!origin) return callback(null, true);

      if (allowedOrigins.includes(origin)) {
        callback(null, true); // Origin is allowed
      } else {
        callback(new Error("Not allowed by CORS")); // Origin is blocked
      }
    },
  })
);
```

#### Full CORS Configuration

```js
app.use(
  cors({
    origin: "http://localhost:3000", // Allowed origin(s)
    methods: ["GET", "POST", "PUT", "DELETE", "PATCH"], // Allowed HTTP methods
    allowedHeaders: ["Content-Type", "Authorization"], // Allowed request headers
    exposedHeaders: ["X-Total-Count"], // Headers the client can read
    credentials: true, // Allow cookies and auth headers
    maxAge: 86400, // Cache preflight response for 24 hours (seconds)
  })
);
```

#### What Is a "Preflight" Request?

For certain requests (POST with JSON, requests with custom headers, PUT, DELETE), the browser sends a **preflight** request first — an `OPTIONS` request to ask the server: "Is it OK if I send this?"

```
Step 1: Browser sends preflight (OPTIONS request)
  ┌──────────────────────────────────────────┐
  │ OPTIONS /api/users HTTP/1.1              │
  │ Origin: http://localhost:3000            │
  │ Access-Control-Request-Method: POST      │
  │ Access-Control-Request-Headers: Content-Type │
  └──────────────────────────────────────────┘

Step 2: Server responds with CORS headers
  ┌──────────────────────────────────────────┐
  │ HTTP/1.1 204 No Content                  │
  │ Access-Control-Allow-Origin: http://localhost:3000 │
  │ Access-Control-Allow-Methods: GET, POST  │
  │ Access-Control-Allow-Headers: Content-Type │
  └──────────────────────────────────────────┘

Step 3: Browser sees "OK, it's allowed" → sends actual request
  ┌──────────────────────────────────────────┐
  │ POST /api/users HTTP/1.1                 │
  │ Origin: http://localhost:3000            │
  │ Content-Type: application/json           │
  │                                          │
  │ {"name": "Alice"}                        │
  └──────────────────────────────────────────┘
```

The `cors` middleware handles OPTIONS preflight requests automatically.

#### Version 5: Production-Ready Logger (Log to File + Console)

```js
const fs = require("fs");
const path = require("path");

function productionLogger(options = {}) {
  // This is a "factory function" — it returns a middleware function
  // This pattern lets you pass configuration options

  const logFile = options.logFile
    ? fs.createWriteStream(path.join(__dirname, options.logFile), { flags: "a" })
    : null;

  const logToConsole = options.console !== false; // Default: true

  return function (req, res, next) {
    const start = Date.now();

    res.on("finish", () => {
      const duration = Date.now() - start;
      const logEntry = {
        timestamp: new Date().toISOString(),
        method: req.method,
        url: req.originalUrl,
        status: res.statusCode,
        duration: `${duration}ms`,
        ip: req.ip,
        userAgent: req.get("User-Agent"),
      };

      const logString = JSON.stringify(logEntry);

      if (logToConsole) {
        console.log(logString);
      }

      if (logFile) {
        logFile.write(logString + "\n");
      }
    });

    next();
  };
}

// Usage:
app.use(
  productionLogger({
    logFile: "requests.log",
    console: true,
  })
);

// Log output (each line is valid JSON):
// {"timestamp":"2026-02-25T14:30:00.000Z","method":"GET","url":"/api/users","status":200,"duration":"5ms","ip":"::1","userAgent":"Mozilla/5.0..."}
```

> **Key Pattern: Factory Functions.** Notice how `productionLogger(options)` **returns** a middleware function. This is the standard pattern when your middleware needs configuration. Morgan, CORS, and many other packages use this same pattern.

### 🔄 Custom Authorization Middleware

Authorization middleware checks if the user is **allowed** to access a resource. This is one of the most important types of custom middleware.

#### Understanding Tokens First

Most modern APIs use **JWT (JSON Web Tokens)** for authentication. Here's how the flow works:

```
1. User logs in:
   POST /api/login { email: "alice@example.com", password: "secret" }

2. Server verifies credentials and creates a token:
   Token: "eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9.eyJ1c2VySWQiOjEsIm5hbWUiOiJBbGljZSJ9.abc123"

3. Server sends the token back:
   { token: "eyJhbGc..." }

4. Client stores the token (in localStorage, cookie, etc.)

5. Client includes the token in future requests:
   GET /api/profile
   Authorization: Bearer eyJhbGc...
                  ^^^^^^ ^^^^^^^^^
                  Type   The actual token

6. Server checks the token in middleware BEFORE the route handler runs
```

#### Version 1: Simple Token Check (No JWT)

Let's start basic before involving JWTs:

```js
// Middleware that checks for a hardcoded API key
function apiKeyAuth(req, res, next) {
  const apiKey = req.headers["x-api-key"];
  // Or from query string: req.query.api_key

  if (!apiKey) {
    return res.status(401).json({
      error: "Authentication required",
      message: "Please provide an API key in the 'x-api-key' header",
    });
  }

  // In a real app, you'd check this against a database
  const validKeys = ["abc123", "def456", "ghi789"];

  if (!validKeys.includes(apiKey)) {
    return res.status(403).json({
      error: "Forbidden",
      message: "Invalid API key",
    });
  }

  // API key is valid! Let the request continue
  next();
}

// Apply to all /api routes
app.use("/api", apiKeyAuth);
```

```bash
# {"error":"Authentication required","message":"Please provide an API key..."}

# With invalid key → 403
```

#### Version 2: JWT Authorization Middleware

```bash
npm install jsonwebtoken
```

```js
const jwt = require("jsonwebtoken");

// Secret key for signing/verifying tokens
// In production, this would be in an environment variable
const JWT_SECRET = process.env.JWT_SECRET || "my-super-secret-key-change-in-production";

// ── Middleware: Verify JWT Token ──
function authenticate(req, res, next) {
  // Step 1: Get the token from the Authorization header
  const authHeader = req.headers.authorization;
  // Authorization header format: "Bearer <token>"

  if (!authHeader) {
    return res.status(401).json({
      error: "No token provided",
      message: "Include an Authorization header with format: 'Bearer <token>'",
    });
  }

  // Step 2: Extract the token (remove "Bearer " prefix)
  const parts = authHeader.split(" ");

  if (parts.length !== 2 || parts[0] !== "Bearer") {
    return res.status(401).json({
      error: "Invalid token format",
      message: "Authorization header must be: 'Bearer <token>'",
    });
  }

  const token = parts[1];

  // Step 3: Verify the token
  try {
    const decoded = jwt.verify(token, JWT_SECRET);
    // decoded = { userId: 1, name: "Alice", role: "admin", iat: ..., exp: ... }

    // Step 4: Attach user info to the request object
    // This is how middleware shares data with route handlers!
    req.user = decoded;

    // Step 5: Move to the next middleware / route handler
    next();
  } catch (err) {
    if (err.name === "TokenExpiredError") {
      return res.status(401).json({
        error: "Token expired",
        message: "Your session has expired. Please log in again.",
      });
    }

    if (err.name === "JsonWebTokenError") {
      return res.status(401).json({
        error: "Invalid token",
        message: "The token provided is not valid",
      });
    }

    return res.status(500).json({ error: "Failed to authenticate token" });
  }
}
```

**Using the auth middleware:**

```js
// Login route (no auth needed)
app.post("/api/login", (req, res) => {
  const { email, password } = req.body;

  // In real app: check email/password against database
  if (email === "alice@example.com" && password === "secret") {
    // Create a token
    const token = jwt.sign(
      { userId: 1, name: "Alice", role: "admin" },
      JWT_SECRET,
      { expiresIn: "24h" }
    );

    return res.json({ message: "Login successful", token });
  }

  res.status(401).json({ error: "Invalid email or password" });
});

// Protected route — requires authentication
app.get("/api/profile", authenticate, (req, res) => {
  // req.user was set by the authenticate middleware!
  res.json({
    message: `Welcome, ${req.user.name}!`,
    user: req.user,
  });
});

// Protected route — all /api/admin/* routes require auth
app.use("/api/admin", authenticate);

app.get("/api/admin/dashboard", (req, res) => {
  res.json({ message: "Admin dashboard", user: req.user });
});
```

#### Version 3: Role-Based Authorization

Authentication answers "**Who are you?**" — Authorization answers "**What are you allowed to do?**"

```js
// This is a FACTORY FUNCTION that returns middleware
// It takes a list of allowed roles and creates a middleware that checks them
function authorize(...allowedRoles) {
  // Return the actual middleware function
  return (req, res, next) => {
    // authenticate middleware must run first to set req.user
    if (!req.user) {
      return res.status(401).json({
        error: "Authentication required",
        message: "You must be logged in to access this resource",
      });
    }

    // Check if the user's role is in the allowed list
    if (!allowedRoles.includes(req.user.role)) {
      return res.status(403).json({
        error: "Forbidden",
        message: `This resource requires one of these roles: ${allowedRoles.join(", ")}`,
        yourRole: req.user.role,
      });
    }

    // User has the right role — proceed
    next();
  };
}
```

**Using role-based authorization:**

```js
// Anyone logged in can view users
app.get("/api/users", authenticate, (req, res) => {
  res.json(users);
});

// Only admins can delete users
app.delete("/api/users/:id", authenticate, authorize("admin"), (req, res) => {
  // This only runs if:
  // 1. authenticate passes (valid token)
  // 2. authorize("admin") passes (user.role === "admin")
  const id = parseInt(req.params.id, 10);
  // ... delete logic
  res.json({ message: `User ${id} deleted by ${req.user.name}` });
});

// Admins and moderators can edit users
app.put(
  "/api/users/:id",
  authenticate,
  authorize("admin", "moderator"),
  (req, res) => {
    // ... update logic
    res.json({ message: "User updated" });
  }
);

// Only superadmins can access system settings
app.get(
  "/api/system/settings",
  authenticate,
  authorize("superadmin"),
  (req, res) => {
    res.json({ settings: { /* ... */ } });
  }
);
```

**The middleware chain for `DELETE /api/users/5`:**

```
Request: DELETE /api/users/5
Headers: Authorization: Bearer eyJhbGc...

  Step 1: authenticate middleware
  ┌────────────────────────────────────────┐
  │ Extract token from header              │

  Step 2: authorize("admin") middleware
  ┌────────────────┴───────────────────────┐
  │ Check req.user exists ✓                │
  │ Check req.user.role === "admin" ✓      │
  │ Call next()                            │
  └────────────────┬───────────────────────┘
                   │
  Step 3: Route handler
  ┌────────────────┴───────────────────────┐
```

### 🔄 Custom Rate Limiter Middleware

A rate limiter restricts how many requests a client can make in a given time period. This prevents abuse and DDoS attacks.

```js
function rateLimiter(options = {}) {
  const windowMs = options.windowMs || 60 * 1000;   // Default: 1 minute
  const maxRequests = options.max || 100;            // Default: 100 requests per window
  const message = options.message || "Too many requests, please try again later";

  // Store request counts per IP
  const requests = new Map();

  // Clean up old entries periodically
  setInterval(() => {
    const now = Date.now();
    for (const [ip, data] of requests.entries()) {
      if (now - data.startTime > windowMs) {
        requests.delete(ip);
      }
    }
  }, windowMs);

  return (req, res, next) => {
    const ip = req.ip;
    const now = Date.now();

    if (!requests.has(ip)) {
      // First request from this IP
      requests.set(ip, { count: 1, startTime: now });
      return next();
    }

    const clientData = requests.get(ip);

    // If the window has expired, reset the counter
    if (now - clientData.startTime > windowMs) {
      clientData.count = 1;
      clientData.startTime = now;
      return next();
    }

    // Increment the counter
    clientData.count++;

    // Check if over the limit
    if (clientData.count > maxRequests) {
      // Set standard rate limit headers
      res.set("Retry-After", Math.ceil((clientData.startTime + windowMs - now) / 1000));
      res.set("X-RateLimit-Limit", maxRequests);
      res.set("X-RateLimit-Remaining", 0);

      return res.status(429).json({
        error: "Too Many Requests",
        message,
        retryAfter: Math.ceil((clientData.startTime + windowMs - now) / 1000) + " seconds",
      });
    }

    // Set rate limit headers on successful requests too
    res.set("X-RateLimit-Limit", maxRequests);
    res.set("X-RateLimit-Remaining", maxRequests - clientData.count);

    next();
  };
}

// Usage:
// Limit all routes: 100 requests per minute
app.use(rateLimiter({ windowMs: 60 * 1000, max: 100 }));

// Stricter limit on login route: 5 attempts per 15 minutes
app.use(
  "/api/login",
  rateLimiter({
    windowMs: 15 * 60 * 1000, // 15 minutes
    max: 5,
    message: "Too many login attempts. Try again in 15 minutes.",
  })
);
```

### 🔄 Custom Response Time Middleware

Track how long each request takes to process:

```js
function responseTime(req, res, next) {
  const start = process.hrtime.bigint(); // High-resolution timer (nanoseconds)

  // Hook into when the response headers are about to be sent
  res.on("finish", () => {
    const end = process.hrtime.bigint();
    const durationMs = Number(end - start) / 1_000_000; // Convert ns to ms

    // You'll see this header in the response:
    // X-Response-Time: 5.234ms
    res.set("X-Response-Time", `${durationMs.toFixed(3)}ms`);
  });

  // Actually, we need to set the header BEFORE finish...
  // Let's use a different approach:
  const startTime = Date.now();

  // Override res.json to add timing before sending
  const originalJson = res.json.bind(res);
  res.json = function (body) {
    const duration = Date.now() - startTime;
    res.set("X-Response-Time", `${duration}ms`);
    return originalJson(body);
  };

  // Override res.send too
  const originalSend = res.send.bind(res);
  res.send = function (body) {
    const duration = Date.now() - startTime;
    res.set("X-Response-Time", `${duration}ms`);
    return originalSend(body);
  };

  next();
}

app.use(responseTime);
```

A simpler approach that many production apps use:

```js
function responseTime(req, res, next) {
  const start = Date.now();
  
  // The 'finish' event fires right after the response is handed to the OS
  res.on("finish", () => {
    const duration = Date.now() - start;
    // Log it instead (since headers are already sent at this point)
    console.log(`${req.method} ${req.originalUrl} - ${duration}ms`);
  });

  // Or set the header using the 'beforeSend' pattern:
  const originalEnd = res.end;
  res.end = function (...args) {
    const duration = Date.now() - start;
    res.setHeader("X-Response-Time", `${duration}ms`);
    originalEnd.apply(res, args);
  };

  next();
}
```

### 💡 Real Example: Protecting an Admin Endpoint

```js
// Define individual middleware
const logRequest = (req, res, next) => {
  console.log(`${req.method} ${req.originalUrl} by ${req.ip}`);
  next();
};

const requireAuth = (req, res, next) => {
  if (!req.headers.authorization) {
    return res.status(401).json({ error: "Login required" });
  }
  // ... verify token, set req.user
  req.user = { id: 1, name: "Alice", role: "admin" }; // Simplified
  next();
};

const requireAdmin = (req, res, next) => {
  if (req.user.role !== "admin") {
    return res.status(403).json({ error: "Admin access required" });
  }
  next();
};

const validateUserInput = (req, res, next) => {
  if (!req.body.name || !req.body.email) {
    return res.status(400).json({ error: "Name and email required" });
  }
  // Sanitize input
  req.body.name = req.body.name.trim();
  req.body.email = req.body.email.trim().toLowerCase();
  next();
};

// Chain them all together
app.post(
  "/api/admin/users",
  logRequest,         // 1. Log the request
  requireAuth,        // 2. Verify authentication
  requireAdmin,       // 3. Verify admin role
  validateUserInput,  // 4. Validate the data
  (req, res) => {     // 5. Route handler — all checks passed!
    // We're confident: user is admin, data is valid
    const newUser = {
      id: Date.now(),
      name: req.body.name,
      email: req.body.email,
      createdBy: req.user.name,
    };
    res.status(201).json(newUser);
  }
);
```

```
{ "error": "Login required" }
```

## 🛡️ Error-Handling Middleware

Error-handling middleware is special in Express. It has **four parameters** instead of three:

```js
function errorHandler(err, req, res, next) {
  //                   ^^^
  //          The error object — this extra parameter
  //          tells Express this is an error handler
}
```

> **Critical:** Express identifies error-handling middleware **solely by the fact that it has 4 parameters**. If you write `(err, req, res, next)` but accidentally name it `(error, request, response, next)`, it still works — Express counts the number of parameters, not their names.

### ⚠️ How Errors Reach Error-Handling Middleware

```js
// Method 1: Call next(error)
app.get("/api/data", (req, res, next) => {
  try {
    // Something that might throw
    const data = riskyOperation();
    res.json(data);
  } catch (error) {
    next(error); // Pass the error to the error handler
  }
});

// Method 2: Express catches errors thrown in synchronous code
app.get("/api/data", (req, res) => {
  throw new Error("Something broke!"); // Express catches this automatically
});

// Method 3: For async code, you MUST catch and pass to next()
app.get("/api/data", async (req, res, next) => {
  try {
    const data = await fetchFromDatabase();
    res.json(data);
  } catch (error) {
    next(error); // Must explicitly pass async errors
  }
});
```

### ⚠️ Basic Error Handler

```js
// Must be defined AFTER all routes and other middleware
app.use((err, req, res, next) => {
  console.error("ERROR:", err.message);

  res.status(err.status || 500).json({
    error: {
      message: err.message || "Internal Server Error",
    },
  });
});
```

### ⚠️ Detailed Error Handler

```js
app.use((err, req, res, next) => {
  // Log the full error (for developers)
  console.error("─────────────────────────────────");
  console.error("ERROR:", err.message);
  console.error("PATH:", req.method, req.originalUrl);
  console.error("TIME:", new Date().toISOString());
  console.error("STACK:", err.stack);
  console.error("─────────────────────────────────");

  // Determine status code
  const statusCode = err.status || err.statusCode || 500;

  // Response payload
  const response = {
    success: false,
    error: {
      message: err.message || "Internal Server Error",
      status: statusCode,
    },
  };

  // Only include stack trace in development (NEVER in production)
  if (process.env.NODE_ENV !== "production") {
    response.error.stack = err.stack;
  }

  res.status(statusCode).json(response);
});
```

### 🚀 Creating Custom Error Classes

For cleaner error handling, create custom error classes:

```js
class AppError extends Error {
  constructor(message, statusCode) {
    super(message); // Call parent Error constructor
    this.statusCode = statusCode;
    this.status = `${statusCode}`.startsWith("4") ? "fail" : "error";
    this.isOperational = true; // Flag to distinguish from programming bugs

    // Capture a clean stack trace
    Error.captureStackTrace(this, this.constructor);
  }
}

class NotFoundError extends AppError {
  constructor(resource = "Resource") {
    super(`${resource} not found`, 404);
  }
}

class ValidationError extends AppError {
  constructor(message) {
    super(message, 400);
  }
}

class UnauthorizedError extends AppError {
  constructor(message = "Authentication required") {
    super(message, 401);
  }
}

class ForbiddenError extends AppError {
  constructor(message = "Access denied") {
    super(message, 403);
  }
}
```

**Using custom errors in routes:**

```js
app.get("/api/users/:id", (req, res, next) => {
  const id = parseInt(req.params.id, 10);

  if (isNaN(id)) {
    return next(new ValidationError("User ID must be a number"));
  }

  const user = users.find((u) => u.id === id);

  if (!user) {
    return next(new NotFoundError("User"));
  }

  res.json(user);
});

// The error handler catches all of these
app.use((err, req, res, next) => {
  if (err.isOperational) {
    // Expected errors (bad input, not found, etc.)
    return res.status(err.statusCode).json({
      status: err.status,
      message: err.message,
    });
  }

  // Unexpected errors (bugs, database crashes, etc.)
  console.error("UNEXPECTED ERROR:", err);
  res.status(500).json({
    status: "error",
    message: "Something went very wrong",
  });
});
```

### ⚠️ Multiple Error Handlers

You can have multiple error handlers for different purposes:

```js
// Error handler 1: Log all errors
app.use((err, req, res, next) => {
  console.error(`[${new Date().toISOString()}] Error:`, err.message);
  next(err); // Pass to the next error handler
});

// Error handler 2: Handle validation errors specifically
app.use((err, req, res, next) => {
  if (err.name === "ValidationError") {
    return res.status(400).json({
      error: "Validation failed",
      details: err.message,
    });
  }
  next(err); // Not a validation error, pass to next handler
});

// Error handler 3: Handle JWT errors specifically
app.use((err, req, res, next) => {
  if (err.name === "JsonWebTokenError") {
    return res.status(401).json({
      error: "Invalid authentication token",
    });
  }
  next(err);
});

// Error handler 4: Catch-all (final fallback)
app.use((err, req, res, next) => {
  res.status(err.statusCode || 500).json({
    error: err.message || "Internal Server Error",
  });
});
```

## ⚡ Middleware Execution Order — Why It Matters

The **order** in which you register middleware with `app.use()` is the **exact order** they execute. Getting this wrong is one of the most common sources of bugs.

### 📝 Correct Order

```js
const express = require("express");
const morgan = require("morgan");
const cors = require("cors");
const helmet = require("helmet");
const app = express();

// ── PHASE 1: Security & Global Middleware ──
app.use(helmet());                                   // Security headers FIRST
app.use(cors());                                     // CORS headers
app.use(morgan("dev"));                              // Request logging

// ── PHASE 2: Body Parsing ──
app.use(express.json());                             // Parse JSON
app.use(express.urlencoded({ extended: true }));     // Parse forms

// ── PHASE 3: Custom Global Middleware ──
app.use(requestId);                                  // Add request ID
app.use(responseTime);                               // Track response time

// ── PHASE 4: Static Files ──
app.use(express.static("public"));                   // Serve static files

// ── PHASE 5: API Routes ──
app.use("/api", rateLimiter({ max: 100 }));          // Rate limit API routes
app.use("/api/admin", authenticate, authorize("admin")); // Protect admin routes

app.get("/api/users", getUsers);                     // Public route (after rate limit)
app.post("/api/users", authenticate, createUser);    // Protected route
app.delete("/api/users/:id", authenticate, authorize("admin"), deleteUser); // Admin only

// ── PHASE 6: Error Handling (ALWAYS LAST) ──
// 404 handler — only reached if no route matched
app.use((req, res) => {
  res.status(404).json({ error: "Route not found" });
});

// Error handler — only reached if next(error) was called
app.use((err, req, res, next) => {
  console.error(err);
  res.status(err.statusCode || 500).json({ error: err.message });
});
```

### 📝 What Happens If Order Is Wrong

```js
// ❌ BAD: Route before body parser
app.post("/api/users", (req, res) => {
  console.log(req.body); // undefined! Body parser hasn't run yet!
});
app.use(express.json()); // Too late! Already matched the route above.

// ❌ BAD: Error handler before routes
app.use((err, req, res, next) => {
  res.status(500).json({ error: err.message });
});
app.get("/api/users", getUsers); // Errors from this won't be caught

// ❌ BAD: 404 handler before routes
app.use((req, res) => {
  res.status(404).json({ error: "Not found" });
});
app.get("/api/users", getUsers); // Never reached! 404 catches everything first.

// ❌ BAD: Static files served before security middleware
app.use(express.static("public")); // No security headers on static files!
app.use(helmet());
```

### 🔄 Visual: Request Flow Through Middleware Stack

```
Incoming Request: POST /api/users
Content-Type: application/json
Authorization: Bearer abc123
Body: {"name":"Alice"}

│
├──► helmet()              → Sets security headers        → next()
├──► cors()                → Sets CORS headers             → next()
├──► morgan("dev")         → Logs: POST /api/users         → next()
├──► express.json()        → Parses body → req.body={name:"Alice"} → next()
├──► express.urlencoded()  → Not URL-encoded, skip         → next()
├──► requestId             → req.id = "a1b2c3..."          → next()
├──► rateLimiter           → Under limit ✓                 → next()
├──► Route: POST /api/users
│    ├──► authenticate     → Token valid, req.user set     → next()
│    └──► createUser       → Creates user → res.status(201).json(...)
│
└──► Response sent! ✓

morgan logs: POST /api/users 201 15.234 ms - 65
```

## 🌍 Real-World Middleware Stack

Here's what a production Express application's middleware stack typically looks like:

```js
const express = require("express");
const morgan = require("morgan");
const cors = require("cors");
const helmet = require("helmet");
const compression = require("compression");
const cookieParser = require("cookie-parser");
const rateLimit = require("express-rate-limit");
const path = require("path");
const app = express();

// ════════════════════════════════════════════════
// SECURITY
// ════════════════════════════════════════════════
app.use(helmet());
app.use(
  cors({
    origin: process.env.CLIENT_URL || "http://localhost:3000",
    credentials: true,
  })
);

// ════════════════════════════════════════════════
// PERFORMANCE
// ════════════════════════════════════════════════
app.use(compression());

// ════════════════════════════════════════════════
// LOGGING
// ════════════════════════════════════════════════
if (process.env.NODE_ENV === "production") {
  app.use(morgan("combined")); // Detailed logs for production
} else {
  app.use(morgan("dev")); // Colorful logs for development
}

// ════════════════════════════════════════════════
// BODY PARSING
// ════════════════════════════════════════════════
app.use(express.json({ limit: "10mb" }));
app.use(express.urlencoded({ extended: true, limit: "10mb" }));
app.use(cookieParser());

// ════════════════════════════════════════════════
// RATE LIMITING
// ════════════════════════════════════════════════
const generalLimiter = rateLimit({
  windowMs: 15 * 60 * 1000, // 15 minutes
  max: 100, // 100 requests per 15 minutes
  message: { error: "Too many requests, try again later" },
});
app.use("/api", generalLimiter);

const authLimiter = rateLimit({
  windowMs: 15 * 60 * 1000,
  max: 10, // Only 10 login attempts per 15 minutes
  message: { error: "Too many login attempts, try again later" },
});
app.use("/api/auth/login", authLimiter);

// ════════════════════════════════════════════════
// STATIC FILES
// ════════════════════════════════════════════════
app.use(express.static(path.join(__dirname, "public")));

// ════════════════════════════════════════════════
// CUSTOM MIDDLEWARE
// ════════════════════════════════════════════════

// Add request ID to every request
app.use((req, res, next) => {
  req.id = require("crypto").randomUUID();
  res.set("X-Request-Id", req.id);
  next();
});

// Track request timing
app.use((req, res, next) => {
  req.startTime = Date.now();
  next();
});

// ════════════════════════════════════════════════
// ROUTES
// ════════════════════════════════════════════════
const authRoutes = require("./routes/auth");
const userRoutes = require("./routes/users");
const postRoutes = require("./routes/posts");

app.use("/api/auth", authRoutes);
app.use("/api/users", userRoutes);
app.use("/api/posts", postRoutes);

// ════════════════════════════════════════════════
// ERROR HANDLING (always last)
// ════════════════════════════════════════════════

// 404 — No route matched
app.use((req, res) => {
  res.status(404).json({
    success: false,
    error: "Route not found",
    path: req.originalUrl,
    method: req.method,
  });
});

// Global error handler
app.use((err, req, res, next) => {
  const statusCode = err.statusCode || 500;

  // Log error details
  console.error({
    requestId: req.id,
    error: err.message,
    stack: err.stack,
    path: req.originalUrl,
    method: req.method,
    duration: `${Date.now() - req.startTime}ms`,
  });

  res.status(statusCode).json({
    success: false,
    error: process.env.NODE_ENV === "production"
      ? "Something went wrong"
      : err.message,
    ...(process.env.NODE_ENV !== "production" && { stack: err.stack }),
    requestId: req.id,
  });
});

// ════════════════════════════════════════════════
// START SERVER
// ════════════════════════════════════════════════
const PORT = process.env.PORT || 5000;
app.listen(PORT, () => {
  console.log(`Server running on port ${PORT} in ${process.env.NODE_ENV || "development"} mode`);
});
```

## ⚠️ Common Mistakes and Gotchas

### 1. Forgetting `next()` in Middleware

```js
// ❌ Request hangs forever
app.use((req, res, next) => {
  console.log("Logging...");
  // Forgot next()!
});

// ✅ Always call next() if you're not sending a response
app.use((req, res, next) => {
  console.log("Logging...");
  next();
});
```

### 2. Calling `next()` After Sending a Response

```js
// ❌ Causes "Cannot set headers after they are sent"
app.use((req, res, next) => {
  res.json({ message: "Done" });
  next(); // This triggers the next handler which also tries to respond!
});

// ✅ Either send a response OR call next(), never both
app.use((req, res, next) => {
  if (someCondition) {
    return res.json({ message: "Done" }); // RETURN stops execution
  }
  next();
});
```

### 3. Wrong Order of Error Handler Parameters

```js
// ❌ This is NOT an error handler — Express sees 3 params
app.use((req, res, next) => {
  // Express thinks this is regular middleware
});

// ✅ Error handlers MUST have exactly 4 parameters
app.use((err, req, res, next) => {
  // Express recognizes this as an error handler
});

// ❌ Even if you don't use next, you MUST include it
app.use((err, req, res) => {
  // Express sees 3 params — NOT an error handler!
  res.status(500).json({ error: err.message });
});

// ✅ Include all 4 params even if next is unused
app.use((err, req, res, next) => {
  // Express sees 4 params — this IS an error handler
  res.status(500).json({ error: err.message });
});
```

### 4. Async Errors Not Being Caught

```js
// ❌ Unhandled promise rejection — error handler never runs
app.get("/api/users", async (req, res) => {
  const users = await User.find(); // If this throws, Express doesn't catch it
  res.json(users);
});

// ✅ Option 1: Try-catch in every async handler
app.get("/api/users", async (req, res, next) => {
  try {
    const users = await User.find();
    res.json(users);
  } catch (err) {
    next(err); // Manually pass to error handler
  }
});

// ✅ Option 2: Create a wrapper function (DRY approach)
const asyncHandler = (fn) => (req, res, next) => {
  Promise.resolve(fn(req, res, next)).catch(next);
};

// Now use it on all async routes — no try-catch needed!
app.get(
  "/api/users",
  asyncHandler(async (req, res) => {
    const users = await User.find(); // If this throws, asyncHandler catches it
    res.json(users);
  })
);

app.get(
  "/api/users/:id",
  asyncHandler(async (req, res) => {
    const user = await User.findById(req.params.id);
    if (!user) throw new NotFoundError("User");
    res.json(user);
  })
);
```

> **Note:** Starting from Express 5 (currently in beta), async errors are automatically caught. But Express 4 (the current stable version) requires manual handling.

### 5. Modifying `res` After Response Is Sent

```js
// ❌ Setting headers after response is sent
app.get("/api/data", (req, res) => {
  res.json({ data: "hello" });      // Response sent!
  res.set("X-Custom", "value");     // Too late! Headers already sent.
});

// ✅ Set headers BEFORE sending the response
app.get("/api/data", (req, res) => {
  res.set("X-Custom", "value");     // Set header first
  res.json({ data: "hello" });      // Then send response
});
```

### 6. Middleware Defined After Routes

```js
// ❌ This middleware never runs for /api/users — route was already matched
app.get("/api/users", (req, res) => {
  res.json([]);
});
app.use(myMiddleware); // Too late!

// ✅ Define middleware BEFORE the routes it should affect
app.use(myMiddleware);
app.get("/api/users", (req, res) => {
  res.json([]);
});
```

## 📋 Summary

### 🔄 Middleware Cheat Sheet

```
What is Middleware?
  → A function with (req, res, next) that runs BETWEEN request and response

The next() Function:
  → next()        = Move to the next middleware
  → next(error)   = Jump to error-handling middleware
  → Forgetting it = Request hangs forever

Types of Middleware:
  → Built-in:      express.json(), express.urlencoded(), express.static()
  → Third-party:  morgan (logging), cors (cross-origin), helmet (security)
  → Custom:        Logger, Auth, Rate limiter — anything you write

Mounting Middleware:
  → app.use(fn)              = Run on ALL requests
  → app.use("/path", fn)     = Run on matching paths
  → app.get("/path", fn, handler) = Run on specific route

Error Handling:
  → Regular middleware:  (req, res, next) — 3 params
  → Error middleware:    (err, req, res, next) — 4 params (MUST have all 4)

Execution Order:
  → Security (helmet, cors)
  → Logging (morgan)
  → Body parsing (json, urlencoded)
  → Custom middleware (requestId, timing)
  → Static files
  → Routes
  → 404 handler (no route matched)
  → Error handler (something went wrong) — ALWAYS LAST
```

### 🧠 Mental Model

```
Request ──► [helmet] ──► [cors] ──► [morgan] ──► [json parser] ──► [auth] ──► [route] ──► Response
               │            │          │              │               │           │
               │            │          │              │               │           │
            Security     Cross-      Logs         Parses          Checks       Your
            headers      origin      request      req.body        token        code
                         access
                         
If any middleware calls next(error):
Request ──► [middleware] ──✗──SKIP──✗──SKIP──✗──► [Error Handler] ──► Error Response
```
