```
┌──────────────────────────────────────────────────────────────┐
│                    THE MERN STACK                              │
├──────────────────────────────────────────────────────────────┤
│                                                               │
│  M = MongoDB       → Database (stores data)                  │
│  E = Express.js    → Backend framework (handles requests)    │
│  R = React         → Frontend library (user interface)       │
│  N = Node.js       → Runtime (runs JavaScript on server)     │
│                                                               │
│  How they work together:                                      │
│                                                               │
│  ┌─────────┐  HTTP/API  ┌──────────────┐  Mongoose  ┌─────┐│
│  │  React  │ ◄────────► │Express/Node  │ ◄────────► │Mongo││
│  │(Browser)│  requests  │  (Server)    │   queries  │ DB  ││
│  └─────────┘            └──────────────┘            └─────┘│
│                                                               │
│  React runs in the BROWSER (client-side)                     │
│  Express runs on the SERVER (server-side)                    │
│  They talk to each other through HTTP API calls              │
│  MongoDB stores all the data                                 │
│                                                               │
└──────────────────────────────────────────────────────────────┘
```

## 🌐 How Frontend and Backend Communicate

```
The Communication Flow:
──────────────────────────────────────────────

  1. User clicks "Load Products" button in React
       │
       ▼
  2. React sends an HTTP request:
     GET http://localhost:5000/api/products
       │
       ▼
  3. Express receives the request
     → Router matches /api/products
     → Controller queries MongoDB
     → Mongoose returns data
       │
       ▼
  4. Express sends HTTP response:
     { status: "success", data: { products: [...] } }
       │
       ▼
  5. React receives the response
     → Updates state with the products
     → Re-renders the UI with the data
       │
       ▼
  6. User sees the products on screen!

  This is a CLIENT-SERVER architecture.
  React (client) NEVER talks to MongoDB directly.
  Express (server) sits in between, controlling access.
```

```
Why not connect React directly to MongoDB?
──────────────────────────────────────────────

  ❌ React → MongoDB (NEVER DO THIS!)
  • Database credentials would be in the browser
  • Anyone could inspect the code and steal them
  • No access control (anyone can read/write anything)
  • No data validation on the server
  • No business logic enforcement

  ✅ React → Express → MongoDB (ALWAYS DO THIS!)
  • Credentials stay on the server (hidden)
  • Express controls who can access what
  • Input validation before database operations
  • Business logic runs on the server
  • Security layers (auth, rate limiting, etc.)
```

## 🔒 CORS — Cross-Origin Resource Sharing

### 🎯 What is the Same-Origin Policy?

Browsers have a built-in security feature called the **Same-Origin Policy**. It prevents web pages from making requests to a different domain/port than the one that served the page.

```
What is an "Origin"?
──────────────────────────────────────────────

  An origin = protocol + domain + port

  http://localhost:3000    ← Origin 1 (React dev server)
  http://localhost:5000    ← Origin 2 (Express API server)
  https://myapp.com        ← Origin 3
  https://api.myapp.com    ← Origin 4

  Two URLs have the SAME origin only if ALL THREE parts match:
  • Protocol (http vs https)
  • Domain (localhost vs myapp.com)
  • Port (3000 vs 5000)

  Examples:
  http://localhost:3000  vs  http://localhost:3000/about  → SAME ✓ (path differs, that's ok)
  http://localhost:3000  vs  http://localhost:5000         → DIFFERENT ✗ (diff port)
  http://localhost:3000  vs  https://localhost:3000        → DIFFERENT ✗ (diff protocol)
  https://myapp.com      vs  https://api.myapp.com         → DIFFERENT ✗ (diff subdomain)
```

```
The Problem in Development:
──────────────────────────────────────────────

  React dev server:    http://localhost:3000
  Express API server:  http://localhost:5000

  React (port 3000) tries to fetch from Express (port 5000)
  → DIFFERENT origins!
  → Browser BLOCKS the request!

  Browser Console Error:
  ┌─────────────────────────────────────────────────────────┐
  │ Access to fetch at 'http://localhost:5000/api/products' │
  │ from origin 'http://localhost:3000' has been blocked    │
  │ by CORS policy: No 'Access-Control-Allow-Origin'       │
  │ header is present on the requested resource.            │
  └─────────────────────────────────────────────────────────┘

  This is the #1 error new MERN developers encounter!
```

---

### 🎯 What is CORS?

**CORS** (Cross-Origin Resource Sharing) is a mechanism that allows servers to explicitly tell the browser: **"It's okay, I trust requests from this other origin."**

```
Without CORS:                        With CORS:
──────────────────────               ──────────────────────

  React (port 3000)                   React (port 3000)
   │                                   │
   │ GET /api/products                 │ GET /api/products
   │──────────────────►                │──────────────────►
   │                                   │
   │  Express (port 5000)              │  Express (port 5000)
   │  "No CORS headers"               │  "Access-Control-Allow-Origin:
   │                                   │   http://localhost:3000"
   │                                   │
   │ ✗ BROWSER BLOCKS!                │ ✓ BROWSER ALLOWS!
   │                                   │
   │ (Note: Express DID                │ Browser sees the header
   │  send the response.              │ and passes the data
   │  The BROWSER blocked             │ to React.
   │  React from reading it)          │
```

> **Key insight:** CORS is enforced by the **browser**, not the server. The server sends the response either way. The browser decides whether to let JavaScript access it based on the response headers.

### 🔗 How CORS Works Under the Hood

```
Simple Request (GET, POST with simple headers):
──────────────────────────────────────────────

  Browser                              Server
  ───────                              ──────
     │
     │  GET /api/products
     │  Origin: http://localhost:3000
     │──────────────────────────────────►│
     │                                   │

     │◄──────────────────────────────────│
     │
     │  Browser checks:
     │  Does "Access-Control-Allow-Origin"

  Browser                              Server
  ───────                              ──────

     │                                   │
     │  204 No Content                   │
     │  Access-Control-Allow-Origin:     │
     │    http://localhost:3000          │
     │  Access-Control-Allow-Methods:    │
     │    GET, POST, PUT, DELETE         │
     │  Access-Control-Allow-Headers:    │
     │    Authorization, Content-Type    │
     │◄──────────────────────────────────│
     │
     │  Browser: "Preflight passed!"
     │
     │  2. DELETE /api/products/123 (actual request)
     │  Origin: http://localhost:3000
     │  Authorization: Bearer eyJhbG...
     │──────────────────────────────────►│
     │                                   │
     │  200 OK                           │
     │  { status: "success" }            │
     │◄──────────────────────────────────│
```

### 🔧 Setting Up CORS in Express

```bash
npm install cors
```

```js
// app.js

const cors = require("cors");

// ──── Option 1: Allow ALL origins (development only!) ─

app.use(cors());
// This adds: Access-Control-Allow-Origin: *
// Any website can now make requests to your API
// ⚠️ NOT safe for production!

// ──── Option 2: Allow specific origin(s) (RECOMMENDED) ─

app.use(
  cors({
    origin: "http://localhost:3000", // Only allow React dev server
  })
);

// ──── Option 3: Allow multiple origins ──────────────

app.use(
  cors({
    origin: [
      "http://localhost:3000",           // React dev
      "http://localhost:5173",           // Vite dev
      "https://myapp.vercel.app",        // Production frontend
    ],
  })
);

// ──── Option 4: Dynamic origin with a function ──────

app.use(
  cors({
    origin: function (origin, callback) {
      const allowedOrigins = [
        "http://localhost:3000",
        "https://myapp.vercel.app",
      ];

      // Allow requests with no origin (like mobile apps or Postman)
      if (!origin) return callback(null, true);

      if (allowedOrigins.includes(origin)) {
        callback(null, true);
      } else {
        callback(new Error("Not allowed by CORS"));
      }
    },
  })
);

// ──── Option 5: Use environment variable ────────────

app.use(
  cors({
    origin: process.env.CLIENT_URL, // "http://localhost:3000" in .env
  })
);
```

### 📨 Preflight Requests Explained

Certain requests trigger a **preflight** — an automatic `OPTIONS` request the browser sends BEFORE the actual request, to check if the server allows it.

```
When does the browser send a preflight?
──────────────────────────────────────────────

  SIMPLE requests (NO preflight):
  • Method: GET, POST, HEAD
  • Headers: only standard headers like Content-Type (with simple values)
  • Content-Type: text/plain, multipart/form-data,
                  application/x-www-form-urlencoded

  PREFLIGHTED requests (sends OPTIONS first):
  • Method: PUT, PATCH, DELETE
  • Custom headers: Authorization, X-Custom-Header
  • Content-Type: application/json  ← Yes, this triggers preflight!

  So almost ALL of your API requests will be preflighted!
  (because you use Content-Type: application/json and Authorization)
```

```js
// Express handles preflight automatically with the cors package:

app.use(
  cors({
    origin: "http://localhost:3000",
    methods: ["GET", "POST", "PUT", "PATCH", "DELETE"], // Allowed methods
    allowedHeaders: ["Content-Type", "Authorization"],   // Allowed headers
    credentials: true,                                    // Allow cookies
    maxAge: 86400,                                        // Cache preflight for 24h
    // This means the browser won't send an OPTIONS request
    // for the same route for 24 hours (performance boost!)
  })
);

// Or handle OPTIONS manually (rarely needed):
app.options("*", cors()); // Respond to all preflight requests
```

### 🔗 CORS with Credentials (Cookies)

If you're using HTTP-only cookies for JWT (recommended), you need extra CORS configuration:

```js
// ──── Backend (Express) ─────────────────────────────

app.use(
  cors({
    origin: "http://localhost:3000",    // MUST be exact origin (not *)
    credentials: true,                  // REQUIRED for cookies
  })
);

// ⚠️ When credentials: true, you CANNOT use origin: "*"
// You MUST specify the exact origin.

// ──── Frontend (React) ──────────────────────────────

// fetch
const response = await fetch("http://localhost:5000/api/products", {
  credentials: "include", // REQUIRED to send/receive cookies
});

// axios
const api = axios.create({
  baseURL: "http://localhost:5000/api",
  withCredentials: true, // REQUIRED to send/receive cookies
});
```

```
CORS with Cookies — The Full Picture:
──────────────────────────────────────────────

  Without credentials: true:
  ─────────────────────────

  React sends request → No cookies sent
  Express sets cookie → Browser ignores it
  No authentication via cookies!

  With credentials: true (both sides):
  ─────────────────────────────────────

  React sends request → Cookies included automatically
  Express sets cookie → Browser stores it
  Next request → Browser sends the cookie back
  Authentication works! ✓

  REQUIREMENTS:
  1. Express: cors({ credentials: true, origin: "exact-url" })
  2. React:   fetch({ credentials: "include" }) or axios withCredentials
  3. origin CANNOT be * — must be the exact frontend URL
```

### 📝 Making API Calls from React

#### Using fetch (built-in)

```jsx
// ──── Basic fetch example ───────────────────────────

import { useState, useEffect } from "react";

function ProductList() {
  const [products, setProducts] = useState([]);
  const [loading, setLoading] = useState(true);
  const [error, setError] = useState(null);

  useEffect(() => {
    const fetchProducts = async () => {
      try {
        const response = await fetch("http://localhost:5000/api/products");

        // fetch doesn't throw on HTTP errors (4xx, 5xx)
        // You must check response.ok manually!
        if (!response.ok) {
          throw new Error(`HTTP error! status: ${response.status}`);
        }

        const data = await response.json();
        setProducts(data.data.products);
      } catch (err) {
        setError(err.message);
      } finally {
        setLoading(false);
      }
    };

    fetchProducts();
  }, []);

  if (loading) return <p>Loading...</p>;
  if (error) return <p>Error: {error}</p>;

  return (
    <ul>
      {products.map((product) => (
        <li key={product._id}>{product.name} - ${product.price}</li>
      ))}
    </ul>
  );
}
```

### 🚀 Setting Up Axios

Axios is preferred over fetch because it has better error handling, automatic JSON parsing, interceptors, and a cleaner API.

```bash
npm install axios   # Run in the client/ directory
```

```js
// client/src/services/api.js — Central API configuration

import axios from "axios";

// Create a reusable Axios instance
const api = axios.create({
  baseURL: import.meta.env.VITE_API_URL || "http://localhost:5000/api",
  withCredentials: true, // Send cookies with every request
  headers: {
    "Content-Type": "application/json",
  },
});

// ──── REQUEST Interceptor ───────────────────────────
// Runs BEFORE every request is sent

api.interceptors.request.use(
  (config) => {
    // If using localStorage for JWT (instead of cookies):
    const token = localStorage.getItem("token");
    if (token) {
      config.headers.Authorization = `Bearer ${token}`;
    }
    return config;
  },
  (error) => {
    return Promise.reject(error);
  }
);

// ──── RESPONSE Interceptor ──────────────────────────
// Runs AFTER every response is received

api.interceptors.response.use(
  (response) => {
    // Any 2xx status: return the data directly
    return response;
  },
  (error) => {
    // Any non-2xx status: handle errors globally
    if (error.response) {
      // Server responded with an error status
      const { status, data } = error.response;

      if (status === 401) {
        // Unauthorized — token expired or invalid
        localStorage.removeItem("token");
        window.location.href = "/login";
      }

      // Use the server's error message if available
      const message = data.message || "Something went wrong";
      console.error(`API Error (${status}): ${message}`);

      return Promise.reject(new Error(message));
    }

    if (error.request) {
      // Request was sent but no response received (network error)
      console.error("Network error — server may be down");
      return Promise.reject(new Error("Network error. Please try again."));
    }

    // Something else went wrong
    return Promise.reject(error);
  }
);

export default api;
```

```
Axios vs Fetch Comparison:
──────────────────────────────────────────────

  Feature                 fetch             axios
  ────────────            ─────             ─────
  Built-in?               ✅ Yes            ❌ Install needed
  Auto JSON parse?        ❌ Manual         ✅ Automatic
  Error on 4xx/5xx?       ❌ No             ✅ Yes (throws)
  Request interceptors?   ❌ No             ✅ Yes
  Response interceptors?  ❌ No             ✅ Yes
  Request cancellation?   ⚠️ AbortController ✅ CancelToken
  Progress tracking?      ❌ No             ✅ Yes
  Timeout?                ❌ No             ✅ Built-in
  XSRF protection?        ❌ No             ✅ Built-in

  Recommendation: Use Axios for real applications.
```

```jsx
// client/src/pages/ProductsPage.jsx

import { useState, useEffect } from "react";
import { getAllProducts, deleteProduct } from "../services/productService";

function ProductsPage() {
  const [products, setProducts] = useState([]);
  const [loading, setLoading] = useState(true);
  const [error, setError] = useState(null);

  useEffect(() => {
    loadProducts();
  }, []);

  const loadProducts = async () => {
    try {
      setLoading(true);
      const data = await getAllProducts({ sort: "-createdAt", limit: 10 });
      setProducts(data.data.products);
    } catch (err) {
      setError(err.message);
    } finally {
      setLoading(false);
    }
  };

  const handleDelete = async (id) => {
    if (!window.confirm("Are you sure?")) return;
    try {
      await deleteProduct(id);
      setProducts((prev) => prev.filter((p) => p._id !== id));
    } catch (err) {
      setError(err.message);
    }
  };

  if (loading) return <div className="spinner">Loading...</div>;
  if (error) return <div className="error">{error}</div>;

  return (
    <div>
      <h1>Products</h1>
      {products.map((product) => (
        <div key={product._id} className="product-card">
          <h3>{product.name}</h3>
          <p>${product.price}</p>
          <button onClick={() => handleDelete(product._id)}>Delete</button>
        </div>
      ))}
    </div>
  );
}
```

```jsx
// client/src/services/authService.js

import api from "./api";

export const signup = async (userData) => {
  const response = await api.post("/auth/signup", userData);

  localStorage.removeItem("token");
  window.location.href = "/login";
};

  return response.data;
};
```

```jsx
// client/src/context/AuthContext.jsx

import { createContext, useContext, useState, useEffect } from "react";
import { getMe, login as loginAPI, logout as logoutAPI } from "../services/authService";

const AuthContext = createContext(null);

export function AuthProvider({ children }) {
  const [user, setUser] = useState(null);
  const [loading, setLoading] = useState(true);

  // Check if user is logged in on app load
  useEffect(() => {
    const checkAuth = async () => {
      const token = localStorage.getItem("token");
      if (!token) {
        setLoading(false);
        return;
      }
      try {
        const data = await getMe();
        setUser(data.data.user);
      } catch {
        localStorage.removeItem("token");
      } finally {
        setLoading(false);
      }
    };
    checkAuth();
  }, []);

  const login = async (credentials) => {
    const data = await loginAPI(credentials);
    setUser(data.data.user);
    return data;
  };

  const logout = () => {
    logoutAPI();
    setUser(null);
  };

  return (
    <AuthContext.Provider value={{ user, login, logout, loading }}>
      {children}
    </AuthContext.Provider>
  );
}

export const useAuth = () => useContext(AuthContext);
```

### ⚠️ Error Handling in API Calls

```jsx
// ──── Custom hook for API calls ─────────────────────

import { useState, useCallback } from "react";

function useApi(apiFunction) {
  const [data, setData] = useState(null);
  const [error, setError] = useState(null);
  const [loading, setLoading] = useState(false);

  const execute = useCallback(
    async (...args) => {
      try {
        setLoading(true);
        setError(null);
        const result = await apiFunction(...args);
        setData(result);
        return result;
      } catch (err) {
        setError(err.message);
        throw err;
      } finally {
        setLoading(false);
      }
    },
    [apiFunction]
  );

  return { data, error, loading, execute };
}

// Usage:
function ProductList() {
  const { data, error, loading, execute } = useApi(getAllProducts);

  useEffect(() => {
    execute({ sort: "-createdAt" });
  }, [execute]);

  // ... render based on data, error, loading
}
```

```jsx
// client/src/hooks/useProducts.js

import { useQuery, useMutation, useQueryClient } from "@tanstack/react-query";
import {
  getAllProducts,
  createProduct,
  deleteProduct,
} from "../services/productService";

// Fetch all products
export function useProducts(params) {
  return useQuery({
    queryKey: ["products", params],
    queryFn: () => getAllProducts(params),
  });
}

// Create a product
export function useCreateProduct() {
  const queryClient = useQueryClient();

  return useMutation({
    mutationFn: createProduct,
    onSuccess: () => {
      // Invalidate the products cache so it refetches
      queryClient.invalidateQueries({ queryKey: ["products"] });
    },
  });
}

// Delete a product
export function useDeleteProduct() {
  const queryClient = useQueryClient();

  return useMutation({
    mutationFn: deleteProduct,
    onSuccess: () => {
      queryClient.invalidateQueries({ queryKey: ["products"] });
    },
  });
}
```

## ⚙️ Environment Variables

### 🎯 What Are Environment Variables?

Environment variables are **configuration values** stored OUTSIDE your code. They allow you to change how your application behaves without modifying the source code.

```
Analogy — Cooking Recipe:
──────────────────────────────────────────────

  Your code is the RECIPE (instructions).
  Environment variables are the INGREDIENTS LIST.

  Same recipe, different ingredients:

  DEVELOPMENT ingredients:          PRODUCTION ingredients:
  ───────────────────────          ───────────────────────
  DATABASE_URL=localhost:27017     DATABASE_URL=atlas.mongodb.net
  PORT=5000                        PORT=80
  JWT_SECRET=dev-secret-123        JWT_SECRET=xK9!mZ2@pQ7...
  NODE_ENV=development             NODE_ENV=production

  Same code. Different behavior based on environment.
```

### 🔧 Serving React from Express

In production, you can serve the React build from Express itself — one server for everything:

```js
// server/app.js — Serve React in production

const express = require("express");
const path = require("path");
const app = express();

// ... all your middleware and API routes ...

app.use("/api/auth", authRoutes);
app.use("/api/users", userRoutes);
app.use("/api/products", productRoutes);

// ──── Serve React in production ─────────────────────

if (process.env.NODE_ENV === "production") {
  // Serve static files from the React build
  app.use(express.static(path.join(__dirname, "../client/dist")));

  // For any route that doesn't match an API route,
  // send back the React app's index.html
  app.get("*", (req, res) => {
    res.sendFile(path.join(__dirname, "../client/dist/index.html"));
  });
}

// ──── Handle undefined API routes ───────────────────
app.all("/api/*", (req, res, next) => {
  next(new AppError(`Cannot find ${req.originalUrl} on this server`, 404));
});

app.use(globalErrorHandler);

module.exports = app;
```

```
How This Works:
──────────────────────────────────────────────

  In production, Express handles EVERYTHING:

  Request: GET /api/products
  → Matches API route → Controller sends JSON data

  Request: GET /dashboard
  → Doesn't match any API route
  → express.static looks for /client/dist/dashboard
  → No file found → falls through to the catch-all
  → Sends /client/dist/index.html
  → React Router takes over and renders the Dashboard page

  Request: GET /assets/main.js
  → express.static finds /client/dist/assets/main.js
  → Sends the file directly (static file serving)

  One server, one port, one deployment!
```

```
│  3. Database connection fails                                │
│     → Check DATABASE_URL is set correctly                    │
│     → Whitelist hosting platform's IP in Atlas               │
│     → Try: 0.0.0.0/0 (allow all) to test                    │
│                                                               │
│  4. React routes return 404                                  │
│     → Add catch-all: app.get("*", ...) for SPA              │
│     → Or add rewrite rule on static site host                │
│                                                               │
│  5. "Cannot find module" errors                              │
│     → Check package.json dependencies (not devDependencies)  │
│     → Run npm install on the server                          │
│                                                               │
│  6. Environment variables not working                        │
│     → Check var names match exactly (case-sensitive!)        │
│     → Redeploy after adding new env vars                     │
│     → For Vite: must start with VITE_ and rebuild            │
│                                                               │
│  7. Slow first request (cold starts)                         │
│     → Free tiers spin down after inactivity                  │
│     → First request after idle takes 10-30 seconds           │
│     → Solution: upgrade to paid tier or use a ping service   │
│                                                               │
│  8. File uploads don't work                                  │
│     → Serverless: filesystem is ephemeral/read-only          │
│     → Use cloud storage: AWS S3, Cloudinary, Firebase Storage│
```
